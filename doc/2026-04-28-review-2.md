## 1. Executive Summary

### Current project state

`os1` is a small, structured `x86_64` teaching kernel that boots from either a Limine/UEFI ISO or a legacy BIOS raw image into a shared higher-half kernel core ([src/kernel/linker/kernel_core.ld](../src/kernel/linker/kernel_core.ld) links the kernel at `kKernelVirtualOffset + physical` while it still loads physically near `0x100000`), runs a 1000 Hz PIT-driven scheduler over a 32-entry thread table, executes ELF64 user programs from a `cpio newc` initrd in PML4 slot 1, exposes a 9-call `int 0x80` syscall surface, and ships an interactive serial-driveable shell that reads structured observability snapshots through a versioned UAPI.

The code matches the milestone status documents ([doc/ARCHITECTURE.md](ARCHITECTURE.md) calls M1–M5 implemented; the source supports that claim). It does not match every prose claim in those documents (see §3).

### Strongest parts

1. **Trap pipeline is small, uniform, and audit-friendly.** [src/kernel/arch/x86_64/interrupt/inthandler.asm](../src/kernel/arch/x86_64/interrupt/inthandler.asm) and [irqhandler.asm](../src/kernel/arch/x86_64/interrupt/irqhandler.asm) funnel every exception, every IRQ, and the `int 0x80` syscall through one `trap_entry_common` path that materializes a single `TrapFrame` shape. Kernel-mode and ring-3 stack handling are split through `test dl, 3` ([irqhandler.asm:57-69](../src/kernel/arch/x86_64/interrupt/irqhandler.asm)). C++ `trap_dispatch` then classifies; the same return path drives both kernel resume (`resume_live_kernel`) and full thread switch (`restore_thread`). This is the kind of simplicity that pays compounding interest in debuggability.
2. **The user-copy boundary is real.** [src/kernel/mm/user_copy.cpp](../src/kernel/mm/user_copy.cpp) checks canonicality, range against `[kUserSpaceBase, kUserStackTop)`, integer overflow, page-by-page translation, and `User`/`Write` flags before any byte crosses the boundary. The 2026-04-24 critical bug (kernel mappings accepted by `copy_to_user`) is genuinely fixed.
3. **Process lifecycle is decomposed correctly.** Threads die via [thread.cpp:270-288](../src/kernel/proc/thread.cpp) (`mark_current_thread_dying` sets `Process` state to `Zombie` if a parent exists, else `Dying`); reclamation runs on a *different* stack via [reaper.cpp](../src/kernel/proc/reaper.cpp); `waitpid` reaps zombies via [wait.cpp:51-92](../src/kernel/syscall/wait.cpp); orphan rebinding happens in [process.cpp:41-55](../src/kernel/proc/process.cpp). The "do not free a thread's stack while running on it" invariant is honored.
4. **`exec` is mechanically tested as no-return.** [CMakeLists.txt:465-469](../CMakeLists.txt) adds two rejected markers (`shell exec returned`, `echo: after-exec`) on top of the standard reject set. If the old shell ever resumes after `exec`, the smoke fails. That is the right way to test in-place image replacement.
5. **CI is unusually disciplined for a one-person hobby OS.** Eight-test `ctest` matrix on UEFI + BIOS, OVMF auto-detection, an `OS1_REQUIRE_UEFI_SMOKE=ON` gate ([CMakeLists.txt:475-493](../CMakeLists.txt)) so a missing OVMF cannot silently turn off half the coverage, and an `act` workflow that pins to host toolchains for fast iteration.

### Biggest risks and gaps

1. **Three boot-image size envelopes for the BIOS path do not agree.** The kernel reservation window in [memory_layout.h:61-62](../src/kernel/handoff/memory_layout.h) is `kKernelReservedPhysicalStart = 0x100000` to `kKernelReservedPhysicalEnd = 0x160000` = **384 KiB**. The BIOS raw image carries `OS1_KERNEL_IMAGE_SECTOR_COUNT × 512 = 128 KiB` ([CMakeLists.txt:91](../CMakeLists.txt)). The actual `kernel_bios.elf` artifact is **~150 KiB**. The `dd bs=512 count=256 conv=notrunc,sync` at [CMakeLists.txt:220-226](../CMakeLists.txt) silently truncates the trailing ELF section table (~22 KiB) to fit the 128 KiB slot. Boot still works because PT_LOAD content lives in the early part of the ELF, but the safety margin is **already gone today**. There is no `file(SIZE ...)` assertion to catch the next 22 KiB of growth.
2. **No interrupt completion model.** [src/kernel/drivers/block/virtio_blk.cpp:246-263](../src/kernel/drivers/block/virtio_blk.cpp) busy-spins up to 1 000 000 times per sector read. [src/kernel/platform/pci.cpp:180-181](../src/kernel/platform/pci.cpp) only reads legacy `interrupt_line` / `interrupt_pin`; there is no MSI/MSI-X capability traversal anywhere in the tree. This is the structural gap that prevents the next driver tier (`virtio-net`, NVMe, USB) from being usable.
3. **APs are off the scheduler completely.** [src/kernel/arch/x86_64/cpu/cpu.cpp:14-24](../src/kernel/arch/x86_64/cpu/cpu.cpp) is `cli; hlt`. [src/kernel/arch/x86_64/cpu/cpu.cpp:169-171](../src/kernel/arch/x86_64/cpu/cpu.cpp) busy-waits unboundedly on `c->booted == 0`. The system is uniprocessor at runtime in spite of being multi-CPU at boot. Every kernel global (the list is in §2.4) is unprotected single-CPU state.
4. **Scheduler picks live threads through a fixed table iteration.** [src/kernel/sched/thread_queue.cpp](../src/kernel/sched/thread_queue.cpp) iterates a 32-entry array. There is no run queue, no priority, no time slice accounting, no per-CPU policy. That is fine for now but it is *the* class of state that needs to be re-shaped before SMP scheduling.
5. **No filesystem, no descriptor model, no multiuser.** [src/kernel/vfs/](../src/kernel/vfs), [src/kernel/security/](../src/kernel/security), and [src/kernel/drivers/net/](../src/kernel/drivers/net) are README-only. The shell relies on `/bin/<name>` literal resolution in [src/user/programs/sh.cpp:524-553](../src/user/programs/sh.cpp).
6. **No host-side test surface.** [cmake/scripts/](../cmake/scripts) has only smoke-runner glue. CPIO parsing ([src/kernel/fs/initrd.cpp](../src/kernel/fs/initrd.cpp)), ELF validation ([src/kernel/proc/user_program.cpp](../src/kernel/proc/user_program.cpp)), ACPI table walking, and `is_user_address_range` edges all rely on full QEMU smokes for coverage.
7. **`kernel_namespaces.hpp` is migration scaffolding with no callers.** [src/kernel/kernel_namespaces.hpp](../src/kernel/kernel_namespaces.hpp) is included by exactly one file ([core/kernel_main.cpp:27](../src/kernel/core/kernel_main.cpp)), which never references any `os1::kernel::*` symbol. A repo-wide `grep -r "os1::kernel"` confirms zero call sites outside the facade itself. Either commit to the migration or delete the file.
8. **Documentation drift.** [doc/ARCHITECTURE.md](ARCHITECTURE.md), [README.md](../README.md), and [GOALS.md](../GOALS.md) all still link to `Review 2026-04-23` as the current review (now four reviews stale counting this one). ARCHITECTURE describes the framebuffer terminal as "fixed 80x25 grid" with "full redraw on every present" — both wrong against [drivers/display/text_display.cpp:200-253](../src/kernel/drivers/display/text_display.cpp) and [core/kernel_main.cpp:150-176](../src/kernel/core/kernel_main.cpp). `/bin/ascii` is staged into the initrd by [src/user/CMakeLists.txt:61-72](../src/user/CMakeLists.txt) but absent from artifact lists.

Net assessment matches the prior review: the project is in a healthy place; the next work should be in *substrate hardening* — interrupt completion, locking, host-side tests, real block-device contract — not in any new vertical (networking, GUI, SSH).

## 2. Current State From Code

### 2.1 Boot path and platform assumptions

Two frontends, one kernel:

- **BIOS:** [boot.asm](../src/boot/bios/boot.asm) (83 lines, MBR) → [kernel16.asm](../src/boot/bios/kernel16.asm) (547 lines: A20, long-mode probe, EDD reads, RSDP scan, E820, builds `BootInfo`) → [long64.asm](../src/boot/bios/long64.asm) (188 lines: temp page tables, `LME`+`NXE`, ELF expansion at `0x100000`). Total BIOS frontend = ~1 270 lines of NASM.
- **UEFI:** [src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp) is **1 155 lines of C++**. It requests HHDM, memmap, framebuffer, modules, RSDP, SMBIOS; walks the active page tables to translate Limine-virtual pointers to physical (`TranslateLimineVirtual`); loads `kernel_bios.elf` as a module by writing PT_LOAD segments through HHDM; patches a low identity window (PML4[0] → PML3[0] → 2 MiB PML2 entries); mirrors the BIOS handoff layout (`BootInfo` at `0x500`, memmap at `0x6000`, modules at `0x7000`, strings at `0x7200`); and jumps to the same `kernel_main(BootInfo*, cpu*)`.

The shared kernel core (`kernel_bios.elf`) is not BIOS-only. The artifact name is misleading; [doc/ARCHITECTURE.md:64-68](ARCHITECTURE.md) acknowledges this. [src/kernel/handoff/boot_info.hpp](../src/kernel/handoff/boot_info.hpp) is `#pragma pack(push, 1)` with offset `static_assert`s; the C++ struct and the NASM `.inc` mirror in [memory_layout.inc](../src/kernel/handoff/memory_layout.inc) stay aligned by hand.

Platform assumptions baked in:
- low identity-linked kernel at `0x100000`;
- physical address ≡ kernel virtual address;
- `kBootInfoMaxMemoryRegions = 128`, `kBootInfoMaxModules = 16` ([boot_info.hpp:13-14](../src/kernel/handoff/boot_info.hpp));
- `kBootInfoMaxBootloaderNameBytes = 64`, `kBootInfoMaxCommandLineBytes = 256` — the kernel will reject larger handoffs after the `own_boot_info` deep copy.

### 2.2 Memory management

- [src/kernel/mm/page_frame.cpp](../src/kernel/mm/page_frame.cpp): bitmap allocator seeded from `std::span<const BootMemoryRegion>`. Backed by a fixed `0x20000`–`0x60000` (256 KiB) bitmap window declared in [memory_layout.h:55-57](../src/kernel/handoff/memory_layout.h).
- [src/kernel/mm/virtual_memory.cpp](../src/kernel/mm/virtual_memory.cpp): four-level page-table builder. Supports `map_physical`, `allocate_and_map`, `protect`, `translate`, `clone_kernel_mappings`, `destroy_user_slot`.
- [src/kernel/mm/user_copy.cpp](../src/kernel/mm/user_copy.cpp): canonical, range, overflow, translate, flags. The translation is per-page — slow but correct.
- [src/kernel/mm/boot_mapping.cpp](../src/kernel/mm/boot_mapping.cpp), [boot_reserve.cpp](../src/kernel/mm/boot_reserve.cpp): one-shot helpers for boot-critical regions (initrd, framebuffer, RSDP page).

Carry-forward concerns:
- **`mm/page_frame.hpp` includes `console/terminal.hpp`** ([page_frame.hpp:11](../src/kernel/mm/page_frame.hpp)). Nothing in the header uses `Terminal`. Stale include, inverted dependency direction. Same finding as 2026-04-27 and 2026-04-28 review-1; still unfixed.
- **`VirtualMemory::translate()` reports leaf flags only.** Effective CPU permission is the AND of all four levels. Today, user mappings propagate `User`/`Write` consistently up the tree, but no test pins this invariant. A code-walk audit in `virtual_memory.cpp` shows all `User` page allocations go through `allocate_and_map`, which sets `User` on each level it creates. No bug today; an invariant nobody checks.
- **`kBootMemoryRegionCapacity = 128`** is a hard limit on the deep-copied memory map. Modern UEFI memory maps regularly exceed 128 entries; the BIOS E820 path won't, but a future dev VM with NVDIMM or many BAR-backed regions could.

### 2.3 Interrupts and exceptions

The trap pipeline merits its own description because it is the cleanest part of the kernel:

- IDT install: [src/kernel/arch/x86_64/interrupt/interrupt.cpp](../src/kernel/arch/x86_64/interrupt/interrupt.cpp) sets up vector handlers including a DPL=3 user-callable interrupt gate at `0x80` for the syscall.
- Vector stubs: [inthandler.asm](../src/kernel/arch/x86_64/interrupt/inthandler.asm) (`EXC_NOERR`/`EXC_ERR` macros for vectors 0–13, 16–19, 29–30) and [irqhandler.asm:122-152](../src/kernel/arch/x86_64/interrupt/irqhandler.asm) (`IRQ_STUB` macro for IRQs 0–15, plus `int_80h`).
- Common path: every stub pushes a fake error code (when the CPU doesn't), pushes the vector, and jumps to `trap_entry_common` ([irqhandler.asm:13-94](../src/kernel/arch/x86_64/interrupt/irqhandler.asm)).
- Frame storage decision: if `gs:CPU_CURRENT_THREAD` is non-null, save into `thread->frame`; else save into `gs:CPU_INTERRUPT_FRAME`. This means the kernel can take an exception even before any thread is current.
- Ring split: `test dl, 3` on the saved CS chooses between user `RSP/SS` (already on the CPU-pushed frame) and a synthesized kernel `RSP` (`lea rdx, [rsp + 56]`) with `SS = 0`.
- Return paths: if `trap_dispatch` returns the same thread and the interrupted CS was kernel, do an in-place `iretq` ([resume_live_kernel:96-120](../src/kernel/arch/x86_64/interrupt/irqhandler.asm)); else `restore_thread` swaps to the new thread's `TrapFrame` and `iretq` from there.

This is sound. The remaining concern is that **the IRQ path in [core/irq_dispatch.cpp:32-37](../src/kernel/core/irq_dispatch.cpp) polls serial RX on every timer tick** (1000 Hz). That conflates the "console input is ready" signal with the timer rather than wiring serial as an actual IRQ. Cheap and works; will be in the way once `virtio-net` introduces its own IRQ-driven completion.

### 2.4 Scheduling and SMP state

State summary:

- Tables: `Process[32]` ([proc/process.hpp:12](../src/kernel/proc/process.hpp)), `Thread[32]` ([proc/thread.hpp:18](../src/kernel/proc/thread.hpp)).
- Selection: [sched/thread_queue.cpp](../src/kernel/sched/thread_queue.cpp) finds the next runnable thread by linear table walk.
- Idle: [sched/idle.cpp:14-19](../src/kernel/sched/idle.cpp) is `sti; hlt`.

- Reaper: [proc/reaper.cpp](../src/kernel/proc/reaper.cpp) frees the dying thread's kernel stack pages and the thread record, but **leaves the process address space alive if the process is `Zombie`** so `waitpid` can still inspect `exit_status`. This is correct.

AP state:
- [src/kernel/arch/x86_64/cpu/cpu.cpp:140-174](../src/kernel/arch/x86_64/cpu/cpu.cpp) sends INIT/SIPI to each AP via `lapic_start_cpu`, busy-waits unboundedly for `c->booted == 0` to flip, and the AP itself runs `init()` ([cpu.cpp:125-138](../src/kernel/arch/x86_64/cpu/cpu.cpp)) which does `cpu_init`, `ioapic_init`, `lapic_init`, sets `booted = 1`, then enters `cpu_idle_loop()` (`cli; hlt`).

Globals not protected by anything because there is only ever one CPU in kernel code:
- `page_frames` ([core/kernel_state.cpp:6](../src/kernel/core/kernel_state.cpp));
- `processTable`, `threadTable`, `g_idle_thread`, `g_next_pid`, `g_next_tid` ([proc/process.cpp:78](../src/kernel/proc/process.cpp), [proc/thread.cpp](../src/kernel/proc/thread.cpp));
- `g_platform`, `g_virtio_blk`, `g_virtio_blk_device` ([platform/state.cpp](../src/kernel/platform/state.cpp), [drivers/block/virtio_blk.cpp:136-137](../src/kernel/drivers/block/virtio_blk.cpp));
- `g_timer_ticks`, `g_text_display`, `g_boot_info`, `g_kernel_root_cr3` ([core/kernel_state.cpp](../src/kernel/core/kernel_state.cpp));
- `terminal[12]`, `active_terminal`, console input buffers ([console/console_input.cpp](../src/kernel/console/console_input.cpp));
- `g_cpu_boot` and the CPU list it heads.

This is fine *only because APs do not run kernel code*. Every line above is a future locking decision.

### 2.5 User/kernel separation state

- 9-call ABI: [src/uapi/os1/syscall_numbers.h](../src/uapi/os1/syscall_numbers.h). Dispatch in [syscall/dispatch.cpp](../src/kernel/syscall/dispatch.cpp).
- `sys_write` accepts only `fd ∈ {1, 2}` ([process.cpp:13-16](../src/kernel/syscall/process.cpp)) and chunks copies through a 128-byte stack buffer.
- `sys_read` accepts only `fd == 0` ([dispatch.cpp:42-46](../src/kernel/syscall/dispatch.cpp)). Asymmetry vs `write` is intentional.
- `sys_spawn` copies the path through `copy_user_string`, calls `load_user_program(... thread->process)` so the new process inherits the caller as parent ([process.cpp:44-70](../src/kernel/syscall/process.cpp)).
- `sys_exec` is in-place: writes the new CR3 with `context.write_cr3(new_cr3)` and tears down the old address space last ([process.cpp:72-115](../src/kernel/syscall/process.cpp)). The thread frame is rebuilt by `prepare_user_thread_entry` ([proc/user_program.cpp:210-229](../src/kernel/proc/user_program.cpp)).
- `sys_waitpid` blocks via `block_current_thread(ThreadWaitReason::ChildExit, frame->rsi, frame->rdi)` ([dispatch.cpp:88-96](../src/kernel/syscall/dispatch.cpp)). **Note:** `wait_address` holds the user `status` pointer (rsi) and `wait_length` holds the pid (rdi). [wait.cpp:106-108](../src/kernel/syscall/wait.cpp) reads them back as `(thread->wait_length, thread->wait_address)` matching the convention. The mapping is consistent but **pid in `wait_length` is a code smell** — typed wait fields would catch a future inversion.

What is missing on the user/kernel boundary:
- no argv/envp (the user stack is `iretq`-shaped only — [user_program.cpp:166-169](../src/kernel/proc/user_program.cpp) reserves a single dummy slot to match SysV function entry);
- no descriptor table beyond the hardcoded 0/1/2 fd slots;
- no signals;
- no current-directory or PATH abstraction;
- no `errno`-style return convention (everything returns `-1` on failure with no detail).

### Process and scheduling

The thread/process model is real. SMP is topology + AP startup, not SMP scheduling. Globals enumerated in §2.4 are the concurrency cliff.

### Storage and filesystems

Right shape (`BlockDevice` facade), wrong depth (single device, polling-only, no writes, no completion model, no error type). Filesystem-backed `exec` is blocked on the contract, not on parser code.

### Networking and SSH

Both unstarted. The dependency ordering in [GOALS.md:368-405](../GOALS.md) (protected userland > SMP shaping > GPU > networking > SSH > GUI) is honored by the current state, with one caveat: SMP shaping has happened *for topology*, not *for scheduling*. The scheduler still assumes one CPU.

### Accelerator awareness

PCI/PCIe discovery exists. Real prerequisites (DMA-safe allocation, MSI-X, memory pinning, user/kernel submission boundary) are absent. Same prerequisites as networking and storage scaling.

## 5. Overlooked Goals And Blind Spots

Items the architect may have under-allocated. Each is a concrete recommendation, not a vague concern.

### 5.1 Architecture

- **Decide low-half vs higher-half deliberately.** The kernel relies on `physical address = kernel virtual address` ([virtual_memory.cpp](../src/kernel/mm/virtual_memory.cpp), syscall copy boundary, `map_identity_range` everywhere). Every additional driver locks this in. The cost of moving to a higher-half model rises monotonically. Not urgent, but the decision is — picking "stay low-half until X" is itself a deliverable.
- **Pick `int 0x80` vs `SYSCALL`/`SYSRET` and document the decision.** Today the syscall is a DPL=3 interrupt gate ([interrupt.cpp](../src/kernel/arch/x86_64/interrupt/interrupt.cpp)). The 9-syscall ABI is small enough to migrate cheaply now; this gets harder with each new call.
- **`kernel_namespaces.hpp` is a fork in the road.** Either commit to the migration or delete it. The current half-state tells a contradictory story to a new reader.
- **The `cpu*` argument in `kernel_main(BootInfo*, cpu*)` is x86-specific.** A future port will want `BootInfo*` only with arch-specific state retrieved through an accessor. Cheap to fix now.

### 5.2 Observability and debuggability

- **Add a small kernel event ring.** A 256-entry ring of `{tsc, cpu, kind, payload}` recording IRQs, context switches, syscall numbers, faults, and major state transitions would make smoke flakes investigable. Cheap to build, expensive to skip.
- **Plan per-CPU counters before APs do anything.** When APs run kernel work, "what is CPU 2 doing right now" must be a structured question. Designing per-CPU counters under contention is harder than designing them under no contention.
- **Capture trap frame on panic.** [src/kernel/core/panic.cpp](../src/kernel/core/panic.cpp) and `halt_forever()` exist; a structured "RIP / CR2 / CR3 / vector / error / saved registers" dump on serial would prevent the next class of "what just happened?" debugging sessions.

### 5.3 Testability

- **Host-side tests for pure logic.** CPIO parsing, ELF validation (header magic / type / machine / phoff / phnum / phentsize), ACPI XSDT/RSDT length checks and entry-type dispatch, PCI capability walk termination guards, `is_user_address_range` edges. These are pure functions of bytes and trivially testable on the host with the system compiler. Today they are exercised only by full QEMU smokes, which is slow feedback.
- **Effective page permission tests.** A host harness that builds a small page table, asks `translate()` for various virtual addresses, and asserts the **AND** of every level matches expectation. Today the kernel reports leaf flags only.
- **A no-userland smoke.** "Kernel boots, never starts /bin/init, lands in idle" isolates scheduler/idle correctness from user behavior.

### 5.4 Security and trust boundaries

- **Document trust boundaries explicitly.** What does the kernel still dereference from bootloader memory after `own_boot_info`? Answer: framebuffer, RSDP page, initrd modules. What does it re-validate before trust? Today: nothing. ACPI signatures are checked but lengths are not bounded against MCFG or MADT entry sizes; PCI capability walks have a `guard < 48` to terminate but no per-cap length validation. None of this is exploitable today (no untrusted ACPI tables in QEMU smokes) but in a teaching OS that aims at SSH, the trust model needs to be in writing.
- **Plan the entropy story.** It does not need to be implemented yet. It needs an owner. Without a designated subsystem, SSH planning will hit a "where does randomness come from?" wall.

### 5.5 Portability boundaries

- **Audit and enforce `arch/x86_64` confinement.** A spot grep shows clean separation today. Add a `target_include_directories` rule per kernel module that *forbids* `arch/` includes by default and lists exceptions explicitly. Compile-time enforcement makes the boundary stick.

### 5.6 Developer workflow

- **Add `cmake --build build --target format`.** [autoformat.sh](../autoformat.sh) exists; expose it via CMake so it's discoverable.
- **A two-line CONTRIBUTING.md.** Not a goal of a solo teaching OS, but if external readers are part of the audience (per [GOALS.md:24-35](../GOALS.md)), one paragraph on "how to add a syscall, a driver, a smoke" pays off.
- **Add a fast-path smoke target.** `smoke_observe` is the smallest meaningful smoke; aliasing it to a `quick` target would tighten the inner loop.

### 5.7 Long-term maintainability

- **The 1 155-line [src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp) is becoming the kind of file the source split was supposed to retire.** It is one frontend with several distinct responsibilities (HHDM walks, ELF expansion, virt→phys translation, low identity window installation, BootInfo build). It would benefit from being split along those seams now while the file still has one author.
- **`kBootInfoMaxMemoryRegions = 128` is a soft fail.** Modern UEFI memmaps can exceed this. Today the BIOS path is the only path that always fits; the Limine path is one weird VM away from clipping. Either grow the constant or surface the truncation as a hard error in `own_boot_info`.

## 6. Roadmap

Strictly dependency-ordered.

### 6.1 Near term (each ≤ 1 week)

1. **Make BIOS image overflow a build error.** Add `file(SIZE …)` checks in [CMakeLists.txt](../CMakeLists.txt) for `kernel_bios.elf` vs `OS1_KERNEL_IMAGE_SECTOR_COUNT × 512` and `initrd.cpio` vs `OS1_INITRD_IMAGE_SECTOR_COUNT × 512`. *Why now:* the kernel ELF (~150 KiB) already exceeds the 128 KiB slot; the `dd ... count=256 conv=notrunc,sync` is silently truncating. The next 22 KiB of growth boots a broken kernel with no warning.
2. **Bound the AP boot wait.** Replace the unbounded `while (c->booted == 0) {}` in [cpu.cpp:169-171](../src/kernel/arch/x86_64/cpu/cpu.cpp) with a bounded loop that emits `cpu N timed out` on serial. *Why now:* a failed AP today is a silent boot hang; a failed AP after this change is a known failure.
3. **Update README, GOALS, and ARCHITECTURE to reflect reality.** Repoint the "current review" link, list `/bin/ascii`, fix the framebuffer terminal description (derived geometry + shadow-buffer dirty tracking), and note that `OS1_KERNEL_IMAGE_SECTOR_COUNT = 256` is the live size envelope. *Why now:* small false claims compound; readers form their first model from these documents.
4. **Resolve `kernel_namespaces.hpp`.** Either port `mm` to consume `os1::kernel::mm::*` or delete the header. *Why now:* unused migration scaffolding contradicts the source-organization story the docs tell.
5. **Remove the dead `console/terminal.hpp` include from `mm/page_frame.hpp`.** *Why now:* it inverts a dependency direction at the lowest layer of the kernel.
6. **Add an initrd manifest smoke marker.** Either in the observe smokes or as a separate `smoke_initrd_manifest`. Verify every staged path is present. *Why now:* `/bin/ascii` already drifted past current observe markers; the next addition will too.
7. **Type the `Thread::wait_*` fields by reason.** Today `wait_address`/`wait_length` carry different meanings depending on `wait_reason` (line-buffer pointer + length for `ConsoleRead`, status pointer + pid for `ChildExit`). A small union tag would prevent a future inversion. *Why now:* before more wait reasons are added.

### 6.2 Mid term (each 1–4 weeks)

1. **`BlockDevice` v2.** Define `BlockRequest`, sector counts, error codes, request lifetime, completion. Polling stays acceptable; the contract is what blocks a filesystem.
2. **Lift shared virtio transport.** Move queue setup, feature negotiation, BAR mapping, and notification out of `virtio_blk.cpp` into [src/kernel/drivers/virtio/](../src/kernel/drivers/virtio). Re-implement `virtio-blk` against it as a sanity check before `virtio-net` lands.
3. **MSI/MSI-X capability traversal in `pci.cpp`.** Surface MSI-X table info in `PciDevice`. Convert `virtio-blk` to MSI-X completion. *Sequence:* must precede networking.
4. **First read-only filesystem and filesystem-backed `exec`.** FAT32 is the obvious choice (good documentation, easy reproducible image workflow, well-tested in QEMU). Read-only first. Filesystem-backed loading is the bridge from "demo initrd" to "actual operating system".
5. **Minimal file descriptor table in `Process`.** Replace the hardcoded fd 0/1/2 special cases. Console becomes fd 0/1/2 by default; filesystem opens become fd 3+.
6. **SMP synchronization contract on paper.** Before any AP scheduler code, document: (a) what locks protect what global; (b) interrupt-masking rules; (c) lock ordering. A document is enough at this stage.
7. **Host-side test harness.** A `tests/` subtree built with the host compiler. Start with CPIO, ELF header validation, `is_user_address_range`, and ACPI table parsers (extract a pure-C++ tablewalker out of `acpi.cpp`).

### 6.3 Later (significant scope each)

1. **APs in scheduling.** Per-CPU run queue, cross-CPU wakeup IPI, IRQ steering through IOAPIC, audited globals.
2. **Networking in layers.** `virtio-net` → packet buffers → Ethernet → ARP → IPv4 → ICMP → UDP → TCP, each tested.
3. **Pty + sessions.** The console-input single-line-buffer model gives way to a pty layer that supports multiple concurrent shells.
4. **Multiuser foundation.** uid/gid in `Process`, file ownership, permission checks at the VFS, login policy.
5. **SSH.** Only after substrate exists.
6. **Higher-half migration or explicit direct-map decision.** Before multiuser/security and DMA isolation become expensive to retrofit.
7. **Framebuffer compositor.** Only after pty/session maturity.
8. **Accelerator path.** PCI discovery → resource ownership → DMA pinning → command submission → completion → access control.

### 6.4 Explicit dependency ordering

- **Before networking:** shared virtio transport, MSI/MSI-X, IRQ allocation API, packet buffer ownership, DMA-safe memory abstraction, driver observability.
- **Before SSH:** networking, credentials, ptys, sessions, filesystem-backed config, entropy, crypto policy.
- **Before GUI:** input event abstraction, terminal/session model, display-buffer ownership, scheduler stability under interactive load.
- **Before SMP expansion:** locking vocabulary, preemption rules, per-CPU state ownership, scheduler IPIs, IRQ steering, audited concurrency rules across allocator/console/platform/driver state.
- **Before broad POSIX growth:** file descriptors, filesystem namespace, argv/envp, error-code conventions, richer process-tree semantics.
- **Before accelerator work:** PCI resource ownership, DMA-safe memory, MSI-X, memory pinning, user/kernel submission boundary.

## 7. Immediate Next Steps

The highest-value actions to take *next*. Each names why it matters now.

1. **Convert silent BIOS truncation into a build error.** [build/artifacts/kernel_bios.elf](../build/artifacts) is ~150 KiB; the BIOS slot is 128 KiB. PT_LOAD content fits today by luck of ELF layout. The next ~22 KiB of kernel growth produces a kernel that boots into broken section state with no diagnostic.
2. **Bound `cpu_boot_others`'s AP wait.** A failed AP becomes an understandable failure instead of a silent hang. ~10 lines.
3. **Repoint the documentation review link and update the framebuffer/initrd descriptions.** The doc surface area for new readers is currently telling them about a system that hasn't existed for a week.
4. **Decide and execute on `kernel_namespaces.hpp`.** Either migrate one subsystem to consume the namespaced symbols (proving the facade has a purpose) or delete it. Half-states are worse than either choice.
5. **Define the `BlockDevice` v2 contract before writing a filesystem.** Sector count, error model, request ownership, completion. Choosing these post-hoc forces filesystem code to either break the abstraction or revise it; both options cost more than designing now.
6. **Add host-side tests for `mm/user_copy`, CPIO, and ELF.** These are the most security-sensitive parsers in the kernel. Today they are tested only end-to-end through QEMU. Pulling them into a host test harness is mostly extraction work and pays back forever.
7. **Draft the SMP synchronization contract on paper.** Before any AP scheduler code, document which locks protect which globals (the list in §2.4). A document preempts a class of subtle bugs; the alternative is debugging contention through serial logs.
8. **Plan `argv`/`envp` for `spawn`/`exec`.** Locking down a SysV initial-stack layout (`argc`, `argv[]`, `NULL`, `envp[]`, `NULL`, `auxv[]`, `AT_NULL`) before the userland surface grows further is materially cheaper than a retrofit.
9. **Type the `Thread::wait_*` fields per `wait_reason`.** Storing a pid in `wait_length` is the kind of code smell that produces hard-to-find bugs once a third or fourth wait reason is added.
10. **Add an MSI/MSI-X capability walk to `platform/pci.cpp`.** Even before any driver consumes it, having the data in `PciDevice` makes the next driver land cleanly.
