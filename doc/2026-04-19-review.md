## 1. Executive summary

### Current project state
`os1` is an educational x86_64 hobby operating system. It consists of a two-stage BIOS bootloader in NASM, a 16-bit loader stage that parses E820 memory, a manual ELF64 loader, and a freestanding C++14 kernel (~4k LoC across `src/kernel`, plus ~1k LoC of assembly). The build chain is `x86_64-elf-{gcc,g++,ld}` + `nasm`, driven by raw `Makefile`s; the output is a 1 MB raw disk image run under QEMU with `-smp 4`. The last meaningful commit is `fea5add "memory map"`; the `Initial commit` lineage begins at `dda1f41` after a long `revive` gap. Captured serial log [os1.log](../os1.log) shows the kernel currently reaches "done" after booting 4 APs and then halts — **everything past AP bring-up is deliberately disabled** (large commented block in [src/kernel/kernel.cpp:614-767](../src/kernel/kernel.cpp#L614-L767)).

### Major strengths
- Clean layered bring-up: MBR → 16-bit loader → long-mode trampoline → ELF64 kernel, each layer isolated and readable.
- Real 4-level page-table allocator with identity mapping of detected E820 blocks ([src/kernel/virtualmemory.cpp](../src/kernel/virtualmemory.cpp)).
- Working legacy MP 1.4 discovery with AP INIT-SIPI-SIPI via a copied 16-bit trampoline at 0x1000 ([src/kernel/cpustart.asm](../src/kernel/cpustart.asm), [src/kernel/cpu.cpp:124-165](../src/kernel/cpu.cpp#L124-L165)). Serial log confirms 4 CPUs reach `cpu init worked!`.
- Serial-port debug logger with a compact operator-chain API ([src/kernel/debug.cpp](../src/kernel/debug.cpp)).
- Bitmap page-frame allocator that copes with gaps and respects reserved regions ([src/kernel/pageframe.cpp](../src/kernel/pageframe.cpp)).

### Major weaknesses / risks
- **Kernel does not reach its feature set on boot.** The explicit `die()` after `cpu_bootothers` means terminals, keyboard, exception handlers, and multitasking (all implemented) are never enabled ([src/kernel/kernel.cpp:612-616](../src/kernel/kernel.cpp#L612-L616)).
- **SMP-unsafe primitives.** The exception `saveregs`/`restoreregs` block is a *single* global buffer at `0x10000` ([src/kernel/inthandler.asm:7](../src/kernel/inthandler.asm#L7)); `current_task` is a single pointer at `0x400` ([src/kernel/task.cpp:5-6](../src/kernel/task.cpp#L5-L6)). Any concurrent trap on a second CPU corrupts state.
- **Latent GDT bug.** `SEGDESC64_CODE(dpl)` uses `&` instead of `|` ([src/kernel/cpu.h:12](../src/kernel/cpu.h#L12)); with `dpl=0` the kernel-code descriptor evaluates to 0 (Null). Harmless only because `cpu_init()` never far-jumps CS and the current execution path halts before any `iretq` uses it.
- **Obsolete hardware discovery.** MP Floating Pointer is not present on post-~2010 firmware; the kernel will see `ismp=0` and skip IOAPIC setup outside QEMU/Bochs. No ACPI parser exists.
- **No tests, no CI, no user-mode, no filesystem, no disk I/O after boot.** Docs consist of a two-line `README.md` and a hand-written memory map in `src/memory.txt`; no requirements or design documents.
- Build artifacts (`build/*.o`, `os1.raw`) and runtime `os1.log` exist at HEAD despite `.gitignore` — remnants of a pre-`.gitignore` commit.

### Overall delivery assessment
The project is a **mid-bring-up educational kernel**, not a deliverable system. A reader can learn from it; a user cannot run anything useful. It is ~40-50% of the way to the "panic-less interactive multi-tasking kernel" the commented-out code hints at, and <5% of the way to anything a 2026 production OS would be evaluated against. A "regarding outcome" in today's software world for this codebase is **not** a competitor to Linux/NT/seL4 — it is a credible learning-artifact OS runnable in QEMU with a shell and a couple of demo user programs. That outcome is reachable in a handful of focused milestones.

## 2. Current functionality review

### Implemented and exercised on boot (evidenced by [os1.log](../os1.log))
- 512-byte MBR with LBA disk read ([src/boot/bios/boot.asm](../src/boot/bios/boot.asm)).
- 16-bit stage 2: A20 enable, long-mode CPUID check, four 64-sector LBA reads into a 128 KB ELF image, E820 memory detection, cursor capture, switch to long mode with identity-mapped first 2 MiB ([src/boot/bios/kernel16.asm](../src/boot/bios/kernel16.asm), [src/boot/bios/long64.asm](../src/boot/bios/long64.asm)).

- Page-frame bitmap allocator initialized against E820 map (bitmap at `0x20000`, 64 GB ceiling).
- `VirtualMemory::Allocate` with identity mapping of every detected RAM block + LAPIC/IOAPIC MMIO; `cr3` switched to kernel PML4.
- MP 1.4 discovery, LAPIC/IOAPIC/PIC init (boot CPU only in the live path), AP bring-up using an 82-byte trampoline copied to 0x1000 — **confirmed reaching `cpu_init worked!` on CPUs 1-3**.

### Implemented but NOT reached on boot (dead-on-arrival in `main`)
- 12 virtual text-mode terminals at dynamically allocated pages, screen linking/unlinking, F1–F12 hotkey switching, scrolling ([src/kernel/terminal.cpp](../src/kernel/terminal.cpp), [src/kernel/kernel.cpp:58-89](../src/kernel/kernel.cpp#L58-L89)).

- IDT setup for all 22 defined exception vectors plus IRQ0-15 via remapped legacy PIC ([src/kernel/interrupt.cpp](../src/kernel/interrupt.cpp)).
- Round-robin task switcher driven by PIT IRQ0 with full register save/restore, per-task stack, runnable `process1/2/3` demos ([src/kernel/multitask.asm](../src/kernel/multitask.asm), [src/kernel/task.cpp](../src/kernel/task.cpp)).
- 22 individual `onExceptionXX` handlers that dump RIP/RSP/CR2/CR3/error code and halt.

All of this is blocked because [kernel.cpp:615](../src/kernel/kernel.cpp#L615) runs `die()` immediately after AP startup, and the multitasking block is inside a `/* ... */` comment. **Inference**: this is a debugging checkpoint left in place after the AP-bringup push, not an intended feature removal.

### Partially implemented
- Exception handling: handlers exist and are wired, but `saveregs` uses a *single* global buffer (`0x10000`) — unusable under any concurrent trap.
- Multiprocessor support: APs boot, but none of the kernel subsystems after AP start are SMP-safe (single IDT is fine; single task list, single register scratch area, no locks, no per-CPU `current_task`).
- Per-task address space: `task->regs.cr3 = 0x60000` is hardcoded to the AP-boot temporary value ([src/kernel/task.cpp:100](../src/kernel/task.cpp#L100)) — all tasks share one CR3.
- `cpu_bootothers` boots APs but `init()` ([src/kernel/cpu.cpp:97-121](../src/kernel/cpu.cpp#L97-L121)) has explicit `TODO: initialize properly <---`; APs land in `die()` after setting `booted=1`.

### Missing (not attempted)
- ACPI (MADT/RSDP) parsing. Legacy MP-only discovery will fail on contemporary hardware.
- Any disk driver after boot (ATA/AHCI/NVMe/virtio).
- Filesystem of any kind.
- User mode / Ring 3 / syscalls / `iretq` to CPL≠0.
- TSS and separate interrupt stacks (IST). The `kstackhi` field exists but is not pointed to by any TSS.
- Spinlocks, atomics beyond `xchg`, memory barriers.
- Timekeeping beyond setting PIT frequency. No TSC, HPET, or monotonic clock.
- Network, USB, framebuffer (only text mode 0xB8000). Video-mode detection is marked TODO in [kernel16.asm:167](../src/boot/bios/kernel16.asm#L167).
- Any form of packaging, installer, UEFI support (BIOS only — becoming extinct in 2026 firmware).
- Tests of any kind, CI, reproducible-build metadata.

## 4. Technical quality assessment

### Architecture
- Layering is sensible: boot stages → kernel → subsystems (pageframe → virtualmemory → cpu/mp/lapic/ioapic → interrupts → terminal/keyboard → task).
- Per-CPU struct at the bottom of the kernel stack is a good xv6-style pattern ([cpu.h:63-67](../src/kernel/cpu.h#L63-L67)).
- Serial debug as an operator-chain object is pleasant to use and compact in binary.

### Code organization
- Source split across `src/boot`, `src/kernel`, `src/libc` is adequate.
- Header/implementation pairing is mostly consistent. A few outliers: `sysinfo.h` duplicated between `kernel/` and `sandbox/`; `terminal.h` lives in both.
- Namespaces: none used. Top-level `interrupts`, `page_frames`, `keyboard`, `terminal[]`, `active_terminal`, `g_cpu_boot`, `lapic`, `ioapic`, `ncpu`, `ismp` are all global. Fine for a toy kernel, but will fight you later.

### Maintainability
- ~6 kLoC total — small enough to hold in one head.
- **Large copy-paste block**: 22 near-identical `onExceptionNN` handlers in [kernel.cpp](../src/kernel/kernel.cpp) differing only by a name string. A macro or a table + a shared body would collapse ~420 LoC.
- Magic numbers are pervasive (`0x10000`, `0x20000`, `0x400`, `0x60000`, `0x1000`, `256 * sizeof(...)`). A `memory_layout.h` (or equivalent linker-script symbols) would document intent.
- Commented-out dead code in `kernel.cpp` and stale TODOs (see list in §5).

### Test coverage / test quality
- **Zero.** No unit tests, no integration tests, no QEMU smoke test, no CI. The only verification signal is the serial log reaching "done".

### Deployment readiness
- The raw image boots under `qemu-system-x86_64 -smp 4` reliably for AP bring-up. That is the full deployment story. No UEFI, no signed boot, no real-hardware 2026 firmware path, no installer, no packaging.

### Operational concerns
- No panic/assertion unwinder beyond `hlt` loops — no backtrace, no log shipping.
- No watchdog.
- Single PIT/LAPIC-timer tick with no calibration.

### Security / reliability
- Kernel runs everything at CPL 0 with no user/kernel split: expected for this stage, but means there is no security model to evaluate.
- No W^X on pages — `PAGE_PRESENT | PAGE_WRITE` is used everywhere, NX bit never set.
- `xchg` is the only synchronization primitive; no locks, no memory fences — the AP wait-loop in `cpu_bootothers` relies on `booted` being marked `volatile`, which is insufficient by the C++ memory model and works only because x86 TSO happens to forgive it.

### Notable technical debt
1. `saveregs` / `restoreregs` single global scratch area (inthandler.asm:7). **Blocker for SMP traps.**
2. `task->regs.cr3 = 0x60000` hardcoded (task.cpp:100).
3. `SEGDESC64_CODE(dpl)` uses `&` instead of `|` (cpu.h:12). Currently latent; will triple-fault any execution path that far-jumps CS via the per-CPU GDT.
4. `current_task` is a single `Task**` at a fixed address (task.cpp:5). **Blocker for SMP scheduling.**
5. `init()` for APs has explicit `TODO: initialize properly <---` (cpu.cpp:100).
6. `microdelay` in lapic.cpp reads the PIT latch in a way whose `& 0x80` check is operator-precedence-wrong (`0 == b & 0x80` parses as `(0 == b) & 0x80` → always 0). The loop exits on first iteration. Latent correctness issue.
7. Page-frame allocator's multi-page path is O(page_count) per call (pageframe.cpp:212-241).
8. Build artifacts committed to git; `os1.raw` (1 MB) and `build/*.o` should be purged.

## 6. Milestone roadmap (high level)

**M0 — Re-green the current feature set (~1 week of focused work)**
- Goal: boot reaches an interactive prompt on a single CPU with terminals, keyboard, exceptions, and the PIT round-robin scheduler, exactly as the commented-out `KernelMain` tail implies.
- Scope: remove `die()`, un-comment the multitasking block, delete or fix the `SEGDESC64_CODE` macro, verify under QEMU.
- Dependencies: none.
- Completion signal: pressing F1–F12 switches terminals, typing echoes, `process1/2/3` demos run across three terminals, PIT-driven preemption is visible.

**M1 — SMP-safe kernel core (~2 weeks)**
- Goal: run the scheduler on all detected CPUs without corrupting state.
- Scope: per-CPU `current_task` and interrupt scratch area (use the per-CPU `cpu` struct's stack page), spinlocks with atomic builtins, TSS + per-CPU kernel stack, fix `microdelay` bug, fix GDT macro.
- Dependencies: M0.
- Completion signal: N CPUs each running independent tasks with no register corruption; assertion-based stress test survives 10 s.

**M2 — Modern firmware discovery and timekeeping (~2 weeks)**
- Goal: boot on hardware that lacks MP 1.4.
- Scope: parse ACPI RSDP/RSDT/MADT; LAPIC calibration from HPET or PIT; a `monotonic_ns()` API; optional multiboot2/limine shim for UEFI delegation.
- Dependencies: M1.
- Completion signal: same kernel boots unmodified on a Ryzen/Intel laptop via a UEFI CSM or a multiboot2 loader.

**M3 — User mode and syscalls (~2 weeks)**
- Goal: first ring-3 program.
- Scope: user code/data GDT entries, TSS RSP0 per CPU, `syscall`/`sysret` MSR wiring, a two-call ABI (`write`, `exit`), copy of a user ELF into a fresh address space, `iretq`-to-ring-3 path.
- Dependencies: M1.
- Completion signal: a `hello` user program prints to a terminal and exits without corrupting the kernel.

**M4 — Block device + read-only filesystem (~2 weeks)**
- Goal: load user programs from disk.
- Scope: one virtio-blk or ATA PIO driver, a read-only FAT12/16 or tar-based `initrd` loader, an `exec` syscall.
- Dependencies: M3.
- Completion signal: kernel reads `/bin/shell` off the image and runs it in user mode.

**M5 — Test harness + CI + docs (~1 week, can run in parallel)**
- Goal: regressions are visible.
- Scope: a `make test` that runs QEMU headless with a scripted input and greps the serial log; a GitHub Actions workflow; `doc/` with architecture + memory-map + build-and-run notes; delete `sandbox/` and `build/*` from tracking.
- Dependencies: M0 minimum.
- Completion signal: every push produces a green (or red) CI status with a clear failure log.

**M6 — Credible hobby-OS outcome (~4-6 weeks cumulative)**
- Goal: the "regarding outcome in today's software world" bar — a kernel that runs on UEFI hardware, supports SMP, schedules preemptively in user mode, loads programs from disk, ships with a trivial shell, has CI, and can be forked by a reader in one afternoon.
- Scope: sum of M0-M5 plus a `sh` user program and basic `read` syscall.
- Completion signal: recording of a real machine booting os1, logging in, running `ls` against an initrd, and exiting.

A **non-goal** for the practical outcome: competing with Linux, supporting arbitrary devices, networking, GUI. These are out of scope for a reasonable single-maintainer horizon.

1. Is real-hardware boot a hard goal, or is QEMU sufficient? (Drives M2 priority.)
2. BIOS-only or eventually UEFI? (If UEFI, a multiboot2-compatible loader like Limine is a one-line win over rewriting `boot.asm`.)
3. Target CPU floor: any x86-64, or 2015+ (can assume AVX2, RDRAND, x2APIC)?
4. Is C++14 a constraint or a preference? (Dropping RTTI/exceptions is already implicit; consider `-std=c++20 -fno-rtti -fno-exceptions -fno-threadsafe-statics`.)
5. Is the "12 terminals + round-robin tasks" demo the desired final UX, or a stepping stone toward a real shell?

**Open technical questions**:
