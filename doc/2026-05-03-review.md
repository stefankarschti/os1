## 1. Executive Summary

### Current project state

`os1` now sits at a substantially different architectural plateau than the prior review described. The kernel weighs in at 19,030 lines of C++/headers across 22 named subsystems (`find src/kernel -name '*.cpp' -o -name '*.hpp' -o -name '*.h' | xargs wc -l`), up roughly 4× from the 5,100-line figure the previous review cited (which was an undercount even at the time). The host-test suite has grown from 38 cases to 86 across 23 source files. The QEMU smoke matrix has grown from eight tests to nine (the new `os1_smoke_xhci` adds an attached USB keyboard).

Concrete substrate, organized so a new reader can pick up cold:

- **Boot and address model**: shared higher-half kernel at `kKernelVirtualOffset + physical`, kernel-owned direct map at `kDirectMapBase`, two boot frontends (BIOS in [src/boot/bios](../src/boot/bios), Limine/UEFI in [src/boot/limine](../src/boot/limine)) converging on `kernel_main(BootInfo*, cpu*)` in [src/kernel/core/kernel_main.cpp:262](../src/kernel/core/kernel_main.cpp#L262). BootInfo is deep-copied into kernel BSS by `own_boot_info()`. Two intentional low-identity exceptions remain: AP trampoline mailbox and the live BSP handoff stack page.
- **Platform layer**: ACPI RSDP → XSDT/RSDT → MADT/MCFG/HPET/FADT, plus a real DSDT/SSDT walker in [src/kernel/platform/acpi_aml.cpp](../src/kernel/platform/acpi_aml.cpp) (1562 LOC) implementing a minimal AML interpreter for `_PRT`, `_CRS`, `_STA`, `_ADR`, `_BBN`, `_HID`, `_UID`, `_PS0`, `_PS3`, plus the `Return`/`Store`/package/buffer/integer/namepath forms needed by current QEMU/q35. - **Interrupts and IRQ routing**: dynamic IRQ vector allocator over `0x50..0xef` in [src/kernel/arch/x86_64/interrupt/vector_allocator.cpp](../src/kernel/arch/x86_64/interrupt/vector_allocator.cpp); per-vector callback table; an `IrqRoute` registry in [src/kernel/platform/irq_registry.cpp](../src/kernel/platform/irq_registry.cpp) tracking `LegacyIsa`/`LocalApic`/`Msi`/`Msix` ownership; PCI MSI-X parsing/programming, MSI fallback, and AML `_PRT`-aware IOAPIC INTx fallback in [src/kernel/platform/pci_msi.cpp](../src/kernel/platform/pci_msi.cpp).

### Strongest parts

1. **Velocity through a coherent substrate.** The 2026-04-30 pass landed eleven phases of the [driver/device/platform implementation plan](2026-04-29-driver-device-platform-implementation-plan.md) (phases 1-11 plus phase 12), each with code, host tests, and matching documentation. Five of the prior review's eight named near-term items closed in this push. None landed half-built. This is the same velocity profile the prior review noted, sustained one more cycle in a row.
2. **The driver substrate is no longer a sketch.** `drivers/bus/`, `drivers/virtio/`, the IRQ vector allocator, `pci_msi.cpp`'s MSI-X-then-MSI-then-INTx fallback, the AML-aware INTx routing through `acpi_resolve_pci_route_details`, and the DMA buffer with owner/direction/lifetime are exactly the architecture the prior three reviews demanded. They are also all *used*. `virtio-blk`, `virtio-net`, and xHCI all go through the same `pci_enable_best_interrupt`, the same DMA allocator, and the same `claim_pci_bar`. A future driver inherits this for free.
3. **The first interrupt-driven block path now exists.** `virtio_blk_submit_request` ([src/kernel/drivers/block/virtio_blk.cpp:288](../src/kernel/drivers/block/virtio_blk.cpp#L288)) is the model: build descriptor chain, `dma_sync_for_device`, `virtqueue_submit`, `virtio_pci_notify_queue`, return. Completion arrives through `virtio_blk_irq` which calls `virtio_blk_drain_used`, records `OS1_KERNEL_EVENT_BLOCK_IO` per request, and signals the completion flag. The lingering "spin until done" path in `virtio_blk_wait_for_boot_completion` is intentionally narrowed to the pre-scheduler boot smoke; the threaded smoke (`virtio-blk threaded smoke ok`) exercises the post-scheduler path.
4. **Real AML, not a stub.** [src/kernel/platform/acpi_aml.cpp](../src/kernel/platform/acpi_aml.cpp) is a deliberate minimum-viable interpreter rather than a pretend one: it actually parses, walks the namespace, handles packages and namepaths, evaluates `_PRT` and `_CRS` to produce IRQ routing data, and drives `_PS0`/`_PS3` for D0/D3 transitions. Crucially, it lives behind a narrow API ([src/kernel/platform/acpi_aml.hpp](../src/kernel/platform/acpi_aml.hpp)) and is host-tested against fixtures ([tests/host/kernel/acpi_aml_tests.cpp](../tests/host/kernel/acpi_aml_tests.cpp)). When it grows, it grows from a known good baseline rather than from a black box.
5. **Observability is now load-bearing.** The kernel event ring carries traps, scheduler transitions, IRQs, block I/O, PCI bind, user-copy failures, smoke markers, timer-source choice, and NIC RX. The observe ABI exposes devices, resources, IRQs alongside the prior six record kinds. The smoke matrix asserts not just "boot succeeded" but specific event sequences (`event smoke-marker seq=`, `device pci 2 started 2 virtio-net`, `timer-source-lapic`). Regressions in the substrate are now visible to CI rather than to a future bring-up session.
6. **Host tests now cover the new code.** 86 cases across `kernel/acpi_aml`, `kernel/acpi_hpet`, `kernel/event_ring`, `kernel/hid_keyboard`, `kernel/irq_route_registry`, `kernel/irq_vector_allocator`, `kernel/pci_bus`, `kernel/pci_capability`, `kernel/pci_msi`, `kernel/pci_resource`, `kernel/timer_source`, `kernel/utility`, `kernel/xhci_controller`, `mm/dma`, `mm/page_frame`, `mm/user_address`, `mm/virtual_memory`, plus the prior `common/`, `fs/cpio_newc`, `handoff/boot_info`, `proc/user_elf`, and `syscall/observe_abi`. The CMake build at [tests/host/CMakeLists.txt](../tests/host/CMakeLists.txt) compiles real kernel sources with `OS1_HOST_TEST=1` rather than reimplementing them, so the tests track the kernel.

### Biggest risks / gaps

1. **The native object/handle decision is still unmade.** Same finding as the previous two reviews. Two design drafts exist ([doc/shell-language/elf_interface_spec.md](shell-language/elf_interface_spec.md) and [doc/shell-language/os1-shell-language-first-draft.md](shell-language/os1-shell-language-first-draft.md)) that assume kernel primitives that do not exist (handles with rights bits, property dispatch, async method calls, event subscription). The kernel exposes nine syscalls, no handle table, no rights bit, no event/subscription syscall. Every milestone past D ("filesystem and multiuser") needs file descriptors *or* handles. The path that gets chosen will lock in a year of follow-on architecture. **The substrate is now mature enough that this choice will affect VFS shape, FD-vs-handle table layout, errno-vs-status convention, and the eventual POSIX shim layer simultaneously.** This finding has now appeared in *four* consecutive reviews and is the single highest-impact unmade decision in the project.
2. **No filesystem, no VFS.** [src/kernel/vfs/](../src/kernel/vfs/) and [src/kernel/security/](../src/kernel/security/) are still placeholder folders with README-only ownership. `exec` and `spawn` still resolve through [src/kernel/fs/initrd.cpp](../src/kernel/fs/initrd.cpp). The block path is now real (interrupt-driven, write-capable, request-shaped) — meaning the *next* milestone is filesystem + VFS rather than block layer. The fact that the block layer is no longer the bottleneck is the largest single shift since the prior review.
3. **No argv/envp; no FD table; no credentials.** [src/kernel/syscall/process.cpp:44](../src/kernel/syscall/process.cpp#L44) shows `sys_spawn(uint64_t user_path)` and [src/kernel/syscall/process.cpp:72](../src/kernel/syscall/process.cpp#L72) shows `sys_exec(uint64_t user_path)` — both still take only a path. The shell tokenizes ([src/user/programs/sh.cpp:938](../src/user/programs/sh.cpp#L938)) but the tokens go nowhere past the shell. `Process` still has no `creds`, no `fd_table`, no `cwd`. Every Goal D and Goal F task is blocked behind one of these. Filesystem-backed `exec` cannot pass the loaded path to the new image meaningfully without argv. Multiuser cannot exist without `uid_t`. SSH cannot exist without either.
4. **APs are still parked.** [src/kernel/arch/x86_64/cpu/cpu.cpp:16](../src/kernel/arch/x86_64/cpu/cpu.cpp#L16) is still `cli; hlt`. The synchronization vocabulary (`Spinlock`, `IrqGuard`, `OS1_BSP_ONLY`, `KASSERT_ON_BSP()`) exists but the tree-wide search shows `Spinlock` used in exactly *one* file: [src/kernel/debug/event_ring.cpp](../src/kernel/debug/event_ring.cpp). Every other BSP-only global remains single-writer-by-construction. The driver substrate that just landed routes all interrupts to the BSP APIC ID through `bsp_apic_id()` ([src/kernel/platform/pci_msi.cpp:55](../src/kernel/platform/pci_msi.cpp#L55)). This is the right shape for the parked-AP world; it is also accumulating BSP-affinity that will need explicit migration when the first AP runs real work.
5. **No kernel `kmalloc` / slab.** Every kernel allocation is page-granular through `PageFrameContainer`. The driver substrate has so far survived this by using fixed-size arrays sized at compile time (request slots, BAR claims, IRQ routes, ACPI devices, device bindings, DMA allocations). The first filesystem will want inode/dentry caches; the network stack already has a single packet buffer per direction in `virtio-net` and will want a real allocator the moment a TCP layer arrives. Naming the constraint now is cheaper than discovering it after the second subsystem has been written assuming `kmalloc` exists.
6. **`virtio-blk` is now bounded multi-sector**, not single-sector. *Resolved* on 2026-05-03: `virtio_blk_submit_request` accepts up to `kVirtioBlkMaxSectorsPerRequest = 8` sectors per request through a single contiguous data descriptor; the per-slot DMA buffer was enlarged to fit the max; the sync wrappers chunk by `device.max_sectors_per_request`; both pre-scheduler and post-scheduler smokes verify a 2-sector read and a 2-sector write/readback round-trip. Per-page user-buffer scatter-gather is the next storage extension and should land together with user-DMA pinning.
7. **xHCI smoke is single-keyboard.** [src/kernel/drivers/usb/xhci.cpp](../src/kernel/drivers/usb/xhci.cpp) brings up boot-HID keyboards (and recognizes mice) but no other USB class. The smoke matrix exercises the keyboard path only. USB hubs, mass storage, and any HID device past the first will be unexplored when they arrive.
8. **Network is one packet, not a stack.** The `virtio-net` driver works, but there is no IP layer, no UDP, no TCP, no DHCP, no DNS, no socket interface. Goal E needs all of those. The driver was the first 5%.
9. **Documentation drift around the syscall ABI** — *resolved* in the 2026-05-03 documentation cleanup pass. [src/uapi/os1/syscall_numbers.h](../src/uapi/os1/syscall_numbers.h) now records the SYSCALL/SYSRET entry mechanism explicitly; [GOALS.md](../GOALS.md) no longer lists `int 0x80` as undecided and the resolved list now cites [src/kernel/arch/x86_64/cpu/syscall.cpp](../src/kernel/arch/x86_64/cpu/syscall.cpp).

## 2. Current State From Code

### Boot path and platform assumptions

`kernel_main` ([src/kernel/core/kernel_main.cpp:262](../src/kernel/core/kernel_main.cpp#L262)) sequence is:

1. `own_boot_info()` deep-copies BootInfo into kernel BSS.
2. `cpu_initialize_record(g_cpu_boot)` → `cpu_init()` (GDT/TSS/`gs` for the BSP).
3. `page_frames.initialize(memory_regions, ...)` builds the bitmap allocator.
4. `reserve_tracked_physical_range` reserves modules and framebuffer.
5. `VirtualMemory kvm` builds the final kernel CR3: bootstrap-low identity for AP/handoff stack, kernel image by `R E`/`R`/`RW` sections, direct map, explicit module/framebuffer/RSDP windows.
6. `kvm.activate()` switches CR3, BSP CPU pointer rebound through direct map.
7. `platform_discover()` runs ACPI → AML namespace load → device inventory → `_PRT` table build → MMIO mapping for LAPIC/IOAPIC/HPET → CPU allocation → PCI ECAM enumeration. Devices are *not* probed yet.
8. `pic_init` / `ioapic_init` / `lapic_init` / `cpu_boot_others`. APs come up but park.
9. Display selection, terminal allocation, IDT registration.
10. `platform_probe_devices(kvm)` — *now* drivers bind. This is the boot-order split the implementation plan called for: probing happens after the IDT and local APIC are online, so MSI/MSI-X works on the first try.
11. Keyboard, console-input init, ISA IRQ routing for keyboard.
12. `init_tasks` allocates fixed `Process[32]` and `Thread[32]` tables.
13. `create_kernel_process` + idle thread + `load_user_program("/bin/init")` + `start_virtio_blk_threaded_smoke`.
14. `initialize_scheduler_timer()` — calibrate LAPIC against HPET, fall back to PIT.
15. `enter_first_thread`. The first thread is the block smoke when `virtio-blk` is present, otherwise `init`.

Hard platform assumptions (unchanged from prior review): x86_64 only; ACPI required; PCI is ECAM/MCFG only; `BootInfo` caps 128 memory regions and 16 modules.

### Memory management

Physical: `PageFrameContainer` bitmap allocator at `kPageFrameBitmapBaseAddress = 0x20000`, `0x40000` bytes, 8 GiB cap. No buddy/slab/freelist.

Virtual: `VirtualMemory` owns root allocation, four-level walks, NX-aware permission writes, user-slot teardown. User CR3s clone `kKernelPml4Index` and `kDirectMapPml4Index` only.

User layout unchanged: `kUserSpaceBase = 0x0000_0080_0000_0000`, `kUserStackTop = 0x0000_0080_4000_0000`, 16 user-stack pages, 4 kernel-thread-stack pages.

`mm/user_copy.cpp` validates canonical addresses, range overflow, region membership, walks page entries to confirm `User+Write`, emits `OS1_KERNEL_EVENT_USER_COPY_FAILURE` with reason codes on failure.

New since prior review: `mm/dma.cpp` and `mm/dma.hpp` add `DmaBuffer` with `owner`, `virtual_address`, `physical_address`, `size`, `direction`, `page_count`, `active` plus `dma_allocate_buffer`/`dma_release_buffer`/`dma_sync_for_device`/`dma_sync_for_cpu`. The sync calls are barrier no-ops on x86 today but the API seam is in place for IOMMU / non-coherent DMA later. Host-tested under [tests/host/mm/dma_tests.cpp](../tests/host/mm/dma_tests.cpp).

### Interrupts / exceptions

IDT, PIC remap, IOAPIC, LAPIC. The IDT now exposes dynamic external interrupt stubs over `0x50..0xef` rather than the 16 legacy slots only ([src/kernel/arch/x86_64/interrupt/vector_allocator.cpp](../src/kernel/arch/x86_64/interrupt/vector_allocator.cpp)). The interrupt callback table is vector-addressed. `irq_allocate_vector` / `irq_free_vector` is a real BSP-only bitmap.

Trap entry path remains the unified `TrapFrame` in [src/kernel/arch/x86_64/interrupt/irqhandler.asm](../src/kernel/arch/x86_64/interrupt/irqhandler.asm). The C dispatcher at [src/kernel/core/trap_dispatch.cpp](../src/kernel/core/trap_dispatch.cpp) remains a 23-line classifier.

Kernel faults dump and halt; user faults emit `TRAP` event, mark thread/process dying, reschedule. The timer hook polls serial RX, increments `g_timer_ticks`, and calls `schedule_next(true)`.

Open: still no IPI infrastructure, no per-CPU vector reservation, no priority/affinity model, no threaded-IRQ mechanism. With APs parked these are not yet load-bearing — but the moment one AP is unparked, the BSP-only routing in `pci_enable_msix_interrupt` ([src/kernel/platform/pci_msi.cpp:140](../src/kernel/platform/pci_msi.cpp#L140)) becomes a constraint that needs an explicit lift.

### Scheduling / SMP state

`Thread`/`Process` are still fixed-size in 32-entry tables. Round-robin `next_runnable_thread`. `schedule_next` records `OS1_KERNEL_EVENT_SCHED_TRANSITION` on every context switch ([src/kernel/sched/scheduler.cpp:39](../src/kernel/sched/scheduler.cpp#L39)).

APs come up to `cpu_idle_loop()` and stay there. `ThreadWaitState` is typed; `BlockIo` reason now exists, used by `block_request_wait` and the threaded virtio-blk smoke.

`Spinlock` exists; usage tree-wide remains the event ring only. `OS1_BSP_ONLY` annotates `processTable`, `threadTable`, `g_next_pid`, `g_next_tid`, `g_idle_thread`, `terminal[]`, `active_terminal`, `console_input` queue, `page_frames`, `g_kernel_root_cr3`, `g_platform`, the IRQ route table, the device-binding table, the BAR-claim table, and the DMA-allocation table.

### User/kernel separation state

Same vertical slice as the prior review. ELF64 `ET_EXEC` from initrd, ring-3 entry via `iretq`, SYSCALL/SYSRET fast path with `IA32_LSTAR` set up in [src/kernel/arch/x86_64/cpu/syscall.cpp](../src/kernel/arch/x86_64/cpu/syscall.cpp).

Syscall ABI unchanged: `write`, `exit`, `yield`, `getpid`, `read`, `observe`, `spawn`, `waitpid`, `exec`. `observe` now serves nine record kinds (was six). `spawn`/`exec` still accept only a path.

Notable absences (unchanged from prior review and in the same priority order):

- No file descriptors / handle table.
- No argv/envp on the initial user stack.
- No `fork`. `spawn(path)` is the only child primitive.
- No signals or event-delivery primitive.
- No credentials surface.
- No `dup`/`pipe`/`open`/`close`.
- No filesystem-backed `exec`.
- No errno; failure is uniform `-1`.
- No `mmap` / anonymous memory; no user heap.

### Storage / filesystem state

`virtio-blk` is now the project's first complete driver: shared transport, DMA buffers, MSI/MSI-X with INTx fallback, interrupt-driven completion through `virtio_blk_drain_used`, request slot table sized by queue capacity, both reads and writes, hot-remove path that releases interrupt + DMA + BARs + IRQ routes + binding record.

`BlockDevice` is now `BlockRequest`-shaped with `submit`/`flush` callbacks. `block_read_sync` / `block_write_sync` are wrappers for early callers and the threaded smoke. The pre-scheduler boot completion still spins (`virtio_blk_wait_for_boot_completion`), but that path is now narrowly scoped to "no thread yet" — the post-scheduler `virtio-blk threaded smoke` exercises the threaded completion via `BlockIoWaitState` and the typed `wake_block_io_waiters` path.

The smoke writes a scratch sector (`OS1 VIRTIO WRITE PATH OK`) at sector 2 and reads it back to validate the round-trip ([src/kernel/drivers/block/virtio_blk.cpp:411-432](../src/kernel/drivers/block/virtio_blk.cpp#L411)). This is the project's first end-to-end I/O round-trip on a real device path.

Filesystem and VFS remain placeholder folders. There is no FAT/ext/native filesystem. The VFS README correctly identifies this as the next ownership home.

Bounded multi-sector now works: `virtio_blk_submit_request` accepts `1 <= sector_count <= kVirtioBlkMaxSectorsPerRequest` (= 8 sectors / 4 KiB) and programs a single multi-sector data descriptor. The synchronous wrappers chunk larger transfers by `device.max_sectors_per_request`. The next storage extension is per-page user-buffer scatter-gather, gated on user-DMA pinning.

### Drivers / device model — depth on the new pieces

Three subsystems are worth a closer look because their architecture, not just their existence, will shape the next year of work:

**`virtio-blk`** ([src/kernel/drivers/block/virtio_blk.cpp](../src/kernel/drivers/block/virtio_blk.cpp)). The structure is now exemplary for future virtio drivers: `VirtioBlkState` owns transport, queue, request slots, capacity; `VirtioBlkRequestSlot` owns one `DmaBuffer` + header pointer + data pointer + status pointer + the `BlockRequest*` it is currently serving; descriptor allocation is bitmap-tracked through `descriptor_in_use_mask`; `virtio_blk_irq` calls `virtio_blk_drain_used` which walks the used ring, resolves each used element back to its slot, copies read data, releases the slot, records `OS1_KERNEL_EVENT_BLOCK_IO` with reason flags, and signals completion. Bounded multi-sector requests up to `kVirtioBlkMaxSectorsPerRequest = 8` are accepted; the per-slot DMA buffer is sized for the max and the data descriptor's `len` varies per request while the status byte sits at a slot-fixed offset. The hot-remove path in `remove_virtio_blk_device` releases interrupt → DMA buffers → virtqueue memory → BAR claims → IRQ routes → binding record → block facade in the documented order.

**`virtio-net`** ([src/kernel/drivers/net/virtio_net.cpp](../src/kernel/drivers/net/virtio_net.cpp), 529 LOC). Reuses the same shared transport, the same DMA buffer type, the same MSI-X allocator. Notable structural shortcut: only one packet buffer per direction (`rx_buffer`, `tx_buffer`). This is fine for the ARP probe smoke and is honest about being a smoke driver. A real network stack will need a packet buffer pool keyed on completion. The driver's RX completion records `OS1_KERNEL_EVENT_NET_RX` with packet length and ethertype, which is the right shape for a protocol stack to consume later.

**xHCI** ([src/kernel/drivers/usb/xhci.cpp](../src/kernel/drivers/usb/xhci.cpp), 1484 LOC + [xhci_controller.cpp](../src/kernel/drivers/usb/xhci_controller.cpp), 525 LOC + [hid_keyboard.cpp](../src/kernel/drivers/usb/hid_keyboard.cpp), 134 LOC). The split is clean: `xhci_controller.cpp` owns capability/operational/runtime register decoding, ring management primitives, controller reset; `xhci.cpp` owns the PCI binding, DMA-backed DCBAA / scratchpad / command-ring / event-ring / ERST allocation, root-port enumeration, slot assignment, control transfers (`SETUP`/`DATA`/`STATUS` TRBs), HID interface parsing; `hid_keyboard.cpp` owns boot-protocol report decoding into the canonical console input. The interrupter-0 IRQ goes through the same `pci_enable_best_interrupt` as `virtio-blk` and `virtio-net` — there is one IRQ allocation path in the kernel, used by all three drivers. That is precisely the substrate-reuse the implementation plan promised.

The driver substrate itself is a small and consistent surface:

- `drivers/bus/device.cpp` (87 LOC): device binding table, `device_binding_publish`/`set_state`/`remove`.
- `drivers/bus/driver_registry.cpp` (85 LOC): static PCI driver registry; `pci_driver_matches_device` does vendor+device or class/subclass/prog-if matching.
- `drivers/bus/pci_bus.cpp` (55 LOC): probe loop that walks enumerated PCI devices, matches against drivers, calls `probe`.
- `drivers/bus/resource.cpp` (111 LOC): `claim_pci_bar`/`release_pci_bar`/`release_pci_bars_for_owner`.

That's 338 LOC for the entire bus/device/registry layer — small, coherent, host-tested, and already exercised by three drivers.

## 5. Overlooked Goals / Blind Spots

### The native-interface decision now blocks four downstream choices simultaneously

In the prior reviews this decision was about FDs vs handles for a future filesystem. After the 2026-04-30 substrate landed, it now also drives:

- **VFS shape**: do mounts produce FDs (Unix), root handles (Plan 9), or capability-bearing object handles (Fuchsia, the shell-language draft)?
- **Network sockets**: socket-as-FD vs socket-as-handle is the same question and the kernel will ship one of the two soon.
- **Device handles**: `devices`/`resources`/`irqs` records are read-only today. The moment they grow a write or method-call path, they need a handle/FD model.
- **POSIX shim direction**: handles-first means a POSIX shim layer maps `int fd` to `Handle*`; FDs-first means a handle ABI is layered above the FD table later.

A two-page design doc that names the chosen direction is still the most valuable single artifact the project can produce in the next week. The cost of *not* deciding has gone up since the prior review because more substrate now hangs off the answer.

### The block layer is no longer the bottleneck — the filesystem is

The implementation plan's "definition of done" is essentially met for the driver/device/platform substrate. The next real engineering work is filesystem + VFS + FD/handle table + filesystem-backed exec + multiuser identity. None of this has started in code. The fact that the block layer is *done enough* changes which item is highest leverage; FS is now it.

### `kmalloc`/slab is now near-term, not long-term

Every kernel allocation is page-granular. The current driver substrate survives this only because every fixed structure is sized at compile time. The first inode cache will exhaust this assumption. The first TCP retransmit queue will exhaust it harder. Naming the constraint now (so no one designs a filesystem on the assumption a small-object allocator exists) costs a paragraph; discovering it after writing one costs a rewrite.

### Per-CPU IRQ steering will need an explicit lift

`pci_enable_msix_interrupt` and `pci_enable_msi_interrupt` both compute the destination through `bsp_apic_id()`. This is correct today and will be wrong the day an AP needs to take a `virtio-net` RX completion. The IRQ resource record already has space for an affinity field; adding it now is a no-op that documents intent.

### A multi-sector / scatter-gather path is the next BlockRequest contract step

`BlockRequest` carries `sector_count` but the driver rejects anything other than 1. A filesystem will want at least multi-sector reads; a buffer cache will want scatter-gather. The contract supports both; the driver does not. This is a small concrete next step.

### Initrd manifest assertion: documented, not enforced

`/bin/ascii` is now documented at [doc/ARCHITECTURE.md:903](ARCHITECTURE.md#L903) as a human-only visual probe deliberately excluded from the headless smoke matrix. That closes the prior carry-forward. The remaining loose end is broader: the project still has no automated assertion that the staged binary set in the initrd matches what `observe initrd` reports, so a future binary that vanishes between build and stage would not be caught by CI. This is a different (and milder) finding than the original `/bin/ascii` gap and should be tracked as a separate item rather than carried forward as the same one.

### A test-program that exercises the new substrate would be cheap and informative

Today the QEMU smoke matrix asserts strings on serial. There is no user-mode test program that exercises the new observe records (`devices`, `resources`, `irqs`, `events`) end-to-end. A small `/bin/diag` or `/bin/test` that walks the records, prints a digest, and exits with 0/1 would catch silent regressions in observe ABI and would be the first user program to exercise more than one syscall in a meaningful way.

### Portability boundaries are still unlabelled

GOALS.md scopes AArch64 and RISC-V as future ports. The current code makes that aspirational. PML4-based 4-level paging, ACPI as the only discovery source, PCI universally, SYSCALL/SYSRET, PIT/PIC/IOAPIC/LAPIC, 4 KiB pages — none of these need to be portable today, but they should be *labelled* as x86-isms. Carry-forward from prior review.

### USB hub class is the next USB tier

The xHCI driver enumerates connected root ports but does not handle hubs. The first attached USB hub will be invisible. This is a small targeted next step that matches the GOALS.md "USB host-controller discovery via PCI" sequence.

### Network stack should be designed for non-blocking from day one

The `virtio-net` driver is interrupt-driven and the kernel event ring records RX. A future TCP layer must not assume the spinning, single-buffered RX path that the smoke uses; the contract should be "complete via callback or wakeup" the same as `BlockRequest`. Naming this expectation in a one-page network plan now is much cheaper than rewriting after the first protocol layer assumes blocking.

## 6. Roadmap

Ordered by dependency, not attractiveness. Items that closed in the 2026-04-30 pass are dropped; items that remain are repeated; items that have grown in priority since the prior review are noted.

### Near-term (next 1–3 milestones)

1. **Decide the native-interface stance** (handles-first, FDs-first, or both side-by-side). Capture as a two-page design doc that names which kernel structures get added (`Handle` vs `FileDescriptor`), where rights bits live, what `read`/`write` look like across both, how `observe` evolves, how the shell-language draft consumes it. *Why now:* this decision now blocks VFS shape, sockets shape, device-handle shape, and the POSIX-shim direction simultaneously. Fourth consecutive review carrying this finding; substrate is now mature enough that the cost of choosing wrong is higher than ever.
2. **Land argv/envp on the initial user stack.** Tiny ABI step. Lets `/bin/sh` invoke commands with arguments, lets `/bin/init` evolve apart from `/bin/sh`, unblocks meaningful filesystem-backed exec when it arrives. *Why now:* small, isolated, decoupled from §1, and explicitly listed in the GOALS.md M5 follow-ups. Fourth consecutive review carrying this finding.
3. **Define and implement filesystem-backed `exec` on top of a first VFS.** The block layer is now real; the filesystem is the next bottleneck. Choose a first FS (FAT32 read-only or a simple bespoke FS), introduce a minimal VFS that supports `open`/`read`/`close`-or-handle-equivalent (per §1), make `exec` go through it. Keep initrd as a fallback. *Why now:* the block layer that used to block this is no longer a constraint. This is the shape of Goal D's userland half.
4. **Add a kernel small-object allocator (`kmalloc` or slab).** Naming and implementing the constraint before the first inode cache is much cheaper than after. *Why now:* every prospective subsystem (FS, network, USB hub, multi-vector IRQ) wants this.
5. **Multi-sector `virtio-blk`.** *Resolved* on 2026-05-03. The driver now accepts `1 <= sector_count <= 8` per request through a single multi-sector data descriptor; the synchronous wrappers chunk by `device.max_sectors_per_request`; both pre-scheduler and post-scheduler smokes verify a 2-sector read and a 2-sector write/readback. Per-page user-buffer scatter-gather and DMA pinning are the next storage extension; they should land together because they share the same lifetime question.
6. **Begin the multiuser foundation.** uid/gid types, credential field on `Process`, file ownership in the new FS, default permission policy. Aim for "single user, but the right shape." *Why now:* this is the half of GOALS.md Milestone D that is *not* the block layer; SSH cannot exist without it.
7. **Documentation cleanup pass.** *Resolved.* The 2026-05-03 cleanup pass updated [src/uapi/os1/syscall_numbers.h](../src/uapi/os1/syscall_numbers.h), [GOALS.md](../GOALS.md), [README.md](../README.md), [doc/ARCHITECTURE.md](ARCHITECTURE.md), and [doc/REFERENCES.md](REFERENCES.md) so all five tracks the substrate that landed on 2026-04-30. The `BootInfo` cap residue (128 memory regions, 16 modules) remains and should be either documented as a deliberate teaching simplification or grown.
8. **Add a generic initrd-manifest assertion to the smoke matrix.** The `/bin/ascii` half of this finding is now closed by [doc/ARCHITECTURE.md:903](ARCHITECTURE.md#L903) (human-only visual probe; deliberately not asserted). What remains is the broader assertion that the staged binary set matches what `observe initrd` reports, so a future binary that fails to stage cannot disappear silently. *Why now:* the original prompt is closed; this is the residue and is small enough to land in the same pass.

### Mid-term (after the near-term cluster lands)

1. **Begin the network protocol stack on top of `virtio-net`.** ARP table, IPv4, ICMP echo, UDP. Defer DHCP and DNS until configuration storage exists. The packet-path contract should mirror `BlockRequest`: completion via callback or wakeup, owner-tagged DMA, MSI-X-driven RX. *Why now:* the driver is in; the substrate is in; the only missing piece is the protocol stack.
2. **Bring up one AP for restricted work.** With the synchronization contract in place and the §1-decided handle/FD model, run timer accounting or a single virtio-net RX queue on a single AP. Do not flip all APs into the global scheduler at once. Migration order is in [doc/2026-04-29-smp-synchronization-contract.md](2026-04-29-smp-synchronization-contract.md).
3. **Per-CPU IRQ steering.** Add an affinity field to the IRQ resource record. Steer one queue to one AP. *Why now:* cheap once an AP runs anything non-trivial; needed before networking matures.
4. **USB hub class and HID mouse routing.** Extend the xHCI work with a hub driver and route mouse reports past recognition into a future input event queue. *Why now:* small, matches GOALS.md sequence, makes USB cover most real keyboards/mice.
5. **TCP and a minimal sockets API (or handle equivalent).** Once §1 is decided, implement the smallest TCP that can sustain SSH eventually. Errno or status discipline.
6. **Native interface kernel surface.** Once §1's decision is implemented and FDs/handles exist, expose properties, methods, and events at a kernel level. Build the user-side runtime that loads `.os.interface` sections and turns them into typed objects. The shell-language draft can then become a real interpreter.
7. **First persistent on-disk filesystem write path.** Once read works and a buffer cache exists, add the journaled or copy-on-write write path appropriate to the chosen FS.

### Later

1. **DHCP, DNS resolver, persistent network config.** Once filesystem exists.
2. **Entropy source, basic crypto.** Once writable filesystem exists for key storage.
3. **PTYs, sessions, login.** Once credentials and FD/handle abstractions are stable.
4. **SSH.** After all of the above.
5. **GUI.** After everything above. The framebuffer text path remains plenty for a long time.
6. **Accelerator support.** After IOMMU, DMA pinning, MSI-X (already done), and a user/kernel submission ABI exist.

### Explicit prerequisites for major goals (carry-forward, lightly updated)

- **Before networking maturity (DHCP/DNS/SSH-grade):** §1 native-interface decision, FD/handle table, IPv4/UDP/TCP, persistent storage for config.
- **Before SSH:** networking through TCP, entropy, crypto, credentials, PTYs/sessions, persistent storage, reliable process I/O.
- **Before GUI:** input event abstraction, framebuffer ownership rules, process/session model, scheduler stability, graphical buffer ownership.
- **Before SMP scheduling:** lock + preemption rules (now present), audited globals (now annotated), per-CPU runqueue (or one global lock), IPI/wakeup path, AP-safe console + allocator + page tables, per-CPU IRQ steering.
- **Before POSIX or handle growth:** §1 decision, FD-or-handle table, path namespace, argv/envp, errno or equivalent error contract, process lifecycle semantics.
- **Before accelerator support:** PCI resource ownership (done), DMA allocation (done), MSI-X (done), IOMMU, user/kernel submission queues, fault/reset path.
- **Before native object interface:** §1 decision, handle/rights primitive in kernel, event/subscription syscalls, ELF `.os.interface` loader in user runtime.
