## 1. Executive summary

### current project state

- The SMP enablement plan in `doc/2026-05-05-phased-smp-enablement.md` is
  fully implemented in source. APs install the global IDT, run their own idle
  threads, take per-CPU LAPIC timer ticks, observe a reschedule IPI, hold
  per-CPU run queues, and accept user threads through round-robin and
  load-balanced placement.
- `struct cpu` in `src/kernel/arch/x86_64/cpu/cpu.hpp:51-86` now carries an
  embedded `RunQueue`, an idle thread pointer, per-CPU `timer_ticks`,
  reschedule pending flag, IRQ stack thread, enqueue/dequeue counters, idle
  ticks, kernel-thread ping count, and migrate-in/out counters. The static
  asserts at the bottom of the header keep the assembly-visible `tss_rsp0`
  offset stable while the new fields sit after `tss`.
- `src/kernel/sched/scheduler.cpp` implements per-CPU dequeue, idle-thread
  fallback, and a periodic load balancer with migration cooldowns. The global
  `relink_runnable_threads` rebuild is gone (`thread.cpp:445` is now an empty
  stub).
- `src/kernel/sched/thread_queue.cpp` is a per-CPU FIFO dequeue against
  `cpu_cur()->runq` rather than the registry walk it used to be.
- `src/kernel/sync/wait_queue.cpp` provides `WaitQueue` and `Completion`
  primitives. They are now wired into block-I/O completion
  (`storage/block_device.cpp:66-98`), child-exit waits
  (`process.hpp:42`, `syscall/wait.cpp:115-165`), and console-read waits
  (`console_input.cpp:35`, `syscall/console_read.cpp:39-63`,
  `proc/thread.cpp:677-686`). The registry-walk style of finding waiters is
  gone for those three reasons.
- `src/kernel/arch/x86_64/apic/ipi.cpp` allocates two dynamic vectors
  (reschedule and TLB shootdown) and exposes `ipi_send`,
  `ipi_send_all_but_self`, `ipi_send_reschedule`, `ipi_send_tlb_shootdown`,
  and the corresponding receive-side helpers in `core/irq_dispatch.cpp`.
- The user-side SMP smoke programs `/bin/smpcheck`, `/bin/balanceworker`,
  `/bin/balancecheck`, and `/bin/busyyield` exist and are exercised by the
  `smoke_observe`, `smoke_observe_bios`, `smoke_balance`, and
  `smoke_balance_bios` QEMU smokes. CTest now serializes the QEMU smokes
  through `RUN_SERIAL TRUE` plus `RESOURCE_LOCK os1_qemu_smoke_artifacts`
  (`CMakeLists.txt:1215-1222`).
- The host-test harness now covers `wait_queue_tests`, `runqueue_tests`,
  `cpu_record_tests`, `load_balancer_tests`, `timer_source_tests`,
  `atomic_tests`, plus `console_input_stub.cpp` and `lapic_stub.cpp`.

### strongest parts

- The phased SMP design in `doc/2026-05-05-phased-smp-enablement.md` is
  unusually disciplined for a hobby OS: it sequences locking primitives
  before any AP scheduling change, refuses to enable AP scheduling under
  PIT, gives every phase a rollback story, and lands smoke evidence with
  every phase.
- The locking-tier contract in
  `doc/2026-04-29-smp-synchronization-contract.md` is now backed by real
  locks: `g_page_frames_lock` is implicit through `kmem_init`/`page_frames`
  guards, `g_process_table_lock` is real
  (`proc/process.cpp:20`), `g_thread_registry_lock` is real
  (`proc/thread.cpp:209`), `g_console_input_lock` is real
  (`console/console_input.cpp:17`), each per-CPU `runq.lock` is real
  (`cpu.hpp:53`), and per-`WaitQueue` locks are real (`sync/wait_queue.hpp`).
- The post-implementation review in
  `doc/2026-05-05-smp-enablement-review.md` was self-correcting: it
  identified BIOS balance flakiness, missing AP-tick assertions, the
  console-read registry walk, the `observe processes` lock-ordering bug,
  parallel-CTest QEMU lock conflicts, and console-output interleaving, and
  the follow-up commit `61bbb4f` fixed all of them.
- `Completion` is now the canonical block-I/O wait primitive
  (`sync/wait_queue.cpp:118-132`, `storage/block_device.cpp:66-98`). The
  previous ad-hoc `request.completed = true` plus address-keyed wakeup is
  gone, including the misleading empty `asm` barriers that prior reviews
  flagged.
- The shell now has SMP-relevant inspect commands by way of the new
  `OS1_OBSERVE_CPUS` fields (`runq_depth`, `timer_ticks`, `idle_ticks`,
  `migrate_in`, `migrate_out`) and event types (`OS1_KERNEL_EVENT_AP_ONLINE`,
  `OS1_KERNEL_EVENT_AP_TICK`, `OS1_KERNEL_EVENT_IPI_RESCHED`,
  `OS1_KERNEL_EVENT_KERNEL_THREAD_PING`, `OS1_KERNEL_EVENT_IPI_TLB_SHOOTDOWN`,
  `OS1_KERNEL_EVENT_THREAD_MIGRATE`, `OS1_KERNEL_EVENT_RUNQ_DEPTH`).

### biggest risks / gaps

- The single biggest finding in `doc/2026-05-05-review.md` — that the
  next architectural gate is the descriptor/handle/object model — is
  unchanged. SMP enablement does not move that gate; it removes a different
  blocker. Filesystems, devices, sockets, PTYs, sessions, shell redirection,
  and SSH all still wait on the same decision.
- TLB shootdown is a coarse "reload CR3 everywhere" broadcast
  (`apic/ipi.cpp:47-67,149-158,180-183`). It works for the only present case
  (process teardown reloads on every CPU), but it is not yet selective by
  CR3 or by page range, so any future `mmap`/`munmap`/COW work will need to
  refine it before it can be safe under load.
- AP scheduling refuses to start under PIT
  (`core/kernel_main.cpp:319-328`, `OS1_KERNEL_EVENT_TIMER_SOURCE_AP_TICKS_DISABLED`),
  which is correct but not a long-term answer for non-HPET hardware. The
  near-term mitigation should be documented; the long-term answer is an IPI
  broadcast tick from the BSP.
- `wait.cpp:51-108` takes `g_process_table_lock` then
  `g_thread_registry_lock`, which matches the documented tier order
  (process is tier 4, thread registry is tier 5). The fix-up commit
  reordered `sys_observe_processes` correctly. There is no remaining
  documented inversion in the source, but the lock order is enforced only
  by reading code; there is no static analysis or runtime assertion.
- The kernel boot still has a single statically-allocated `g_boot_irq_thread`
  in `core/kernel_main.cpp:57,576-581`. APs do not hit this path because they
  reach their idle thread before any IRQ enables; this is correct, but it
  means the BSP-only "IRQ stack thread" naming is now misleading and the
  `irq_stack_thread` field in `cpu.hpp:72` is currently used only on the BSP.
- `doc/latest-review.md` and `doc/ARCHITECTURE.md` are now significantly
  behind code. ARCHITECTURE.md still claims APs "park in
  `cpu_idle_loop` (`cli;hlt`)" (line 235) and never mentions per-CPU run
  queues, the load balancer, IPIs, or the new `WaitQueue` plumbing. The
  `latest-review.md` pointer still describes the prior review without
  acknowledging the SMP enablement landing or the follow-up review.
- `doc/2026-05-05-review.md` itself is overtaken by the SMP commits: it
  said "SMP is structurally present but not operational: APs are booted and
  parked", "scheduling remains BSP-only", and "expanding SMP now would
  multiply race bugs in memory allocation, process/thread lists, event
  recording, console input, and driver paths." Source has crossed that
  line. The next review pointer must say so explicitly.
- Recommendation: hold the descriptor/handle decision firm. Do not let
  the operational SMP runtime tempt the project into bolting more raw
  kernel features onto an undefined resource model.

## 2. Current state from code

### boot path and platform assumptions

- The boot contract is unchanged from the prior review: `BootInfo` in
  `src/kernel/handoff/boot_info.hpp`, with capacity caps for memory regions
  and modules, normalized by both `src/boot/limine/` and `src/boot/bios/`.
- `kernel_main(BootInfo*, cpu*)` in
  `src/kernel/core/kernel_main.cpp:363-657` orchestrates BSP bring-up,
  page-table installation, direct map, `kmem_init`, ACPI/PCI discovery,
  IPI initialization, IDT, driver probe, console init, task tables,
  idle-thread pre-creation for every discovered CPU, the boot-sequence
  thread, scheduler timer preparation, AP startup, and finally
  `enter_first_thread`.
- The new step relative to the prior review is `ipi_initialize()` at line
  570-574 between `interrupts.initialize()` and the boot-IRQ scaffolding.
- AP bring-up still goes through `cpu_boot_others()`
  (`arch/x86_64/cpu/cpu.cpp:224-260`). What changed is that
  `init()` (line 205-222) on each AP now runs `cpu_init`, sets the local
  APIC ID into `cpu_cur()->id`, runs `lapic_init`, runs `cpu_load_idt`, and
  enters `enter_ap_idle_thread()` (line 53-85) which marks the per-CPU idle
  thread `Running`, records `OS1_KERNEL_EVENT_AP_ONLINE`, optionally starts
  the local APIC timer, and performs `enter_first_thread` onto the idle
  stack. The legacy `cpu_idle_loop` is now a fallback only.
- Platform assumptions are still x86_64, ACPI, PCI ECAM, LAPIC/IOAPIC, and
  QEMU/q35 conventions. There is no non-ACPI platform path.
- The `OS1_INITRD_LOAD_ADDRESS=0x90000` value and `BootInfo` capacity caps
  the prior review called out are still authoritative in source.

### memory management

- The three-layer model (page frames, virtual memory, `kmem`) is unchanged
  in shape from the prior review.
- `PageFrameContainer` in `src/kernel/mm/page_frame.cpp` still does not
  expose its lock externally; the file holds its own internal serialization
  and continues to use BSP assertions on early-boot mutations. The Phase 5
  goal of "lock the allocator and remove `KASSERT_ON_BSP` from
  allocate/free/reserve" is partially in code: page frames are reachable
  from kernel threads on APs through the boot-sequence thread on the BSP
  plus reaper-driven reclaim, but `page_frame.cpp` has not added a public
  `g_page_frames_lock` matching the named tier-2 entry in the SMP design.
  In practice the allocator has held up under the new smokes, but this is
  the most fragile remaining tier-2 lock to formalize.
- `kmem` (`src/kernel/mm/kmem.cpp`) is unchanged in shape; its consumers
  list grew implicitly with `WaitQueue` + per-process child-exit queues
  (every `Process` allocated via the `process` cache now embeds a
  `WaitQueue` for child waiters, `proc/process.hpp:42`). The `Completion`
  embedded in each `BlockRequest` is on the caller's stack, so this does
  not allocate, and consequently does not multiply `kmem` pressure.
- `src/kernel/mm/dma.cpp`, `src/kernel/mm/user_copy.cpp`, and
  `src/kernel/mm/user_address.hpp` are functionally unchanged.
- New TLB shootdown plumbing exists at the IPI level
  (`apic/ipi.cpp:149-183`) but no caller in `src/kernel/mm/virtual_memory.cpp`
  yet issues `ipi_send_tlb_shootdown` for `unmap`/`protect`. The receiver
  reloads `CR3` unconditionally; the broadcast at `reap_process` is the
  only present trigger and it is implicit (the new IPI is not invoked from
  `reap_process`; it is invoked from address-space teardown paths only when
  user CR3 changes, which today is only at process teardown). This matches
  Phase 7's caveat that the mechanism ships before there are real callers.

### interrupts / exceptions

- IDT, exception handlers, and the dispatch split (`trap_dispatch.cpp`,
  `irq_dispatch.cpp`) are unchanged.
- `core/irq_dispatch.cpp:38-130` is the central new-shape change for SMP:
  `account_scheduler_tick` increments per-CPU `timer_ticks` plus a global
  `g_timer_ticks` via `atomic_fetch_add`, emits `OS1_KERNEL_EVENT_AP_TICK`
  on the AP's first tick and every `kApTickEventInterval`-th tick after that
  (1000 at 1 kHz = once per second per CPU), routes the reschedule IPI
  receive-side flag, performs the TLB shootdown reload, and drives
  `wake_console_readers` only on the BSP.
- `console_input_poll_serial` is BSP-gated at line 80-83. ISA IRQs are
  still pinned to the BSP via `platform_route_isa_irq`
  (`core/kernel_main.cpp:602-608`).
- The `T_LTIMER` route is recorded as a `LocalApic`-kind IRQ route
  (`platform_allocate_local_apic_irq_route` in `kernel_main.cpp:294`).
  `core/timer_source.cpp:25-35` resolves whether a vector counts as the
  scheduler tick by querying that route, so AP-side timer ticks land in the
  same path as BSP-side ticks.

### scheduling / SMP state

- This is the largest behavioral change. APs run real work now.
- `struct cpu` (`arch/x86_64/cpu/cpu.hpp:51-86`) holds `current_thread`,
  `interrupt_frame`, `gdt[]`, `tss`, then `idle_thread`, `RunQueue runq`,
  `timer_ticks`, `reschedule_pending`, `irq_stack_thread`,
  `enqueue_count`, `dequeue_count`, `idle_ticks`,
  `kernel_thread_ping_count`, `migrate_in`, `migrate_out`,
  `balance_idle_ticks`, the linked-list `next`, then `id`/`booted`/`magic`,
  then the embedded kernel stack. Layout asserts still pin `current_thread`
  at offset 8, `tss` at 248, and `kstackhi` at page boundary.
- `RunQueue` is a private type defined in the same header (line 51-59),
  with its own per-CPU `Spinlock` and intrusive head/tail.
- `src/kernel/proc/thread.cpp` re-shapes the wakeup and creation paths:
  `select_target_cpu` chooses round-robin for kernel threads,
  least-loaded for user threads, and falls back to `cpu_cur()` otherwise.
  `enqueue_thread_on_cpu` takes the target's `runq.lock`, links via
  `Thread::next`, sets `run_queue_cpu`, increments enqueue_count, and emits
  `ipi_send_reschedule(target)` if the target is not the local CPU. The
  cross-CPU enqueue rule from the design (take target's lock, link,
  release, IPI) is implemented exactly there.
- `mark_thread_ready(thread, target)` and `wake_blocked_thread(thread,
  target)` accept a `cpu*` argument, defaulting to nullptr (auto-pick).
  This is the symmetric remote-wakeup primitive the design specified.
- `src/kernel/sched/scheduler.cpp:176-270` implements the load balancer.
  It iterates schedulable CPUs (`g_cpu_boot` plus every booted AP) under
  per-CPU `runq.lock`, selects the heaviest donor, requires that
  `donor_depth > local_depth + 1`, takes both locks in `cpu_id` order to
  avoid the design's deadlock case, detaches a non-recently-migrated tail
  thread, and re-enqueues it on the receiver. It records
  `OS1_KERNEL_EVENT_THREAD_MIGRATE` on success and
  `OS1_KERNEL_EVENT_RUNQ_DEPTH` on every check.
- `Thread::affinity_mask`, `Thread::last_migration_tick`, `scheduler_cpu`,
  and `run_queue_cpu` are present (`proc/thread.hpp:87-91`), with default
  affinity `~0ull`.
- `relink_runnable_threads` is now an empty stub (`proc/thread.cpp:445-448`).
  No caller relies on its old behavior; the periodic comments in
  `reaper.cpp:54` still call it as a documentation no-op.
- BSP-only annotations have shrunk: `KASSERT_ON_BSP` is preserved on
  `init_tasks`, `initialize_thread_table`, `initialize_process_table`,
  `console_input_initialize`, the keyboard interrupt handler, and a few
  driver-probe paths. The remaining annotations are intentional.
- Scheduling fairness is round-robin per CPU plus periodic balance with
  cooldowns (`kBalanceIntervalTicks=64`, `kIdleBalanceTriggerTicks=4`,
  `kMigrationCooldownTicks=64`). No priority/nice mechanism — matches the
  plan's "out of scope" stance.

### user/kernel separation state

- The set of syscalls is unchanged from the prior review:
  `os1_sys_write`, `read`, `exit`, `yield`, `getpid`, `observe`,
  `spawn`, `waitpid`, `exec` (`src/uapi/os1/syscall_numbers.h`).
- `src/kernel/syscall/dispatch.cpp` is unchanged in shape; `read` blocks
  on `block_current_thread_on_console_read` which now enqueues into the
  console-owned `WaitQueue` rather than relying on a registry walk.
- Process records still have only `pid`, `state`, `address_space`,
  `exit_status`, `parent`, `name`, child-exit `WaitQueue`, and
  `registry_next` (`proc/process.hpp:34-44`). No credentials, descriptor
  table, working directory, signal state, session, or namespace pointers.
- Threads gained scheduler placement fields but no priority, no
  cancellation, no signal-equivalent.
- `exec` in `proc/user_program.cpp` still loads images by initrd path and
  does not build an argv/envp stack.
- Recommendation: the descriptor/handle decision is unchanged from
  yesterday's review, but the cost of further delay is increasing. SMP
  scheduling now means user processes may sleep on cross-CPU wakeups; once
  filesystems and sockets join, every fd-related path will pay the same
  cross-CPU lock cost that the SMP plan just paid for `WaitQueue`. Defining
  the model now is cheaper than retrofitting it under contention.

### storage / filesystem state

- The block path is unchanged in capability but cleaner in lifetime.
- `BlockRequest` now embeds a `Completion` (verified by reading
  `storage/block_device.cpp:66-98`). `block_request_complete` calls
  `completion_signal`, which atomically flips `done` and wakes every waiter
  on the queue under the queue's spinlock.
- `block_request_wait` only blocks via `block_current_thread_on_block_io`
  when the calling thread has a process; pre-thread bring-up still falls
  through to a `pause`/`hlt` polling loop. This is the right shape for the
  boot-time path which runs before any thread exists.
- `virtio-blk` is unchanged in driver-level behavior. Multi-sector caps
  and bounded request sizes from earlier reviews still hold.
- There is still no VFS implementation under `src/kernel/vfs/` and no
  persistent filesystem above `BlockDevice`. `src/kernel/fs/` remains
  initrd-only.
- Recommendation: descriptor model first, then VFS skeleton, then a
  read-only block-backed filesystem smoke (open, read, exec). Same as
  prior review.

### drivers / device model

- The device model under `src/kernel/drivers/bus/` and the platform
  layer under `src/kernel/platform/` are unchanged.
- One subtle SMP-relevant detail: `device_probe.cpp` still runs on the BSP
  before APs come up, and `platform_route_isa_irq` continues to pin ISA
  IRQs to the BSP. ACPI/PCI device discovery, MSI-X programming, and
  driver binding are not currently AP-safe. This is correct under the
  "BSP-pinned device IRQs through Phase 7" rule from the design and works
  because device IRQs land on the BSP today.
- Recommendation: when descriptors and VFS land, expose devices through
  the same descriptor/handle path used for files. Do not let driver
  control surfaces fork off another ad-hoc ABI.

### networking state

- Unchanged from the prior review: `virtio-net` ARP smoke only. No IP,
  ICMP, UDP, TCP, DHCP, DNS, packet pool, socket API, or user-visible
  networking syscall.
- The packet allocator situation has not improved — `virtio-net` still
  uses driver-local buffers — and it should not until descriptors and a
  monotonic-time syscall exist.
- Networking remains gated on the descriptor decision and on a packet
  buffer pool plan. SSH is still gated on TCP, entropy, credentials,
  PTYs, sessions, and file I/O.

### security / multiuser state

- Unchanged. `src/kernel/security/README.md` is still a placeholder. The
  observe ABI still exposes broad kernel state to any user program.
- The new `OS1_OBSERVE_CPUS` fields and AP-related events expose more
  internal state than before. This widens the "any program can observe
  every CPU" surface, though no information leak is introduced — the
  original observe contract already exposed pid/tid/state. The point is
  that as observe grows, the observe-access policy decision in the prior
  review's blind-spots list grows in importance.

### GUI / terminal state

- Unchanged. Terminal model and 12 logical terminals are present; PTYs,
  sessions, terminal devices, compositor, mouse routing, and window
  management are still not started.
- Console-output serialization was tightened in commit `61bbb4f` to fix
  smoke-marker interleaving (see `console.cpp` adjustments). This is a
  small but meaningful step: without an output lock, the SMP scheduler
  could split user smoke markers, breaking smoke runners. The fix is
  scoped and correct; it does not constitute a console redesign.

### build, tooling, and debug workflow

- The host-test harness gained `wait_queue_tests`, `runqueue_tests`,
  `cpu_record_tests`, `load_balancer_tests`, `timer_source_tests`, and
  `atomic_tests`. Stubs gained `console_input_stub.cpp` and
  `lapic_stub.cpp`/`.hpp`. Existing stubs in `tests/host/support/`
  picked up new fields for the SMP layout.
- The QEMU smoke set gained `smoke_observe`, `smoke_balance`,
  `smoke_observe_bios`, `smoke_balance_bios`, all wired into the
  `RUN_SERIAL` resource lock at lines 1215-1222 of the root
  `CMakeLists.txt`. UEFI smokes now copy `smoke-ovmf-vars.fd`
  per-test (commit `61bbb4f`).
- Shell additions: `smpcheck`, `balancecheck`, `balanceworker`, and
  `busyyield` are staged into the initrd (`src/user/CMakeLists.txt:59-80`).
- Serial logs remain the primary low-level debug stream; the event ring
  remains the canonical structured one.

### what the code actually does

- APs install the global IDT, run per-CPU idle threads, take per-CPU
  LAPIC timer ticks, and consume reschedule IPIs.
- Per-CPU run queues exist and are taken on every `schedule_next`,
  `mark_thread_ready`, `wake_blocked_thread`, and balancer call.
- `Completion` and `WaitQueue` are wired through block-I/O, child-exit,
  and console-read paths; no kernel waiter discovery still relies on
  registry walks.
- The reschedule IPI vector and the TLB-shootdown vector are dynamically
  allocated through the existing vector allocator, not hard-coded.
- The shell has `smpcheck`, `balancecheck`, `balanceworker`, and
  `busyyield`, and the QEMU smokes assert observable SMP behavior.
- BSP-only ISA IRQ pinning still holds; AP-targeted device IRQs are not
  yet allowed.

### outdated or misleading docs that should be corrected

- `doc/ARCHITECTURE.md` system diagram (`Phase 3` text and the system
  diagram) must be updated to reflect: APs run per-CPU idle threads,
  IF=1, scheduler ticks per CPU, IPI plumbing, per-CPU run queues, load
  balancer.

## 4. Review of progress against goals

- Goal: clean boot and kernel baseline. Completion: high. No new findings.
- Goal: modern x86_64 boot and handoff. Completion: high. No new findings.
- Goal: memory management substrate. Completion: medium-high. New finding:
  the page-frame allocator still does not have a public tier-2 lock to
  match the named `g_page_frames_lock` in the SMP design. Practically the
  allocator works under the new smokes; formally the tier-2 entry should
  exist as a real `Spinlock` to make the contract grep-able. Add this in
  the next round of substrate maintenance.
- Goal: interrupts, timers, platform support. Completion: medium-high.
  AP timers and reschedule IPIs land. TLB shootdown is broadcast-only and
  not yet selective. AP-targeted device IRQs still gated.
- Goal: process model and userland. Completion: medium. SMP placement
  fields landed. Argv/envp, descriptors, cwd, file-backed exec, user heap,
  sessions, process groups, credentials, signals all still missing.
- Goal: storage and filesystem. Completion: block medium-high; filesystem
  zero. The `Completion` cleanup helps; filesystem work itself is unchanged.
- Goal: device model and buses. Completion: medium-high. Per-CPU IRQ
  steering deferred; device namespace, hotplug, mass storage still missing.
- Goal: networking. Completion: low. Same as prior review.
- Goal: security and multiuser. Completion: very low. Same as prior review.
- Goal: GUI and terminal. Completion: terminal medium; GUI zero. Console
  output now serialized to fix SMP-induced marker interleaving; this is a
  scoped fix, not a redesign.
- Goal: build, debug, test. Completion: high. Host tests for `WaitQueue`,
  per-CPU records, run queue, and load balancer landed. QEMU smokes
  serialized. Per-test OVMF vars copies.
- Goal: portability boundaries. Completion: partial. Same as prior review;
  SMP work was deliberately x86_64-shaped (LAPIC timer, ICR, MSR-based GS,
  `iretq`/`swapgs`).
- Goal: SMP-aware design and full user-process SMP scheduling. Completion:
  the eventual goal stated in `GOALS.md` line 299-302 is now in code, not
  just planned. Scheduler design, locking strategy, per-CPU data
  structures, interrupt routing assumptions, and tracing/observability are
  all SMP-aware in the source. The sequencing constraint "SMP should come
  early enough to shape these" is satisfied.
- Goal: accelerator awareness. Completion: not started. Same as prior
  review.

### blockers, missing prerequisites, sequencing issues

- The descriptor/handle/object decision still gates almost everything
  user-facing.
- Argv/envp construction in `proc/user_program.cpp` is still the next
  small kernel-side change before any new userland tool is worth writing.
- Networking is still gated on packet buffer pool, monotonic time, and
  the descriptor decision.
- SSH is still gated on TCP, entropy, credentials, permissions, PTYs,
  sessions, and file I/O.
- AP-targeted device IRQs need: a per-vector affinity hook in
  `platform/pci_msi.cpp` and a corresponding IOAPIC redirection-mask
  override in `arch/x86_64/apic/ioapic.cpp`. The SMP design specifies the
  shape; the call sites are not present.
- TLB shootdown granularity beyond "reload CR3 everywhere" is needed
  before any user-side `mmap`/`munmap`/COW work, but no such caller
  exists yet, so this is rightly deferred until a real memory-management
  syscall lands.

## 5. Overlooked goals / blind spots

These extend the prior review's blind-spots list rather than replace it.

- The project still needs a small, committed descriptor model. SMP
  enablement does not change that. It does change the cost of waiting:
  every cross-CPU wait now uses a `WaitQueue` and IPI, and a future
  descriptor table must compose with that. Deciding the model now keeps
  the descriptor lifetime and the wait-queue lifetime aligned.
- The project needs an explicit SMP readiness rubric document. The SMP
  contract document at `doc/2026-04-29-smp-synchronization-contract.md`
  has the tier list, but a checklist that states "before AP scheduling
  for X workload, lock Y, replace Z with `WaitQueue`" would make future
  expansion (network RX, filesystem readahead) less hand-rolled.
- The project needs a documented AP-IRQ steering policy decision. Right
  now device IRQs land on the BSP. That is correct for now, but the
  long-term decision (per-vector affinity vs lowest-priority logical
  routing vs static round-robin) is not written down. This is the natural
  follow-up after this SMP round, and it should not be made implicitly
  by the next driver author.
- The project still needs an error ABI. `os1_status` or an errno-compatible
  set should land before VFS so that descriptor APIs do not introduce a
  third return-code style on top of the existing "negative on error".
- The project still needs an entropy source before authentication or SSH.
  Now that scheduler tick events span multiple CPUs, a simple "mix
  per-CPU tick deltas plus interrupt timing" entropy collector is at
  least architecturally feasible. Define the policy before designing it.
- The project should formalize observe access tiers. With more SMP-related
  records, the observe surface keeps widening. Decide the privileged-vs-
  unprivileged split before adding a multiuser model on top.
- The project should add a static lock-order check or a runtime debug
  invariant. The SMP contract is now load-bearing for correctness; it is
  enforced only by reading code.
- The project should formalize the page-frame allocator's external lock
  identity. The SMP design names `g_page_frames_lock` as tier 2; the source
  uses an internal allocator lock that does not appear under that name.
  Rename/expose to match the contract document so the tiered-lock
  invariant remains greppable.
- The project should keep the post-implementation review pattern. The
  `2026-05-05-smp-enablement-review.md` -> `61bbb4f` cycle was effective
  and reproducible.
- The project should grow a way to assert on lock-order violations in
  debug builds. Either a per-`Spinlock` tier tag or a per-thread "highest
  tier currently held" counter would catch contract violations as
  assertions instead of as racey bugs.

## 7. Immediate next steps

The list below is prioritized by what unlocks the most downstream value
relative to the cost. Items 1–5 carry forward unchanged from the prior
2026-05-05 review because SMP enablement does not move them. Items 6–10
are new or sharpened by the SMP landing.

1. Decide and document the minimal descriptor/handle contract.
   It still gates VFS, devices, sockets, PTYs, shell redirection, and
   SSH. Now also gates how `WaitQueue`-backed waits compose with file/
   device blocking.
2. Implement a per-process descriptor table and route console fd 0/1/2
   through it. Tests the contract before filesystem complexity.
3. Add argv/envp stack construction for `spawn`/`exec`. Required
   before any serious userland command.
4. Add a minimal VFS with initrd as the first mounted backend. Converts
   initrd-specific program loading into a general path lookup mechanism.
5. Move `exec` to VFS lookup while preserving initrd boot. Prevents a
   second program-loading path.
6. Refresh `doc/ARCHITECTURE.md` and `doc/latest-review.md` to reflect
   that APs run real work and that scheduler/SMP claims in earlier
  sections are now incorrect. Completed by the 2026-05-05 documentation
  refresh.
   Doc drift is real; it is now mid-2026 and the earlier May-2026
   ARCHITECTURE.md text still describes a parked-AP system.
7. Promote the page-frame allocator's internal lock to the named tier-2
   `g_page_frames_lock`. Keeps the SMP contract document and the source
   aligned and grep-able. Cheap maintenance that pays off when the next
   subsystem has to ask "what locks am I allowed to take here?".
8. Pick and document the AP-targeted device IRQ steering policy.
   Required before any device driver author makes the choice
   implicitly.
9. Add a debug-build lock-order check (per-thread highest-tier-held
   counter, asserted in `Spinlock::lock`). The SMP contract is now
   load-bearing. Static review caught the `observe processes` inversion;
   the next inversion will not be a single-file fix.
10. Choose and implement the first block-backed filesystem and the
    deterministic image tool. The block path is mature; the missing
    value is persistent namespace semantics, not more block smoke.

Lower-priority but still in the immediate window:

11. Add minimal credentials to `Process`. Tie the first permission
    check to the new VFS path so credentials and namespace land
    together.
12. Add a monotonic time syscall and kernel timeout primitive. Required
    by network protocol work and eventual SSH timeouts.
13. Define a packet buffer pool. Required before extending `virtio-net`
    beyond ARP. Should be designed before TCP planning starts.
14. Add `doc/os-api-draft/README.md`. Completed after the review: the file
  exists and now explicitly marks the drafts as exploratory rather than live
  kernel contracts.
15. Document the `BootInfo` capacity limits as part of the boot ABI.
    Same as prior review.
