## 1. Executive Summary

### Current project state

Substrate-level: the kernel is materially stronger than at the prior 2026-05-03 review. The small-object allocator now has seven consumer subsystems and is exercised under interrupt context. Process/thread tables are dynamic. The four driver registries (device binding, BAR claims, IRQ routes, DMA allocations) grow on demand. Kmem corruption is on the event ring. User-visible: still mostly unchanged at the syscall and shell layers. Same `/bin/init` → `/bin/sh` boot, same nine-call SYSCALL/SYSRET ABI, same six user binaries, same parked APs, same ELF-from-initrd loader, same lack of argv/envp, same lack of credentials, same lack of VFS, same lack of filesystem. What *is* user-visible: the smoke matrix's `kmem` shell command now reports six kmem caches (was zero builtin-only at the prior review and would have been four after the kmem-only commits) — `kmalloc-16` through `kmalloc-1024` plus the named caches `process`, `thread`, `arp_entry`, `device_binding`, `pci_bar_claim`, `irq_route`, `dma_allocation`. The right way to read this delta: the project has spent one cycle building substrate (the allocator) and one cycle wiring substrate to consumers. That is exactly the prior review's recommended order; it produced an architecture pass that is unusually rare for hobby OSes — substrate-then-use, not substrate-without-use. The native-interface drafts remain the only major design surface that has been *added* without a corresponding kernel commitment.

### Biggest risks / gaps

1. **The kmem allocator has no kernel consumers.** A grep across `src/kernel` for `kmalloc`, `kcalloc`, `kmem_cache_create`, or `kmem_cache_alloc` outside `mm/kmem.{cpp,hpp}` finds exactly one hit: [src/kernel/syscall/observe.cpp:730](../src/kernel/syscall/observe.cpp#L730), which calls `kmem_cache_stats_count()`. Every BAR claim, IRQ route, DMA allocation, and device binding still lives in compile-time-sized tables. This is fine *now* (CI will catch any failure to initialize the allocator), but it means a review next cycle could rediscover the same finding the prior review made: every prospective subsystem (FS, network stack, USB hub, multi-vector IRQ) wants this. Substrate without a consumer subsystem is not yet a *result*.
2. **The native-interface decision is still unmade.** This is the fifth consecutive review carrying this finding. The drafts have grown from two (2026-04-29-review-2: `elf_interface_spec.md` + `os1-shell-language-first-draft.md`) to four. Their combined length is 3,492 lines. None of them have produced a single kernel type, syscall, or test. The prior review's wording — "the cost of choosing wrong has gone up; the cost of choosing right has not" — applies more strongly today, because two of the four drafts ([native_object_kernel_contract.md](os-api-draft/native_object_kernel_contract.md) and [object_oriented_vfs_spec.md](os-api-draft/object_oriented_vfs_spec.md)) are now elaborated enough that walking back from them will be noticeably more expensive than committing to a slimmer first version of either.

4. **APs are still parked.** [src/kernel/arch/x86_64/cpu/cpu.cpp](../src/kernel/arch/x86_64/cpu/cpu.cpp) still terminates with `cli; hlt`. `Spinlock` is now used in *two* files ([src/kernel/debug/event_ring.cpp](../src/kernel/debug/event_ring.cpp) and [src/kernel/mm/kmem.cpp](../src/kernel/mm/kmem.cpp)) — which is technically progress, but the additional locks are guarding BSP-only paths. The `kmem` allocator's per-cache `Spinlock` and its named-cache-registry `Spinlock` will become real cross-CPU synchronization the day an AP first calls `kmalloc`; today they are pure paperwork. The driver substrate's BSP-affine MSI-X destinations through `bsp_apic_id()` ([src/kernel/platform/pci_msi.cpp:55](../src/kernel/platform/pci_msi.cpp#L55)) remain unchanged.

The remaining drift item from the prior listing — "the observe smoke now chains an extra `kmem complete` step" — is still implicit; the README's high-level shell description does not enumerate every step but the smoke marker is asserted in CI, so the user-facing risk is low.
6. **The kmem allocator's first crash mode in production code will be panic-on-corruption with no kernel consumer to crash.** `kmem_corruption` calls `halt_forever` (or `__builtin_trap` in host tests). This is the right behavior for a small-object allocator, but it currently means: if an interrupt path or a host-test path were to mistakenly free a kernel pointer through `kfree` (a pointer that was never `kmalloc`'d), the kernel would halt. There is no telemetry path that emits an `OS1_KERNEL_EVENT_*` for kmem corruption — the prior review's `OS1_KERNEL_EVENT_USER_COPY_FAILURE` precedent suggests there should be. Today the failure is silent to the event ring.
7. **The substrate-vs-consumer asymmetry is starting to compound.** Three substrate items have landed in the last two cycles without their consumers: (a) per-CPU sync vocabulary (`Spinlock`, `IrqGuard`, `OS1_BSP_ONLY`) — used in 2 files of 156; (b) DMA buffer with sync seam (`dma_sync_for_device`/`dma_sync_for_cpu`) — barrier no-ops on x86, no IOMMU consumer; (c) kmem allocator — zero kernel consumers. None of these are wrong; each was the right next thing at the time. But the *project shape* is becoming "substrate, then more substrate", and the next review's most important question is going to be: *is the next phase finally a consumer subsystem, or is it more substrate?*

### Boot path and platform assumptions

Boot frontends unchanged: BIOS at [src/boot/bios](../src/boot/bios), Limine/UEFI at [src/boot/limine](../src/boot/limine). The current sequence is:

1. `own_boot_info()` deep-copies BootInfo into kernel BSS.
2. `cpu_initialize_record(g_cpu_boot)` + `cpu_init()` — BSP GDT/TSS/`gs`.
3. `page_frames.initialize(memory_regions, ...)` — bitmap allocator.
4. Module + framebuffer reservations.
5. `VirtualMemory kvm` — bootstrap-low identity, kernel image by section, direct map, framebuffer/RSDP/module windows.
6. `kvm.activate()` switches CR3.
7. **`kmem_init(page_frames)`** — *new* in this review cycle. Initializes builtin caches and clears named-cache registry. Lives at [kernel_main.cpp:396](../src/kernel/core/kernel_main.cpp#L396), between `page_frames.enable_direct_map_access()` and rebinding `g_cpu_boot` through the direct map.
8. CPU pointer rebound through direct map; `cpu_init()` re-issues GDT/IDT.
9. `platform_discover()` → ACPI → AML → device inventory → `_PRT` → MMIO mappings → CPU allocation → PCI ECAM enumeration. (Discovery only — no driver probes yet.)
10. PIC / IOAPIC / LAPIC init; `cpu_boot_others` brings APs up to `cpu_idle_loop`.
11. Display selection, terminal allocation, IDT registration.
12. `platform_probe_devices(kvm)` — driver registry runs; `virtio-blk`, `virtio-net`, xHCI bind here.
13. Keyboard, console-input, ISA IRQ routing.
14. `init_tasks` (fixed `Process[32]`/`Thread[32]`).
15. `create_kernel_process` + idle thread + `kernel_boot_sequence_thread`.
16. `OS1_KERNEL_EVENT_SMOKE_MARKER` recorded; `initialize_scheduler_timer()` (HPET-calibrated LAPIC, PIT fallback).
17. `enter_first_thread(boot_sequence_thread, ...)`.

The placement of `kmem_init` ahead of `platform_discover` is correct: any subsystem that wants slab-backed allocation can do so from the moment the platform layer starts. The placement also matches the design specification at [doc/2026-05-04-kernel-small-object-allocator.md:73-77](2026-05-04-kernel-small-object-allocator.md#L73) which explicitly called for kmem before the boot-sequence thread starts.

Hard platform assumptions unchanged: x86_64 only; ACPI required; PCI is ECAM/MCFG only; `BootInfo` capped at 128 memory regions and 16 modules.

### Memory management

Three layers now:

1. **Physical**: `PageFrameContainer` bitmap allocator at `kPageFrameBitmapBaseAddress = 0x20000`, 8 GiB cap, 4 KiB granularity. No buddy/freelist.
2. **Virtual**: `VirtualMemory` owns root allocation, four-level walks, NX-aware permission writes, user-slot teardown. User CR3s clone `kKernelPml4Index` and `kDirectMapPml4Index` only.
3. **Slab/large**: `kmem` ([src/kernel/mm/kmem.cpp](../src/kernel/mm/kmem.cpp), 1736 LOC). Builtin caches at 16/32/64/128/256/512/1024-byte slot sizes ([src/kernel/mm/kmem.cpp:282-290](../src/kernel/mm/kmem.cpp#L282)). `kmalloc(size)` selects the smallest cache that fits via `select_cache(size)`, falling back to a page-run large-allocation path through `LargeAllocationHeader` for requests > 1024 bytes. `kcalloc(count, size)` overflow-checks via `__builtin_mul_overflow` and forwards with `Zero`. `kmem_cache_create(name, object_size, alignment)` registers a named cache up to a fixed `kMaxNamedCaches` slot pool with a separate registry lock. Slabs are one page each, with a `SlabPageHeader` carrying magic, object size, slot stride, free count, free-list pointer, owning cache pointer, and (debug builds) an `allocation_bitmap[8]` qword bitmap. Slab list management keeps one empty slab per cache to absorb churn (`KmallocFlags::NoGrow` honors a current-capacity-only request). `kfree(ptr)` resolves through the page-aligned `SlabPageHeader.magic` word to dispatch to slab or large-allocation paths.

The kmem flag surface is `enum class KmallocFlags : uint32_t` at [src/kernel/mm/kmem.hpp:9-16](../src/kernel/mm/kmem.hpp#L9): `None`, `Zero`, `NoGrow`, `Atomic`, `PanicOnFail`. The `Atomic` flag is reserved for a future preemption-safe path; the locking discipline today is `IrqGuard` plus per-cache `Spinlock`, which is already interrupt-safe by construction. `PanicOnFail` calls `kmem_halt` on failure; the default path returns `nullptr` so callers can choose recovery.

Debug-build instrumentation (`OS1_KMEM_DEBUG=1` — on by default for host tests, off for the freestanding build) adds 8-byte redzones front and back of each slot, `0xA5` allocation poison, `0x5A` free poison, `0xFD` redzone bytes, `0xDD` page-released poison, and `__builtin_return_address(0)` capture for caller tracking. `kmem_corruption(reason, ptr, ...)` at [src/kernel/mm/kmem.cpp:514](../src/kernel/mm/kmem.cpp#L514) emits debug logs for header address, physical base, cache name, expected/observed magic, then calls `kmem_halt` (`halt_forever` in the freestanding build, `__builtin_trap` under `OS1_HOST_TEST`).

Builtin cache statistics are observable through `kmem_get_cache_stats(index, KmemCacheStats&)` at [src/kernel/mm/kmem.cpp:1628](../src/kernel/mm/kmem.cpp#L1628). The `KmemCacheStats` struct ([src/kernel/mm/kmem.hpp:36-54](../src/kernel/mm/kmem.hpp#L36)) carries: `name`, `object_size`, `alignment`, slab counts (total/empty/partial/full), object counts (total/free/live/peak-live), `alloc_count`, `free_count`, `failed_alloc_count`, `slab_growth_count`, `slab_return_count`. Global stats (`KmemGlobalStats` at [src/kernel/mm/kmem.hpp:56-68](../src/kernel/mm/kmem.hpp#L56)) add `cache_count`, `slab_page_count`, `live_large_allocation_count`/`bytes`, `failed_large_alloc_count`, `invalid_free_count`, `corruption_count`. The observe layer at [src/kernel/syscall/observe.cpp:725](../src/kernel/syscall/observe.cpp#L725) walks all caches and emits one `Os1ObserveKmemRecord` per cache.

The DMA layer at [src/kernel/mm/dma.cpp](../src/kernel/mm/dma.cpp) is unchanged from the prior review. `dma_sync_for_device`/`dma_sync_for_cpu` remain barrier no-ops on x86; the seam exists for IOMMU and non-coherent platforms but no consumer drives it yet.

## 5. Overlooked Goals / Blind Spots

### The five-stage commit cadence should not become the project's stable shape

The Stage 1..Stage 5 commits all share author and same-day landing — they are effectively one PR's worth of work checkpointed mid-flight. That cadence is fine for a single complex feature, but if the next subsystem (filesystem, network stack, native interface) follows the same `Stage N` pattern, the git history starts to look like a flight recorder instead of an architectural record. Future merges should preferably collapse to the granularity at which a reviewer would naturally rewind — typically per-API-surface or per-host-test-pass — and use commit messages that describe the *what* rather than the *when*. The Stage commits are also nameless past their numeric label, which makes `git blame` less useful than the existing PR-shaped history (e.g., `Device Driver model (#20)`).

### The kmem allocator needs at least one consumer next cycle

The allocator is otherwise a standing exhibit of "we built the substrate but nothing uses it." The first consumer should ideally be small, isolated, and motivated by an actual constraint (not a contrived demo). Three candidates, in priority:

- **An ARP cache for `virtio-net`.** The driver currently has zero state above the single packet buffer. An ARP cache is the smallest meaningful network state, would exercise `kmem_cache_create("arp_entry", ...)` end-to-end, validate the named-cache registry under interrupt context (since RX completes from MSI-X), and unblock IPv4 transmit-side work. It is also the only realistic kmem consumer that can be exercised today by an existing smoke (the `virtio-net smoke ok` ARP path) without first adding a VFS or filesystem.
- **A driver binding table that grows.** Today the binding, BAR-claim, IRQ-route, and DMA-allocation tables are compile-time-sized. Replacing one of them with a `kmem_cache_create` per-record allocator would prove the path under interrupt context.
- **The first inode/dentry cache.** This is the canonical case the prior review pointed at, but it depends on a VFS existing first.

The risk of deferring all three is that the next allocator change (lock granularity, slab geometry, magazines) lands without ever having been exercised by a real consumer. Lock-discipline regressions in particular are easy to miss when the only call sites are tests that allocate from a single thread.

A secondary risk is that the next contributor reads `kmem.cpp`, infers the design from the code (rather than from the design document), and replicates a non-trivial subset of it inside a separate driver-local pool — a pattern that is common in early-stage kernels and that the kmem implementation was built specifically to displace.

### kmem corruption is not on the event ring

`kmem_corruption(...)` panics. Every other failure surface in the kernel that reaches a user-visible state goes through `OS1_KERNEL_EVENT_*` — user-copy failures, traps, scheduler transitions, IRQ delivery, block I/O completion, NIC RX. A kernel allocator panic is currently silent to the event ring; the only signal is the panic message itself. Adding `OS1_KERNEL_EVENT_KMEM_CORRUPTION` or extending `OS1_KERNEL_EVENT_USER_COPY_FAILURE`-style coverage would mean a CI run that hit a corruption could surface it through `observe events` rather than only through the panic banner. Lightweight; aligns with prior observability discipline.

### The os-api-draft folder needs an executive summary

Four documents totalling 3,492 lines is more than any single contributor can hold in mind. There is no top-level `README.md` or `index.md` under [doc/os-api-draft/](os-api-draft/) that names the order of reading, the relationship between the four documents, or the "if you read only one, read this" pointer. New contributors will struggle to triangulate. A 30-line index file is cheap and would also force the architects to articulate the directional commitment that the four drafts implicitly imply.

The drafts also have internal-consistency seams worth resolving before they grow further. [doc/os-api-draft/native_object_kernel_contract.md](os-api-draft/native_object_kernel_contract.md) specifies a uniform syscall surface (`object_query_interface` etc.), but [doc/os-api-draft/object_oriented_vfs_spec.md](os-api-draft/object_oriented_vfs_spec.md) presents the VFS path-resolution semantics as if the eventual API takes path arguments at the syscall boundary. Either both layers map down to the same `object_*` calls and the VFS doc is a mapping, or one of them needs to surrender to the other. The reader cannot tell from the documents alone which is intended.

### "Decide native vs FD" is now overdue

Five reviews. The cost of the decision is no longer "two-page design doc"; it is "two-page design doc plus mechanical reconciliation against four drafts plus a kernel commit that creates the first `Handle` or `FileDescriptor` type and a single syscall that uses it." The reason to do that *next* is that the kmem allocator is exactly the kind of low-level substrate that a `kmem_cache_create("handle", sizeof(Handle), alignof(Handle))` consumer would naturally use as its first non-stub allocation, which would tie the substrate to its first consumer in one move.

### Thread/process tables are still fixed-size

The kmem allocator now exists. The 32-entry process and thread tables that were sized small to avoid needing `kmalloc` no longer have that justification. Replacing `Process[32]`/`Thread[32]` with named caches and a registry list is a small and high-confidence first kmem consumer; it would also remove an artificial cap that smoke matrices will eventually stress.

### Per-CPU `kmalloc` magazines are not present yet

Many slab allocators (Solaris, Linux, FreeBSD UMA) add per-CPU magazines to avoid contending the global cache lock on every allocation. The current kmem implementation has no magazine layer; it acquires the per-cache `Spinlock` on every alloc/free. This is correct for a single-CPU world but will become a measurable bottleneck the moment two CPUs allocate concurrently. Not urgent today; document the constraint so a future "we need a magazine" change does not surprise anyone. The `KmallocFlags::Atomic` flag is reserved in the API but unimplemented; it would be the natural place to flag an allocation that may not block (and therefore must succeed against per-CPU magazines or fail fast) once magazines exist.

### The kmem design specification predicts more than the implementation does

[doc/2026-05-04-kernel-small-object-allocator.md](2026-05-04-kernel-small-object-allocator.md) is 1,047 lines and describes the design as "for implementation". The implementation is only ~1,700 LOC. Either the design has more content than is needed (over-spec'd), or the implementation is a slice of a larger plan. A future reviewer should be able to tell which; right now, neither the design doc nor the source carries a "what was deferred from the spec" annotation.

### The 2026-05-04 dated design doc is in tree before 2026-05-04

The design doc filename is `2026-05-04-kernel-small-object-allocator.md` and it was committed on 2026-05-03 (commit `ecd3f6f`). This is harmless once but the convention "filename date = creation date" is more useful than "filename date = aspirational target" if dates are to remain a stable index. The next dated artifact should pick the date of authorship.

### Initrd manifest assertion is still missing

Carry-forward from prior review: the staged binary set is not asserted against `observe initrd` by CI. A future binary that fails to stage would not be caught. Small; should land in a future doc/CI cleanup pass.

### The host-test build silently enables `OS1_KMEM_DEBUG=1`

[tests/host/CMakeLists.txt:5](../tests/host/CMakeLists.txt#L5) declares `option(OS1_KMEM_DEBUG ... ON)`. This is the right default — host tests should exercise debug paths. But the production build does *not* set this. A class of bugs that are caught in host tests (redzone violations, double-free, leaks) will be silent in the freestanding build. Either turn `OS1_KMEM_DEBUG=1` on in CI's freestanding configure step too, or document that the silence is intentional. The simpler half of this — making the asymmetry explicit in the kernel CMakeLists — costs one comment line.

### Boot-order and AP awareness on `kmem_init`

`kmem_init(page_frames)` runs before `cpu_boot_others`, which is correct — APs that come up later already see an initialized allocator. But APs do not yet *call* it; they go straight to `cpu_idle_loop`. When the first AP runs anything substantive, the per-cache `Spinlock` and the `IrqGuard` already cover it, so nothing in the kernel needs to change at the kmem boundary. The thing that *will* need to change is callers that today hold a BSP-only assumption about which cache is "free" (none today, but every future named cache built around BSP-affine state is a potential incident). This should be called out in [doc/2026-04-29-smp-synchronization-contract.md](2026-04-29-smp-synchronization-contract.md) when the next SMP migration step lands.

## 6. Roadmap

Ordered by dependency, not attractiveness. Items resolved since the prior review are dropped; items that remain are repeated; items that have shifted in priority are noted.

### Near-term (next 1-3 milestones)

1. **Decide the native-interface stance and commit one syscall in that direction.** The drafts under [doc/os-api-draft/](os-api-draft/) are now elaborated enough to slice. The smallest meaningful commit: define a `Handle` type (or `FileDescriptor` if FD-first), allocate it through `kmem_cache_create("handle", ...)`, expose one syscall (`object_query_interface` or `open`-style) that returns one. *Why now:* fifth consecutive review. The cost of further drafting without commitment is no longer "we might choose wrong" — it is "we are accumulating spec without the stress test of one consumer."
2. **Land argv/envp on the initial user stack.** Tiny ABI step. *Why now:* fifth consecutive review carrying. Independent of §1. Unblocks meaningful filesystem-backed `exec`.
3. **Replace at least one fixed-size kernel table with a kmem-backed allocator.** Candidate: device-binding table or BAR-claim table. *Why now:* substantiates the kmem implementation, exercises the named-cache path under a real workload, removes an arbitrary cap.
4. **Begin VFS scaffolding and first read-only filesystem.** Pick FAT32 read-only or a bespoke simple FS; mount through a first-cut VFS; teach `exec` to resolve paths through it; keep initrd as fallback. Use `kmem_cache_create` for the inode/dentry caches. *Why now:* the block layer and the small-object allocator that used to block this are no longer constraints. This is the highest-leverage substrate-to-product step available.
5. **Plumb uid/gid + credentials field on `Process`.** Single user, but the right shape; `/etc/passwd`-equivalent in initrd. *Why now:* unblocks Goal D's multiuser half; needed before any meaningful permission check; small and isolated.
6. **Per-page user-buffer scatter-gather and DMA pinning.** Already named in the prior review; gates user-driven I/O. *Why now:* the storage path's next extension; directly unblocks user-mode I/O experiments.
7. **Documentation cleanup (kmem + os-api-draft links)** — *resolved* in the 2026-05-04 documentation update pass. [doc/ARCHITECTURE.md:1108/1118/1120](ARCHITECTURE.md#L1108) and [GOALS.md:480/481](../GOALS.md#L480) now reflect the actual kmem state and the renamed [os-api-draft/](os-api-draft/) folder; [README.md:79](../README.md#L79) now mentions the kmem host-test coverage. The remaining doc items (ARCHITECTURE.md header date, BootInfo cap residue, plan-with-status hybrid, missing os-api-draft index) are smaller and less urgent.
8. **Add a generic initrd-manifest assertion to the smoke matrix.** Carry-forward. *Why now:* small; prevents silent drift if a future binary fails to stage.
9. **Emit kmem corruption to the event ring.** Add `OS1_KERNEL_EVENT_KMEM_CORRUPTION`. *Why now:* aligns with the rest of the kernel's observability discipline; small; would be retroactively useful the first time kmem catches a real bug.

### Mid-term (after the near-term cluster lands)

1. **Begin the network protocol stack on top of `virtio-net`.** ARP cache (using kmem named caches), IPv4, ICMP echo, UDP. Defer DHCP and DNS until configuration storage exists. *Why now:* the driver is in; the substrate including `kmalloc` is in; only the protocol stack is missing.
2. **Bring up one AP for restricted work.** With the synchronization vocabulary in place and §1 decided, run timer accounting or a single virtio-net RX queue on a single AP. The kmem allocator's per-cache lock will see its first contended use here. Migration order in [doc/2026-04-29-smp-synchronization-contract.md](2026-04-29-smp-synchronization-contract.md).
3. **Per-CPU IRQ steering.** Add an affinity field to the IRQ resource record. Steer one queue to one AP. *Why now:* cheap once an AP runs anything non-trivial; needed before networking matures.
4. **USB hub class and HID mouse routing.** Extend the xHCI work with a hub driver and route mouse reports past recognition into a future input event queue.
5. **TCP and a minimal sockets/handle equivalent.** Once §1 is decided, implement the smallest TCP that can sustain SSH eventually.
6. **Native object kernel surface beyond the first syscall.** Build out `object_get_property`, `object_call`, `object_subscribe` against one or two kernel-native objects (e.g., `os` and a single `device`). Land the user-side runtime that loads `.os.interface` sections.
7. **First persistent on-disk filesystem write path.** Once read works and a buffer cache exists, add the journaled or copy-on-write write path appropriate to the chosen FS.
8. **Replace `Process[32]`/`Thread[32]` with kmem-backed allocators.** Removes the arbitrary cap; second non-trivial kmem consumer.
9. **Per-CPU kmem magazines.** Once two CPUs allocate concurrently in steady state.

## 7. Immediate Next Steps

In priority order. Each item is justified by what it unblocks now.

1. **Make the native-interface stance decision and commit one syscall in that direction.** *Why now:* fifth consecutive review; drafts have grown to 3,492 lines without a single kernel commitment. A `Handle` (or `FileDescriptor`) type allocated through the new `kmem_cache_create` and exposed through one syscall is the smallest possible commitment that ties the spec to the substrate. Without it, every additional spec line raises the cost of changing direction.
2. **Land argv/envp on the initial user stack.** *Why now:* small, decoupled from §1, unlocks meaningful shell tests with arguments. Fifth consecutive review carrying.
3. **Documentation cleanup pass** — *resolved* on 2026-05-04. [doc/ARCHITECTURE.md:1108/1118/1120](ARCHITECTURE.md#L1108), [GOALS.md:480/481](../GOALS.md#L480), and [README.md:79](../README.md#L79) now match the substrate. The next-tier residue (ARCHITECTURE.md header date, BootInfo caps, plan-with-status split, os-api-draft index) is smaller and can land alongside the next implementation pass.
4. **Replace one fixed-size kernel table with a kmem-backed allocator.** Recommended candidate: device-binding table or BAR-claim table. *Why now:* the kmem allocator currently has zero kernel consumers; without a first consumer the substrate accumulates without validation, and a future filesystem implementation will be the first thing to stress it under realistic load.
5. **Begin VFS scaffolding and first read-only filesystem.** *Why now:* the block layer and small-object allocator that used to block this are no longer constraints; this is the highest-leverage substrate-to-product step available.
6. **Begin uid/gid plumbing on `Process`.** *Why now:* unblocks Goal D's multiuser half; needed before any meaningful permission check; small and isolated.
7. **Add `OS1_KERNEL_EVENT_KMEM_CORRUPTION` (or similar) and emit on `kmem_corruption`.** *Why now:* aligns kmem with the rest of the kernel's observability discipline.
8. **Add an executive summary index under [doc/os-api-draft/](os-api-draft/).** A 30-line `README.md` that names the read order and the relationship between the four documents. *Why now:* four documents at 3,492 lines is more than any contributor can triangulate without a guide; the index forces the architectural commitment to be articulated in one place.
9. **Per-page user-buffer scatter-gather and DMA pinning.** *Why now:* gates user-driven I/O; carry-forward.
10. **Per-CPU IRQ steering field on the IRQ resource record.** *Why now:* cheap; future BSP-affinity migration will be expensive.
