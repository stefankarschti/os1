## 1. Executive summary

### current project state

- Current reality: `os1` is no longer just a boot/kernel shell experiment.
- It now has a real x86_64 kernel substrate with two boot paths, higher-half execution,
  a direct map, ring-3 user programs, ACPI/PCI discovery, MSI-capable drivers, USB input,
  virtio block I/O, a virtio-net ARP smoke path, host tests, and QEMU smoke tests.
- The code supports a Limine/UEFI ISO path through `src/boot/limine/` and a BIOS raw-image
  path through `src/boot/bios/`.
- The kernel entry path in `src/kernel/core/kernel_main.cpp` builds the owned `BootInfo`,
  installs kernel page tables, initializes physical memory and `kmem`, discovers ACPI/PCI,
  brings APs to a parked state, starts interrupts, creates processes/threads, and runs `/bin/init`.
- The process model is functional but still minimal: `spawn`, `exec`, `waitpid`, `read`,
  `write`, `yield`, `exit`, `getpid`, and `observe` are the current syscall surface in
  `src/uapi/os1/syscall_numbers.h`.
- The operator shell in `src/user/programs/sh.cpp` is still an observability and smoke shell,
  not yet a file-backed or session-backed operating environment.
- The strongest recent improvement is that `kmem` is not just an allocator demo anymore.
- Code now uses `kmem` for process and thread registries, ARP cache entries, device bindings,
  PCI BAR claims, DMA allocation records, and IRQ route records.
- That means the allocator has crossed from infrastructure into kernel lifetime management.
- The main gap is the absence of kernel object semantics for user-visible resources.
- Without descriptors or handles, the kernel cannot grow coherent files, directories,
  devices, sockets, PTYs, sessions, or SSH.
- Without credentials, it cannot grow a meaningful multiuser model.
- Without a VFS and filesystem, `exec` remains initrd-only and storage remains driver smoke.
- Without a packet stack, virtio-net remains an ARP-level demonstration rather than networking.
- Without an SMP scheduling plan, secondary CPUs are discovered and started but not used.

### strongest parts

- Boot handoff is substantially better than a typical early hobby OS.
- `src/kernel/handoff/boot_info.hpp` creates a shared boot contract used by both BIOS and Limine.
- The Limine shim in `src/boot/limine/` is intentionally small and hands a normalized contract
  to the real kernel instead of becoming a second kernel.
- The higher-half/direct-map transition in `src/kernel/core/kernel_main.cpp` is explicit and
  gives later code one consistent physical-memory access model.
- The page-frame allocator in `src/kernel/mm/page_frame.cpp` and virtual-memory manager in
  `src/kernel/mm/virtual_memory.cpp` are testable and already used for process address spaces.
- The user-copy layer in `src/kernel/mm/user_copy.cpp` and user address checks in
  `src/kernel/mm/user_address.hpp` are an important hard boundary for all later syscalls.
- Interrupt routing is concrete, not aspirational: IDT setup, LAPIC, IOAPIC, HPET fallback,
  vector allocation, IRQ route registry, MSI-X/MSI/INTx fallback, and event recording all exist.
- The platform layer is now a real subsystem under `src/kernel/platform/`, not scattered probe code.
- ACPI parsing in `src/kernel/platform/acpi.cpp` and AML support in
  `src/kernel/platform/acpi_aml.cpp` are enough to support PCI routing and power hooks.
- The device model under `src/kernel/drivers/bus/` is simple but already enforces driver matching,
  BAR ownership, binding records, remove paths, and observable resource ownership.
- `virtio-blk` in `src/kernel/drivers/block/virtio_blk.cpp` is interrupt-driven and supports
  reads, writes, bounded multi-sector requests, and smoke verification.
- xHCI is unusually far along for the project stage: controller setup, command/event rings,
  device descriptors, boot keyboard input, and HID mouse recognition exist in `src/kernel/drivers/usb/`.
- The observe ABI in `src/uapi/os1/observe.h` is a major strength because it makes kernel state
  inspectable from ring 3 without relying only on serial logs.
- Host tests in `tests/host/` cover many low-level units that are usually left untested:
  CPIO, BootInfo, ACPI, AML, HPET, PCI, MSI, resources, ARP cache, xHCI helpers, `kmem`,
  DMA, page frames, virtual memory, user address validation, user ELF loading, and observe ABI.
- QEMU smokes in the root `CMakeLists.txt` exercise user-visible behavior across UEFI and BIOS.

### biggest risks / gaps

- The largest architectural risk is the still-unresolved native interface decision.
- `doc/os-api-draft/native_object_kernel_contract.md`,
  `doc/os-api-draft/object_oriented_vfs_spec.md`, and related drafts are large,
  but the kernel code has not committed to a descriptor, object, handle, vnode, or file model.
- That decision now blocks the highest-value next feature work in `GOALS.md`.
- `sys_read` and `sys_write` currently special-case console fds rather than routing through
  per-process descriptor state in `src/kernel/syscall/process.cpp` and
  `src/kernel/syscall/console_read.cpp`.
- Processes in `src/kernel/proc/process.hpp` have pid, state, address space, parent, name,
  and registry linkage, but no credentials, descriptor table, working directory, signal state,
  session, or namespace pointers.
- Threads in `src/kernel/proc/thread.hpp` have typed wait states, but no priority,
  CPU affinity, cancellation, robust blocking object, or wakeup ownership model.
- `exec` in `src/kernel/proc/user_program.cpp` and `src/kernel/syscall/process.cpp`
  loads images by initrd path and does not build an argv/envp stack.
- There is no VFS implementation; `src/kernel/vfs/README.md` is a placeholder.
- There is no security implementation; `src/kernel/security/README.md` is a placeholder.
- There is no persistent filesystem despite a working block path; initrd CPIO in
  `src/kernel/fs/` remains the only file source.
- Networking is below the line needed for `GOALS.md`: `virtio-net` and ARP cache exist,
  but there is no IP, ICMP, UDP, TCP, DHCP, DNS, socket API, packet buffer pool, or user API.
- SMP is structurally present but not operational: APs are booted and parked by
  `src/kernel/arch/x86_64/cpu/cpu.cpp`, while scheduling remains BSP-only.
- The memory allocator stack is still BSP-constrained through `OS1_BSP_ONLY` and
  `KASSERT_ON_BSP()` patterns in memory-management code.
- Documentation drift is now significant enough to mislead planning if not fixed.
- `doc/ARCHITECTURE.md` still says task tables are fixed arrays and that `kmem` has no consumers.
- `GOALS.md` still says the next `kmem` step is the first kernel consumer, but the source already
  has several.
- Recommendation: treat descriptor/VFS/credentials as the next architectural gate and stop growing
  shell language, SSH, GUI, or SMP scheduling until that gate is closed.

## 2. Current state from code

### boot path and platform assumptions

- Current reality: the build produces a UEFI/Limine ISO and a BIOS raw image.
- UEFI/Limine is the default modern path through `src/boot/limine/entry.cpp`.
- BIOS compatibility remains through `src/boot/bios/boot.asm` and
  `src/boot/bios/kernel16.asm`.
- The shared contract is `BootInfo` in `src/kernel/handoff/boot_info.hpp`.
- `BootInfo` carries boot source, memory map, kernel physical range, RSDP, SMBIOS,
  bootloader name, command line, text console, framebuffer, and module records.
- The kernel takes ownership of the handoff object early through
  `src/kernel/handoff/boot_info.cpp`.
- The Limine shim asks Limine for HHDM, memmap, framebuffer, bootloader info, cmdline,
  modules, RSDP, and SMBIOS.
- The Limine shim then loads the real `kernel.elf` module and passes the normalized
  `BootInfo` to `kernel_main`.
- The BIOS path loads kernel and initrd from fixed raw-image slots and also fills the
  same `BootInfo` shape.
- Current platform assumption: x86_64, four-level paging, ACPI, PCI ECAM, LAPIC/IOAPIC,
  and QEMU/q35-style devices.
- `src/kernel/platform/init.cpp` treats ACPI discovery as the foundation for CPUs,
  IOAPICs, PCI ECAM, HPET, FADT, and AML namespace loading.
- `src/kernel/platform/acpi.cpp` requires enough ACPI structure to parse MADT, MCFG,
  FADT, HPET, and DSDT/SSDT definition blocks.
- There is no non-ACPI platform path for modern boot.
- `GOALS.md` says the primary target is x86_64 with later AArch64/RISC-V boundaries.
- Current code has some boundaries, but the runtime still assumes x86 page-table,
  interrupt, ACPI, LAPIC, and port-I/O semantics in important paths.
- The boot contract has fixed capacities: 128 memory regions and 16 modules in
  `src/kernel/handoff/boot_info.hpp`.
- Those caps are acceptable for the current QEMU target but should be documented as
  boot ABI limits before real hardware experiments.
- The generated layout uses `OS1_INITRD_LOAD_ADDRESS=0x90000` in `CMakeLists.txt`
  and `tests/host/CMakeLists.txt`.
- `doc/ARCHITECTURE.md` still lists a BIOS initrd load buffer at `0x80000`; the code
  is the source of truth here.

### memory management

- Current reality: memory management has three distinct layers.
- The physical allocator is `PageFrameContainer` in `src/kernel/mm/page_frame.cpp`.
- It is seeded from the `BootInfo` memory map and reserves the bitmap, early low memory,
  kernel image, modules, framebuffer, RSDP mappings, and other boot-owned regions.
- The allocator still has BSP-only assertions for allocation, free, and reserve operations.
- Virtual memory is handled by `src/kernel/mm/virtual_memory.cpp`.
- It can allocate page tables, map ranges, protect ranges, translate virtual addresses,
  clone kernel mappings into user address spaces, destroy user slots, and free page tables.
- The kernel establishes a higher-half mapping and direct map in
  `src/kernel/core/kernel_main.cpp`.
- `src/kernel/mm/boot_mapping.cpp` provides direct-map and MMIO mapping helpers.
- `src/kernel/mm/kmem.cpp` is a direct-map-backed small-object allocator.
- `kmem` has builtin caches for 16, 32, 64, 128, 256, 512, and 1024 byte objects.
- It also supports named caches through `kmem_cache_create`.
- It has a large-allocation path for requests larger than 1024 bytes.
- Debug builds can use redzones, poisoning, leak dumps, and corruption detection.
- `kmem` corruption records `OS1_KERNEL_EVENT_KMEM_CORRUPTION` before halting.
- `kmem` is no longer unused infrastructure.
- `src/kernel/proc/process.cpp` uses `kmem` for process records.
- `src/kernel/proc/thread.cpp` uses `kmem` for thread records.
- `src/kernel/drivers/net/arp_cache.cpp` uses a named cache for ARP entries.
- `src/kernel/drivers/bus/device.cpp` uses a named cache for device binding records.
- `src/kernel/drivers/bus/resource.cpp` uses a named cache for PCI BAR claim records.
- `src/kernel/mm/dma.cpp` uses a named cache for DMA allocation records.
- `src/kernel/platform/irq_registry.cpp` uses a named cache for IRQ route records.
- `src/kernel/mm/dma.cpp` provides coherent DMA buffers, owner IDs, direction,
  and observable allocation records.
- DMA sync is effectively a barrier/no-op on x86_64, which is valid for the current
  target but should not be treated as portable.
- `src/kernel/mm/user_address.hpp` and `src/kernel/mm/user_copy.cpp` enforce user range,
  mapping, and writeability checks for syscall buffers.
- The user-copy path records `OS1_KERNEL_EVENT_USER_COPY_FAILURE` on failures.
- Missing memory work: user heap, `brk` or `mmap`, demand paging, copy-on-write,
  page-fault recovery, pinned user buffers, shared mappings, and a buffer cache.
- Recommendation: add pinned or copied I/O buffer rules before sockets and filesystem
  read/write paths start accepting arbitrary user buffers.

### interrupts / exceptions

- Current reality: interrupt and exception handling are usable on the BSP.
- `src/kernel/arch/x86_64/interrupt/interrupt.cpp` builds the IDT, remaps the PIC,
  installs exception handlers, and exposes handler registration.
- Assembly stubs in `src/kernel/arch/x86_64/interrupt/irqhandler.asm` save a
  `TrapFrame`, call C++ dispatch, and return or switch threads.
- `src/kernel/arch/x86_64/interrupt/trap_dispatch.cpp` separates syscall vector,
  external IRQs, and exceptions.
- `src/kernel/arch/x86_64/interrupt/irq_dispatch.cpp` records IRQ events,
  dispatches registered handlers, sends EOI through LAPIC/PIC paths, and integrates
  scheduler tick behavior.
- `src/kernel/arch/x86_64/interrupt/fault.cpp` kills the current user thread on
  user exceptions and halts on kernel exceptions.
- `src/kernel/arch/x86_64/interrupt/vector_allocator.cpp` provides dynamic interrupt
  vector allocation for PCI MSI paths.
- Timer setup prefers an HPET-calibrated LAPIC timer and falls back to PIT behavior
  through the current timer-source code.
- IRQ route records are observable through `OS1_OBSERVE_IRQS`.
- Missing interrupt work: per-CPU timer ownership, inter-processor interrupts,
  TLB shootdowns, threaded IRQs, interrupt affinity, and full AP-safe handler paths.
- Recommendation: do not expand AP scheduling until interrupt routing and TLB shootdown
  rules are designed.

### scheduling / SMP state

- Current reality: SMP discovery and AP startup exist, but scheduling remains single-CPU.
- `src/kernel/platform/topology.cpp` fills CPU topology from ACPI.
- `src/kernel/arch/x86_64/cpu/cpu.cpp` starts APs using the low-memory trampoline
  in `src/kernel/arch/x86_64/asm/cpu_start.asm`.
- APs initialize enough CPU state to mark themselves booted and then enter an idle loop.
- `src/kernel/sync/smp.hpp` defines `OS1_BSP_ONLY`, `KASSERT_ON_BSP`, `Spinlock`,
  and `IrqGuard`.
- The scheduler uses round-robin selection over process/thread registries in
  `src/kernel/sched/scheduler.cpp` and `src/kernel/sched/thread_queue.cpp`.
- Thread state supports ready, running, blocked, dying, and typed wait reasons.
- Blocking exists for console read, child exit, and block I/O completion.
- Dead thread reaping is deferred through `src/kernel/proc/reaper.cpp`.
- The dynamic process/thread registries remove the older fixed-table ceiling.
- `OS1_OBSERVE_CPUS` can show CPU boot state and current pid/tid fields.
- Missing scheduling work: per-CPU run queues, CPU affinity, scheduler locks,
  AP timer interrupts, remote wakeups, IPI reschedule, TLB shootdowns, accounting,
  priorities, and preemption rules for shared kernel state.
- Expanding SMP now would multiply race bugs in memory allocation, process/thread lists,
  event recording, console input, and driver paths.

### user/kernel separation state

- Current reality: ring-3 execution exists and has a narrow syscall ABI.
- User ELF loading is in `src/kernel/proc/user_elf.cpp` and
  `src/kernel/proc/user_program.cpp`.
- User programs are statically linked ET_EXEC images at `0x0000008000400000`
  through `src/user/linker.ld`.
- The kernel maps PT_LOAD segments with user permissions based on ELF flags.
- The kernel maps a fixed 16-page user stack below `kUserStackTop`.
- `src/kernel/arch/x86_64/cpu/syscall.cpp` sets up SYSCALL/SYSRET machinery.
- Syscalls are dispatched in `src/kernel/syscall/dispatch.cpp`.
- The current syscall set is enough for the shell and smokes but not for a general
  user environment.
- `sys_spawn` and `sys_exec` take only an initrd path string.
- There is no argv stack.
- There is no envp stack.
- There is no userspace heap API.
- There is no file descriptor table.
- There is no close, open, stat, chdir, getcwd, dup, pipe, socket, or ioctl-like control path.
- There is no errno ABI beyond negative syscall returns.
- Process parent/child and wait behavior exist, but lifecycle policy is still minimal.
- Recommendation: the next user/kernel work should define a minimal descriptor table,
  then route console read/write through it before adding files or sockets.

### storage / filesystem state

- Current reality: block I/O exists; filesystems do not.
- The block abstraction is `src/kernel/storage/block_device.hpp`.
- `BlockRequest` carries operation, sector, buffer, sector count, completion,
  status, byte count, and driver context.
- `src/kernel/storage/block_device.cpp` implements submit/wait helpers and chunked
  synchronous read/write wrappers.
- `virtio-blk` in `src/kernel/drivers/block/virtio_blk.cpp` supports interrupt-driven
  read and write requests.
- The driver bounds each request to the device maximum, currently up to 8 sectors
  for the 4 KiB smoke path.
- The driver has smoke coverage for read prefixes, write/readback, and multi-sector I/O.
- Initrd CPIO support is in `src/kernel/fs/cpio_newc.cpp` and
  `src/kernel/fs/initrd.cpp`.
- Initrd lookup is enough to load `/bin/init`, `/bin/sh`, `/bin/yield`,
  `/bin/fault`, `/bin/copycheck`, and `/bin/ascii`.
- There is no VFS implementation under `src/kernel/vfs/`.
- There is no persistent namespace, mount table, vnode, inode, dentry cache,
  path lookup, current working directory, per-process file table, or filesystem-backed exec.
- Recommendation: the first storage milestone should not be another block-driver feature.
- It should be a minimal VFS plus a read-only block-backed filesystem smoke that can
  list a directory, open a file, read it through a descriptor, and exec from it.

### drivers / device model

- Current reality: the device model is small but real.
- PCI devices are enumerated through `src/kernel/platform/pci.cpp`.
- Driver registration lives in `src/kernel/drivers/bus/driver_registry.cpp`.
- PCI driver probing lives in `src/kernel/drivers/bus/pci_bus.cpp`.
- Device binding records live in `src/kernel/drivers/bus/device.cpp`.
- BAR ownership records live in `src/kernel/drivers/bus/resource.cpp`.
- IRQ route ownership lives in `src/kernel/platform/irq_registry.cpp`.
- DMA buffer ownership lives in `src/kernel/mm/dma.cpp`.
- `src/kernel/platform/device_probe.cpp` registers and probes `virtio-blk`,
  `virtio-net`, and `xhci`.
- `src/kernel/drivers/virtio/pci_transport.cpp` centralizes modern virtio PCI capability
  discovery and transport setup.
- `virtio-blk` uses the shared virtio transport for block I/O.
- `virtio-net` uses the shared virtio transport for one RX and one TX queue.
- xHCI handles root-port enumeration and HID boot keyboard input.
- PS/2 keyboard support still exists in `src/kernel/drivers/input/ps2_keyboard.cpp`.
- Text display support includes VGA and framebuffer-backed terminal output in
  `src/kernel/drivers/display/text_display.cpp`.
- Missing driver-model work: device tree hierarchy, stable device names, user-visible
  device nodes, hotplug policy, hub support, mass storage, power-state policy, reset
  recovery, and per-device permissions.
- Recommendation: expose devices through VFS or handles only after the descriptor
  model is chosen, otherwise the kernel will grow incompatible control paths.

### networking state

- Current reality: networking is a driver smoke, not a network stack.
- `src/kernel/drivers/net/virtio_net.cpp` negotiates a virtio-net device, sets up
  RX/TX queues, obtains MAC information, sends an ARP probe, parses ARP replies,
  and records RX events.
- `src/kernel/drivers/net/arp_cache.cpp` stores IPv4-to-MAC entries using a `kmem`
  named cache.
- `OS1_KERNEL_EVENT_NET_RX` and observe/event plumbing make the smoke visible.
- There is no packet allocator beyond current driver-local buffers.
- There is no Ethernet dispatch layer.
- There is no IPv4 input/output layer.
- There is no ICMP, UDP, TCP, DHCP, DNS, routing table, socket API, or user-visible
  networking syscall.
- There is no entropy source for secure networking.
- Recommendation: before TCP or SSH, add a packet buffer pool, Ethernet demux,
  IPv4/ARP ownership, checksum helpers, ICMP ping, UDP, DHCP/static configuration,
  and net observability.

### security / multiuser state

- Current reality: hardware user/kernel separation exists; OS-level security does not.
- `src/kernel/security/README.md` is only a placeholder.
- User pages are marked user-accessible and kernel pages are not.
- Syscall copies validate user addresses.
- User exceptions kill the current thread rather than corrupting kernel state.
- Process records do not contain uid, gid, groups, capabilities, umask, resource limits,
  login identity, session, controlling terminal, or namespace state.
- There is no permission model for files, devices, processes, networking, or observe data.
- There is no authentication, password storage, key handling, entropy, or audit log.
- `observe` currently exposes broad kernel state to any user program.
- That is acceptable for a teaching debug shell but incompatible with multiuser goals.
- Recommendation: add credentials and an observe-access policy before remote login.

### GUI / terminal state

- Current reality: terminal is usable; GUI is not started.
- Kernel console output is routed through `src/kernel/console/console.cpp`.
- Canonical console input is managed by `src/kernel/console/console_input.cpp`.
- The terminal buffer implementation is `src/kernel/console/terminal.cpp`.
- There are 12 logical terminals and F1-F12 switching through
  `src/kernel/console/terminal_switcher.cpp`.
- Framebuffer text output supports RGB/BGR, 24/32 bpp, 8x16 font rendering,
  shadow-buffer diffing, scrolling, and cursor inversion.
- The shell is a single interactive user program backed by console read/write.
- There is no PTY.
- There is no session model.
- There is no terminal device node.
- There is no compositor, window server, GUI input model, or mouse event routing.
- Recommendation: the first GUI milestone should wait for PTYs and a file/handle model.
- A framebuffer terminal compositor would then have a clean way to host multiple shells.

### build, tooling, and debug workflow

- Current reality: the build and debug workflow is stronger than the runtime feature set.
- Root `CMakeLists.txt` builds `os1.iso`, `os1.raw`, `kernel.elf`,
  `kernel_limine.elf`, `initrd.cpio`, and `virtio-test-disk.raw`.
- The build expects an `x86_64-elf-*` cross toolchain, NASM, CMake, Ninja, CPIO,
  xorriso, and QEMU/OVMF for full smoke coverage.
- User programs are built under `src/user/` and staged into the initrd.
- Host tests are configured by `tests/host/CMakeLists.txt`.
- The root smokes cover shell startup, observe, spawn, exec, BIOS variants, and xHCI.
- `smoke_all` groups the QEMU smoke suite.
- The shell exposes `sys`, `ps`, `cpu`, `pci`, `initrd`, `devices`, `irqs`,
  `resources`, `kmem`, `events`, `exec`, and basic command smokes.
- Serial logs remain the primary low-level debug stream.
- Recommendation: keep growing observe records with each new subsystem and make
  smoke tests assert new invariants rather than only checking boot text.

## 4. Review of progress against goals

### project purpose

- Intended design from `GOALS.md`: a transparent educational OS with real modern features.
- Current reality: the code satisfies the educational substrate goal well.
- Evidence: boot, memory, interrupts, user mode, ACPI, PCI, virtio, xHCI, block I/O,
  observe, and host tests are all inspectable in source.
- Gap: the next lessons are about namespace, persistence, credentials, and protocols,
  and those are not yet represented in code.
- Recommendation: write the next milestone as a vertical slice, not a broad design essay.

### modern x86_64 boot and handoff

- Completion level: high for current project scope.
- Implemented: Limine/UEFI default, BIOS compatibility, shared `BootInfo`,
  higher-half kernel, direct map, module handoff, framebuffer handoff, ACPI handoff.
- Remaining: document hard caps, reduce fixed BIOS-era staging assumptions where practical,
  and keep the Limine shim from accumulating kernel policy.
- Blocker to later goals: none for VFS, networking, or GUI.
- Sequencing: boot should now receive maintenance, not lead the roadmap.

### memory management and allocator substrate

- Completion level: medium-high.
- Implemented: page frames, page tables, direct map, user address spaces, user copy,
  DMA allocation records, `kmem`, debug instrumentation, observe stats, real `kmem` consumers.
- Missing: user heap, mmap-like API, page-fault recovery, buffer cache, pinning rules,
  copy-on-write, and SMP-safe allocator policy.
- Blocker to later goals: buffer cache and pinned/copying I/O rules block robust file
  and socket I/O.
- Sequencing: add descriptor/VFS first, then buffer cache and user I/O rules as soon as
  file read/write paths exist.

### interrupts, timers, and platform support

- Completion level: medium-high for QEMU/q35.
- Implemented: IDT, exception dispatch, LAPIC, IOAPIC, HPET, vector allocator,
  MSI-X/MSI/INTx fallback, IRQ registry, event recording.
- Missing: AP timers, IPIs, TLB shootdowns, interrupt affinity, threaded interrupts,
  real-hardware recovery paths.
- Blocker to later goals: AP scheduling and high-throughput networking need the missing
  per-CPU pieces.
- Sequencing: keep BSP-only scheduling while filesystem and credentials land.

### process model and userland

- Completion level: medium.
- Implemented: ring-3 ELF loading, address-space isolation, spawn, exec, waitpid,
  exit, yield, getpid, read/write console, observe, dynamic process/thread registries.
- Missing: argv, envp, descriptor table, cwd, file-backed exec, user heap, sessions,
  process groups, credentials, signals or equivalent cancellation.
- Blocker to later goals: descriptors and argv/envp block usable shell programs,
  file commands, PTYs, networking tools, and SSH.
- Sequencing: argv/envp and descriptors should land before adding more shell language.

### storage and filesystem

- Completion level: block medium-high; filesystem low.
- Implemented: request-shaped block device, virtio-blk read/write, multi-sector smoke,
  initrd CPIO.
- Missing: partition policy, first filesystem, VFS, path lookup, fd-backed file I/O,
  filesystem-backed exec, metadata, permissions, cache, sync semantics.
- Blocker to later goals: Milestone D cannot complete without this layer.
- Sequencing: VFS skeleton and descriptor table must come before a user-visible FS API.

### device model and buses

- Completion level: medium-high.
- Implemented: PCI enumeration, driver registry, bind/remove records, BAR claims,
  DMA records, IRQ route records, virtio transport, virtio-blk, virtio-net, xHCI,
  PS/2 keyboard, display backends.
- Missing: stable device namespace, device nodes, user access controls, hotplug policy,
  hub/mass-storage support, recovery after driver failure, richer power policy.
- Blocker to later goals: device namespace and permissions are needed before multiuser
  and userland tooling can safely touch devices.
- Sequencing: expose devices through the same descriptor/VFS path used for files.

### networking

- Completion level: low to low-medium.
- Implemented: virtio-net device startup, basic TX/RX, ARP probe parsing, ARP cache,
  NET_RX event recording.
- Missing: packet allocator, Ethernet demux, IPv4, ICMP, UDP, TCP, DHCP, DNS,
  sockets, net device configuration, routing, user tools, entropy-backed security.
- Blocker to later goals: SSH and remote shell are not reachable from the current stack.
- Sequencing: filesystem and credentials should come before SSH; packet foundation
  can start before full filesystem if kept kernel-internal.

### security and multiuser

- Completion level: very low.
- Implemented: ring separation and safe user-copy checks.
- Missing: credentials, file permissions, device permissions, process permissions,
  session/login model, authentication, audit, observe restrictions, key handling,
  entropy, least-privilege service separation.
- Blocker to later goals: SSH requires almost all of these.
- Sequencing: minimal credentials should land with the first persistent namespace,
  not after it.

### GUI and terminal

- Completion level: terminal medium; GUI low.
- Implemented: console input, terminal buffers, multiple virtual terminals,
  framebuffer text rendering, shell.
- Missing: PTY, sessions, terminal devices, compositor, graphics primitives,
  mouse routing, window management, GUI server boundary.
- Blocker to later goals: GUI should wait for PTYs and descriptor-backed I/O.
- Sequencing: terminal compositor after VFS/descriptors/PTY, not before.

### build, debug, and test goals

- Completion level: high relative to runtime stage.
- Implemented: host tests, QEMU smokes, BIOS and UEFI smoke variants, xHCI smoke,
  serial-first debug, observe ABI, event ring, shell inspect commands.
- Missing: filesystem smoke, network protocol smoke, credentials smoke, SMP invariant tests,
  documentation freshness checks.
- Blocker to later goals: no direct blocker, but missing smokes will slow safe growth.
- Sequencing: every new major subsystem should add both a host test and a QEMU smoke marker.

### portability boundaries

- Completion level: partial.
- Implemented: code is organized into `arch/x86_64`, `platform`, `drivers`, `mm`,
  and generic kernel subsystems.
- Missing: architecture-neutral interrupt model, portable DMA/cache semantics,
  portable page-table abstraction, non-ACPI platform story, target-independent UAPI review.
- Blocker to later goals: not blocking x86_64 completion, but easy to erode while
  adding filesystems, networking, and GUI.
- Sequencing: enforce boundary comments and headers before adding a second architecture.

### accelerator awareness

- Completion level: not started beyond PCI discovery foundations.
- Implemented: PCI enumeration, BAR ownership, DMA allocation, MSI routing.
- Missing: GPU/accelerator class discovery policy, IOMMU model, pinned memory,
  command queues, user ownership, fault isolation, scheduling, reset handling.
- Blocker to later goals: not a near-term blocker.
- Sequencing: should come after descriptor/security/DMA ownership is much stronger.

## 5. Overlooked goals / blind spots

- The project needs a small, committed descriptor model now.
- The current drafts explore native objects, VFS objects, ELF interfaces, and shell language,
  but code needs a minimal contract that can be implemented.
- Recommendation: define per-process handles with rights, type tags, refcounts,
  and `read`/`write`/`close` behavior before designing a large object system.
- The project needs a rule for compatibility with POSIX concepts.
- `GOALS.md` says POSIX compatibility is later, but fd-like numbers already exist in
  `read` and `write`.
- Recommendation: preserve fd compatibility at the syscall edge even if the internal
  kernel object model is native.
- The project needs an error ABI.
- Current syscalls return negative values, but there is no documented stable errno set.
- Filesystems, sockets, and permissions will multiply error cases.
- Recommendation: define a small `os1_status` or errno-compatible set before VFS.
- The project needs a time model before networking security.
- HPET/LAPIC timer support exists for scheduler ticks, but networking and SSH need
  monotonic timeouts and eventually wall-clock time.
- Recommendation: add monotonic time syscall and timeout primitives before TCP.
- The project needs entropy before authentication or SSH.
- There is no RNG or entropy collection path in current code.
- Recommendation: explicitly block SSH until an entropy source and key storage policy exist.
- The project needs PTY/session goals before remote shell.
- SSH is not just TCP plus crypto; it needs login sessions, terminal allocation,
  process lifetime, and permissions.
- Recommendation: implement local PTYs and sessions before remote login.
- The project needs observe-access policy.
- `observe` is excellent for teaching and debugging, but it leaks kernel-wide state.
- Recommendation: preserve full observe for debug builds or privileged users, then define
  restricted views for ordinary users.
- The project needs documentation freshness checks.
- Current docs already contradict code on `kmem`, task tables, timer source, and memory layout.
- Recommendation: add a lightweight checklist or CI check for dated docs when major
  subsystem claims change.
- The project needs resource lifetime rules for asynchronous work.
- Block I/O, network RX, process exit, hot-remove, and future sockets all need cancellation,
  ownership, and wakeup rules.
- Recommendation: define wait objects or wait queues before adding many more blocking APIs.
- The project needs an SMP readiness rubric.
- `OS1_BSP_ONLY` is useful, but there is no checklist for removing it subsystem by subsystem.
- Recommendation: document which locks, data structures, and interrupt paths must change
  before APs can run user threads.
- The project needs a buffer cache plan.
- A filesystem without a cache is simple but will either be slow or force ad hoc caches later.
- Recommendation: design the first FS so a cache can be inserted without rewriting VFS.
- The project needs a test image format decision.
- `virtio-test-disk.raw` exists for block smokes, but the first filesystem needs a deterministic
  image generator and inspection tools.
- Recommendation: choose the FS format and image tooling together.
- The project needs a hardware support boundary.
- Real hardware support is a goal, but current assumptions are QEMU-friendly.
- Recommendation: state which real machines or device classes are in scope before expanding
  USB, storage, or GPU work.
- The project needs ABI versioning discipline.
- `observe` has `OS1_OBSERVE_ABI_VERSION`; future file, socket, process, and security APIs
  need the same discipline.
- Recommendation: keep UAPI headers under `src/uapi/os1/` authoritative and small.

## 6. Roadmap

### near-term milestones

- Milestone 1: decide the user-visible resource model.
- Deliverable: one short document and UAPI sketch for descriptors or handles.
- Required scope: per-process table, rights bits, type tags, refcounting, close semantics,
  blocking behavior, inheritance across spawn/exec, and console descriptors 0/1/2.
- Recommendation: implement a minimal handle table with fd-compatible integer slots.
- Why first: it unblocks files, devices, PTYs, sockets, shell redirection, and SSH.
- Milestone 2: add argv/envp support to user process startup.
- Deliverable: kernel stack builder for argc/argv/envp and shell support for passing
  tokens to spawned programs.
- Why now: every serious userland command needs arguments.
- Milestone 3: route console I/O through the descriptor table.
- Deliverable: `read(0)` and `write(1/2)` should go through process-owned descriptor entries.
- Why now: this validates the descriptor design before files and sockets depend on it.
- Milestone 4: add a minimal VFS skeleton.
- Deliverable: vnode/inode or object equivalent, mount root, path lookup, open, close,
  read, stat-like metadata, and initrd as the first backend.
- Why now: initrd-backed VFS lets the kernel prove path and descriptor behavior without
  disk-format complexity.
- Milestone 5: add filesystem-backed exec through VFS.
- Deliverable: `/bin/init` and shell programs can still come from initrd, but `exec`
  uses VFS lookup rather than direct initrd lookup.
- Why now: this prevents the filesystem milestone from becoming a second exec path.
- Milestone 6: choose and implement the first block-backed filesystem.
- Deliverable: deterministic image tool, read-only mount, directory listing, open/read,
  host tests for parser/path behavior, and QEMU smoke reading a file from virtio-blk.
- Why now: Milestone D in `GOALS.md` is blocked on persistent storage above `virtio-blk`.
- Milestone 7: add minimal credentials with the filesystem.
- Deliverable: uid/gid fields in `Process`, root/non-root distinction, permission checks
  for observe and filesystem metadata, and a policy for init.
- Why now: retrofitting permissions after a writable namespace is harder and riskier.

### mid-term milestones

- Milestone 8: add a write path only after read-only filesystem behavior is stable.
- Deliverable: create/write/truncate or a narrower append/update contract, dirty cache
  policy, and a sync story.
- Dependency: descriptor table, VFS, block-backed FS, buffer-cache plan.
- Milestone 9: add PTYs and sessions.
- Deliverable: PTY master/slave objects, controlling terminal, session/process-group
  minimum, shell launched on a PTY, and terminal observe records.
- Dependency: descriptor table and VFS/device namespace.
- Milestone 10: build networking foundation above virtio-net.
- Deliverable: packet buffers, Ethernet demux, ARP ownership, IPv4, ICMP ping,
  UDP, DHCP/static IPv4 config, and net observe records.
- Dependency: memory buffer rules, timer/timeouts, and a stable internal netdev API.
- Milestone 11: add TCP only after UDP/DHCP/timeouts are stable.
- Deliverable: TCP state machine sufficient for one client and one server path,
  retransmit timers, connection objects, and socket descriptors.
- Dependency: packet buffers, monotonic time, descriptors, and wait queues.
- Milestone 12: add local login and permissions before SSH.
- Deliverable: account database format, authentication placeholder or password hash policy,
  login program, permission checks for files/devices/process observe, and audit events.
- Dependency: persistent filesystem, credentials, sessions, PTYs, entropy policy.
- Milestone 13: add SSH as a late-mid-stage integration project.
- Deliverable: remote shell over TCP with authenticated login and PTY allocation.
- Dependency: TCP, entropy, credentials, permissions, PTYs, sessions, file I/O,
  process isolation, and enough timekeeping for protocol timeouts.

### later milestones

- Milestone 14: expand SMP from parked APs to real scheduling.
- Deliverable: per-CPU run queues or a locked global scheduler, AP timer interrupts,
  IPI reschedule, TLB shootdowns, remote wakeups, allocator/process/thread lock audit,
  and SMP observe fields.
- Dependency: explicit locking plan and subsystem readiness checklist.
- Milestone 15: first GUI as a framebuffer terminal compositor.
- Deliverable: compositor process or kernel service, multiple PTY-backed terminal windows,
  keyboard focus, mouse event routing, redraw discipline, and crash recovery policy.
- Dependency: PTYs, sessions, framebuffer/input event model, descriptors, and memory policy.
- Milestone 16: USB expansion.
- Deliverable: hub support, multiple HID devices, mass storage, hotplug policy,
  and stable device naming.
- Dependency: device namespace, permissions, descriptor model, and driver remove safety.
- Milestone 17: accelerator discovery and constrained experiments.
- Deliverable: PCI GPU/accelerator inventory, BAR/DMA ownership, command queue prototype,
  IOMMU or explicit no-IOMMU threat model, and user ownership rules.
- Dependency: security model, DMA pinning, descriptor/device access policy.
- Milestone 18: portability beyond x86_64.
- Deliverable: architecture-neutral interrupt/time/page abstractions and a second target
  booting far enough to run host-equivalent kernel unit tests or a serial smoke.
- Dependency: stabilized x86_64 kernel interfaces and reduced ACPI/x86 leakage.

### explicit ordering constraints

- Before networking: finish descriptor semantics or keep networking entirely kernel-internal.
- Before networking: add packet buffer ownership and monotonic timeout primitives.
- Before networking: decide whether sockets are descriptors and how blocking/wakeup works.
- Before SSH: finish TCP, entropy, credentials, permissions, PTYs, sessions, and file I/O.
- Before GUI: finish PTYs, sessions, input event routing, and framebuffer ownership.
- Before SMP expansion: complete the locking audit, TLB shootdown design, AP timers,
  and scheduler wakeup model.
- Before POSIX growth: decide which POSIX ideas are compatibility requirements and which
  native APIs are intentionally different.
- Before accelerator support: define DMA pinning, device permissions, reset policy,
  and the no-IOMMU risk model.
- Before writable filesystems: define cache, sync, crash, and corruption behavior.
- Before real-hardware claims: test at least one non-QEMU machine class and record
  which firmware/device assumptions failed.

## 7. Immediate next steps

1. Decide and document the minimal descriptor/handle contract.
   This matters now because VFS, devices, sockets, PTYs, shell redirection, and SSH all
   depend on the same resource lifetime model.

2. Implement a per-process descriptor table and route console fd 0/1/2 through it.
   This matters now because it tests the design with existing behavior before adding
   filesystem complexity.

3. Add argv/envp stack construction for `spawn` and `exec`.
   This matters now because the shell cannot become useful without argument passing,
   and later file/network tools would otherwise need another ABI break.

4. Add a minimal VFS with initrd as the first mounted backend.
   This matters now because it converts current initrd-specific program loading into
   a general path lookup mechanism.

5. Move `exec` to VFS lookup while preserving initrd boot behavior.
   This matters now because filesystem-backed userland should not create a second
   program-loading path.

6. Choose the first block-backed filesystem and add a deterministic image builder.
   This matters now because `virtio-blk` is already capable enough; the missing value
   is persistent namespace semantics, not more block smoke.

7. Add read-only filesystem QEMU smoke coverage.
   This matters now because the next milestone should prove open/read/exec behavior
   from disk, not just driver-level sector access.

8. Add minimal credentials to `Process`.
   This matters now because file permissions, observe restrictions, login, and SSH
   should not be retrofitted after the namespace is already broad.

9. Add a monotonic time syscall and kernel timeout primitive.
   This matters now because block waits, network retransmits, DHCP, TCP, and SSH
   all need coherent timeouts.

10. Define a packet buffer pool before extending `virtio-net`.
    This matters now because IP/UDP/TCP should not be built on one-off driver-local
    buffers that will be hard to replace.

11. Start an SMP readiness checklist, but keep AP scheduling disabled.
    This matters now because the code already boots APs, and the next step must be
    deliberate locking and TLB-shootdown work rather than opportunistic scheduling.

12. Correct the stale documentation called out in this review.
    This matters now because `GOALS.md` and `doc/ARCHITECTURE.md` are beginning to
    understate current code and misdirect next-step planning.

13. Add `doc/os-api-draft/README.md`.
    This matters now because the API drafts are large enough to look authoritative,
    but the kernel has not committed to them.

14. Document the `BootInfo` capacity limits.
    This matters now because the limits are part of the boot ABI and can affect real
    hardware before the rest of the system notices.

15. Keep the next milestone narrow.
    This matters now because the project has enough substrate; completion now depends
    on sequencing the dependency chain, not adding isolated impressive demos.
