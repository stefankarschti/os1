## 1. Executive Summary

### Current project state

`os1` is a working x86_64 educational OS kernel with two boot frontends, a shared low-half kernel entry point, physical and virtual memory management, interrupt handling, a cooperative/preemptive single-CPU scheduler, ring-3 ELF user programs, an initrd-backed shell environment, ACPI/PCIe enumeration, a framebuffer/text terminal path, and CI-backed QEMU smoke tests.

The implemented system is best described as a strong Milestone 5 vertical slice, not as a general OS substrate yet. It can boot through BIOS/raw and Limine/UEFI paths, start userland from an initrd, run shell builtins and spawned user programs, observe kernel/platform state, contain user exceptions, and perform a narrow `virtio-blk` sector-read smoke test. It does not yet have persistent storage, a filesystem, a real block layer, a network stack, SSH, multiuser security, a GUI, or SMP scheduling beyond bringing APs up to an idle loop.

The most important correction to the current narrative is that ring-3 isolation exists at the CPU permission level, but the syscall copy boundary is not yet robust. `CopyToUser` and `CopyFromUser` translate addresses but do not enforce the `PageFlags::User` bit, and `CopyToUser` also does not enforce `PageFlags::Write` (`src/kernel/kernel.cpp`). Because user address spaces clone the kernel PML4 slot, syscalls can become copy gadgets against supervisor mappings. That is an immediate correctness and security issue, not a future hardening item.

### Strongest parts

- The boot architecture is coherent. The BIOS path (`src/boot/bios/boot.asm`, `src/boot/bios/kernel16.asm`, `src/boot/bios/long64.asm`) and Limine path (`src/boot/limine/entry.cpp`) converge on a packed `BootInfo` contract (`src/kernel/bootinfo.h`, `src/kernel/bootinfo.cpp`) before calling `KernelMain` (`src/kernel/kernel.cpp`).
- The project has real end-to-end execution slices. User programs are built, staged into `initrd.cpio`, loaded as ELF, scheduled, and exercised by QEMU smoke tests (`src/user/CMakeLists.txt`, `scripts/smoke_qemu.py`, `CTestTestfile` generated by CMake).
- The current diagnostic surface is useful for a young kernel. `sys_observe` exposes structured kernel/process/platform/initrd state through `src/uapi/os1/observe.h`, and the shell has `sys`, `ps`, `cpu`, `pci`, and `initrd` builtins (`src/user/programs/sh.cpp`).
- Fault containment is real for basic user-mode exceptions. User exceptions are routed through the unified trap frame path and kill the current process rather than halting the kernel (`src/kernel/irqhandler.asm`, `src/kernel/interrupt.cpp`, `src/kernel/kernel.cpp`).
- The build workflow is unusually mature for this stage: cross-toolchain CMake checks, raw and ISO images, UEFI detection, QEMU runners, and GitHub Actions CI (`CMakeLists.txt`, `run.sh`, `.github/workflows/ci.yml`).

### Biggest risks / gaps

- User-copy validation is permission-incomplete. This is the highest-priority bug because it undermines the user/kernel boundary exposed by syscalls.
- The kernel is still a low identity-mapped design cloned into every process address space. That is acceptable for the current educational stage, but it amplifies user-copy mistakes and will complicate later security work.
- Storage is a smoke test, not a subsystem. `virtio-blk` probing and single-sector polling live inside `src/kernel/platform.cpp`; there is no block device abstraction, request lifecycle, filesystem, cache, or persistent namespace.
- SMP is discovery plus AP idle. APs are started, but scheduling remains BSP-only. Global console, terminal, allocator, process, and driver state should not be assumed SMP-safe.
- `src/kernel/kernel.cpp` and `src/kernel/platform.cpp` are too large and own too many subsystem boundaries. This will slow the next phase unless syscall handling, ELF/initrd loading, process control, ACPI, PCI, and virtio code are split deliberately.
- The docs are mostly current, but some statements now overstate security and platform maturity. The code is ahead of some historical docs and behind some intended-design statements.

## 2. Current State From Code

### Boot path and platform assumptions

Current reality:

- There are two boot frontends and one shared kernel core.
- The BIOS/raw path starts in `src/boot/bios/boot.asm`, transitions through `src/boot/bios/kernel16.asm` and `src/boot/bios/long64.asm`, loads the kernel at low physical memory, builds a `BootInfo`, and enters `KernelMain`.
- The Limine/UEFI path in `src/boot/limine/entry.cpp` requests Limine memory map, HHDM, framebuffer, modules, RSDP, SMBIOS, bootloader, and command-line data. It translates Limine virtual addresses to physical addresses by walking the active page tables, loads the low kernel module, installs identity mappings, builds `BootInfo`, and calls the same `KernelMain`.
- The shared handoff contract is `BootInfo` in `src/kernel/bootinfo.h`. It carries boot source, kernel physical range, RSDP/SMBIOS addresses, bootloader strings, framebuffer details, memory map entries, and module descriptors.
- `KernelMain` in `src/kernel/kernel.cpp` owns the boot sequence after handoff: serial init, `BootInfo` ownership, page-frame allocator init, page-table setup, platform discovery, interrupt controller setup, AP startup, display/terminal setup, IDT setup, task table setup, user init loading, PIT setup, and scheduler start.

Platform assumptions:

- The primary target is x86_64 under QEMU.
- ACPI is the main platform discovery path. Legacy MP fallback exists for BIOS cases where ACPI is absent or invalid (`src/kernel/platform.cpp`, `src/kernel/mp.cpp`).
- PCIe ECAM is expected through ACPI MCFG for modern device enumeration (`src/kernel/platform.cpp`).
- Limine/UEFI is the more capable boot path because it provides framebuffer and richer firmware data. BIOS still exists and is tested.
- The kernel is linked and run in low memory with identity mappings, not as a higher-half kernel.

Assessment:

The boot design is consistent and useful. The risk is not boot fragility anymore; it is that the low identity-mapped kernel and cloned kernel PML4 slot are now becoming security-relevant as userland grows.

### Memory management

Current reality:

- `src/kernel/pageframe.cpp` implements a bitmap physical page-frame allocator initialized from `BootInfo` memory regions.
- The allocator reserves the bitmap itself, low bootstrap memory, the kernel image range, boot modules, and framebuffer pages.
- `src/kernel/memory_layout.h` defines fixed early physical addresses and user virtual layout constants, including `kUserBase`, `kUserImageBase`, `kUserStackTop`, user stack page count, kernel stack page count, and the user PML4 slot.
- `src/kernel/virtualmemory.cpp` implements page table allocation, `MapPhysical`, `AllocateAndMap`, `Protect`, `Translate`, `CloneKernelPml4Entry`, and `DestroyUserSlot`.
- `KernelMain` builds a kernel PML4 that identity maps low physical memory, usable memory, modules, framebuffer, and platform tables before switching CR3 (`src/kernel/kernel.cpp`).
- User processes get their own PML4, but the kernel identity slot is cloned into them (`src/kernel/kernel.cpp`).

Important limitations:

- There is no heap allocator in the usual kernel sense. Most core resources are static, fixed tables, or page-frame allocations.
- There is no copy-on-write, demand paging, page fault allocation, mmap, shared memory, or kernel/user direct-map distinction.
- `src/kernel/pageframe.cpp` still carries allocator TODOs around checking whether the bitmap fits in reserved RAM and optimizing the first-free scan.
- User address validation is currently address-translation based, not permission-aware. That belongs in this section and in the user/kernel separation section because it crosses both boundaries.

Assessment:

The memory system is sufficient for the current boot, initrd, and fixed-process model. It is not yet safe enough for adversarial userland, persistent storage buffers, or network packet paths.

### Interrupts / exceptions

Current reality:

- `src/kernel/interrupt.cpp` builds the IDT, installs exception handlers, IRQ handlers, and an `int 0x80` syscall gate.
- The syscall gate is DPL 3, so user programs invoke syscalls with `int $0x80` through stubs in `src/user/lib/syscall.c` and `src/user/lib/syscall_cpp.cpp`.
- `src/kernel/irqhandler.asm` saves a common trap frame and enters `trap_dispatch`.
- `trap_dispatch` in `src/kernel/kernel.cpp` routes timer IRQs, keyboard IRQs, syscalls, and CPU exceptions.
- User-mode exceptions mark the current process as dying and schedule away. Kernel-mode exceptions halt.
- The PIT is initialized to 1000 Hz through `init_timer(1000)` (`src/kernel/kernel.cpp`, `src/kernel/timer.cpp`).
- PIC, IOAPIC, and LAPIC code all exist. `src/kernel/pic.cpp`, `src/kernel/ioapic.cpp`, and `src/kernel/lapic.cpp` are used depending on platform setup.

Important limitations:

- The interrupt story still carries both legacy PIC-era behavior and APIC behavior. That is workable during transition, but the ownership model is not clean.
- There is no MSI/MSI-X support yet, which blocks a mature virtio path.
- Syscalls are still interrupt-gate based. `GOALS.md` explicitly leaves `int 0x80` vs `SYSCALL/SYSRET` as an open decision.

Assessment:

The interrupt/exception layer is functional for one running CPU and simple devices. It needs cleanup before becoming a multi-device, SMP, interrupt-driven driver substrate.

### Scheduling / SMP state

Current reality:

- `src/kernel/task.cpp` implements fixed-size process and thread tables.
- `kMaxProcesses` and `kMaxThreads` are both 32 (`src/kernel/task.h`).
- There is one kernel idle thread and one runnable-thread selection path.
- Scheduling is round-robin over runnable threads. Threads can be blocked for console input or child exit.
- `spawn`, `exec`, `exit`, `waitpid`, `yield`, and user exception death are represented in the process model (`src/kernel/kernel.cpp`, `src/kernel/task.cpp`).
- APs can be discovered and started through ACPI/MADT or legacy MP data (`src/kernel/platform.cpp`, `src/kernel/cpu.cpp`, `src/kernel/mp.cpp`).
- AP entry initializes per-CPU state and then calls `cpu_idle_loop()` (`src/kernel/cpu.cpp`).

Important limitations:

- Scheduling is BSP-only. APs are alive but not participating in the scheduler.
- The locking model is not ready for general SMP. There are spinlocks in places, but global process, terminal, console input, device, and allocator state should be audited before enabling work on APs.
- There are no per-CPU run queues, load balancing, scheduler IPIs, or cross-CPU wakeups.
- There is no process hierarchy beyond parent/child wait and reaping. No sessions, process groups, credentials, or signals.

Assessment:

The current scheduler is appropriate for Milestone 5 userland. It should not be expanded to AP execution until the kernel has an explicit preemption, locking, and interrupt-affinity design.

### User/kernel separation state

Current reality:

- User programs are statically linked ELF executables placed in the initrd by `src/user/CMakeLists.txt`.
- The shell binary is staged as both `/bin/init` and `/bin/sh`; `src/user/programs/init.c` exists but is not built into the initrd.
- User programs run at high virtual addresses defined by `src/user/linker.ld` and `src/kernel/memory_layout.h`.
- The syscall ABI is defined in `src/kernel/syscall_abi.h` and mirrored by user stubs.
- Basic process isolation works for direct user-mode memory access: user page faults are caught, reported, and the offending process is killed.
- The kernel clones its PML4 slot into process address spaces so the kernel remains mapped while handling traps and syscalls.

Critical finding:

- `CopyToUser` and `CopyFromUser` in `src/kernel/kernel.cpp` call `Translate` and then copy through translated physical addresses, but they do not reject supervisor-only pages. `CopyToUser` also does not require the destination mapping to be writable.
- This means the kernel is not merely validating "is this address mapped"; it needs to validate "is this address mapped as user-accessible, and for writes, user-writable." The current code obtains page flags but does not enforce those permission bits.
- Because user CR3 values include the cloned kernel mapping, a user process can potentially pass kernel virtual addresses to syscalls and cause the kernel to read or write supervisor memory on its behalf.

Assessment:

The ring transition, ELF loading, and fault path are real. The user/kernel boundary is still incomplete until user-copy helpers enforce user permissions and tests prove syscalls cannot access supervisor mappings.

Assessment:

Storage is currently a platform smoke. It is not a storage subsystem. Filesystem work should not start until a minimal `BlockDevice` interface exists and virtio-blk is moved behind it.

## 5. Overlooked Goals / Blind Spots

- Treat adversarial userland as a current test case, not a later security milestone. The user-copy bug shows that even a single-user OS needs hostile syscall tests as soon as ring 3 exists.
- Define the kernel trust boundary explicitly. The project needs one small, audited user-copy API that checks canonicality, user range, page presence, `User`, and `Write` permissions as appropriate.
- Add host-side tests for pure logic. CPIO parsing, ELF validation, ACPI table walking, PCI capability parsing, page-table permission checks, and syscall argument validation can be tested without QEMU.
- Create a subsystem ownership map. `kernel.cpp` and `platform.cpp` are now boundary violations. The architecture should define owners for process, syscall, initrd, ELF, ACPI, PCI, virtio, console, and terminal code.
- Define lock ordering and interrupt context rules before SMP work. Without that, AP scheduling will create intermittent bugs in console, process, allocator, and device paths.
- Decide what "educational simplicity" means when it conflicts with security. Low identity mapping is simple, but the project should document the risks and the migration point.
- Version or centralize the syscall ABI. `src/kernel/syscall_abi.h` is the right anchor, but userland and kernel should avoid drift as calls grow.
- Add observability for timing and state transitions. Current observe calls are useful snapshots, but storage and networking will need event traces, error counters, IRQ counters, and queue state.
- Plan filesystem image tooling early. Persistence is not just kernel code; it also needs reproducible images, mkfs integration, test fixtures, and corruption cases.
- Establish portability boundaries. The code is intentionally x86_64-specific today, but `arch/x86_64` boundaries should be planned before platform code spreads further.
- Define non-goals. For example: no POSIX completeness before storage; no GUI before ptys/sessions; no SMP scheduling before lock audit; no accelerators before DMA ownership.

## 6. Roadmap

### Near-term milestones

1. Fix syscall user-copy permissions.

   Require `PageFlags::User` for all user copies. Require `PageFlags::Write` for `CopyToUser`. Reject non-canonical and out-of-user-range addresses before translation. Add a negative user program or smoke path that attempts to pass kernel addresses to `read`, `observe`, `waitpid`, or other copying syscalls.

2. Extract subsystem boundaries from `src/kernel/kernel.cpp`.

   Move syscall dispatch/user-copy, ELF loading, initrd lookup, process spawn/exec/wait glue, and observe handlers into separate files. This should come before expanding POSIX or filesystem work.

3. Extract platform drivers from `src/kernel/platform.cpp`.

   Separate ACPI table parsing, PCI enumeration, virtio transport, and virtio-blk code. This should come before adding virtio-net, MSI-X, or persistent storage.

4. Define a minimal `BlockDevice` interface.

   Put virtio-blk behind a small read-only block API first. Keep polling if needed, but remove storage logic from platform probing. This is the prerequisite for filesystem work.

5. Add host tests for core parsers and permission logic.

   Start with CPIO, ELF validation, page-table translation flags, PCI capability walking, and user-copy range checks. This gives faster feedback than QEMU-only tests.

6. Resolve `/bin/init`.

   Either delete `src/user/programs/init.c` or build it deliberately. Today the shell is staged as `/bin/init`, which is fine, but the unused file creates false architecture.

7. Add argv/envp to spawn/exec.

   This should happen before a larger command set or filesystem-backed exec so the ABI does not need immediate redesign.

### Mid-term milestones

1. Implement read-only filesystem-backed exec.

   Choose a simple filesystem and image toolchain. Start read-only. Load programs from the block device through the filesystem rather than only from initrd.

2. Introduce a file descriptor table.

   Add a minimal descriptor model for console and files. This must precede meaningful POSIX growth and SSH-like terminal plumbing.

3. Add interrupt-driven virtio-blk.

   Add MSI/MSI-X support and completion handling. Move from polling smoke reads to request submission and completion.

4. Build local security foundations.

   Add credentials, ownership, permissions, and process identity before remote login. This does not need to be Unix-complete, but it must exist before SSH.

5. Start virtio-net only after device and buffer infrastructure exists.

   Networking should begin with virtio-net probe, RX/TX queue setup, packet buffers, and counters. ARP/ICMP should come before TCP.

6. Design SMP scheduling before enabling AP work.

   Define lock ordering, per-CPU queues or a global run queue policy, scheduler IPIs, timer ownership, and console/input concurrency before letting APs run user tasks.

### Later milestones

1. Grow networking from Ethernet to TCP.

   Add ARP, IPv4, ICMP, UDP, DHCP or static configuration, DNS only if needed, then TCP. Keep observability built in from the start.

2. Add ptys, sessions, login, and SSH.

   SSH should wait for TCP, credentials, filesystem-backed config, entropy/crypto strategy, and pty/session support.

3. Expand POSIX compatibility selectively.

   Add only the calls needed by target userland: process args/env, file descriptors, filesystem metadata, pipes, signals if necessary, and terminal ioctls when ptys exist.

4. Enable real SMP scheduling.

   Bring APs into the scheduler after locks and interrupt routing are ready. Treat this as a kernel-wide concurrency milestone, not a small scheduler patch.

5. Decide on higher-half or direct-map kernel layout.

   The current low identity mapping is good for learning and bootstrapping. It is a liability for long-term hardening and should be revisited before multiuser security is taken seriously.

6. Explore GUI only after terminal/session maturity.

   A framebuffer terminal can evolve into panes or a simple compositor, but a desktop should wait until input, process, filesystem, and scheduling foundations are stable.

7. Approach accelerators as a device-memory and security problem first.

   PCIe discovery is not enough. Accelerator support requires DMA-safe allocation, memory pinning, command submission, completion handling, and access control.

### Explicit dependency ordering

- Before networking: fix user-copy, split platform/device code, add IRQ/MSI infrastructure, add packet-buffer allocation, and add device observability.
- Before SSH: implement networking, credentials, authentication storage, ptys, sessions, filesystem-backed config, and entropy/crypto policy.
- Before GUI: stabilize input events, terminal/session behavior, display-buffer ownership, scheduler behavior, and process control.
- Before SMP expansion: define locks, preemption rules, per-CPU state, scheduler IPIs, interrupt routing, and console/input concurrency.
- Before POSIX growth: add file descriptors, filesystem namespace, argv/envp, process lifecycle rules, and error-code conventions.
- Before accelerator support: define PCI resource ownership, DMA-safe memory, interrupt completion, memory pinning, and user/kernel submission boundaries.

## 7. Immediate Next Steps

1. Fix `CopyToUser` and `CopyFromUser` in `src/kernel/kernel.cpp`.

   This matters now because the current syscall boundary can access mapped supervisor pages if a user process passes kernel addresses. Ring 3 is not enough if syscalls do not enforce user permissions.

2. Add a negative syscall-copy smoke test.

   This matters now because the bug class will recur. A small user program should try to pass kernel-looking addresses to copying syscalls and verify the kernel rejects them without corrupting memory or killing the wrong process.

3. Split syscall and user-copy code out of `src/kernel/kernel.cpp`.

   This matters now because the user/kernel boundary needs to be small and auditable before more syscalls, file descriptors, networking, or SSH are added.

4. Define and implement a minimal read-only `BlockDevice` abstraction.

   This matters now because filesystem work, persistence, and storage tests should not depend on virtio details embedded in `src/kernel/platform.cpp`.

5. Move virtio-blk out of platform discovery.

   This matters now because networking and future PCI drivers will otherwise copy the same pattern and make device ownership harder to repair.

6. Decide the first filesystem and image workflow.

   This matters now because the filesystem choice affects boot images, tests, program loading, file descriptors, and persistence semantics.

7. Resolve the orphan `src/user/programs/init.c`.

   This matters now because the build currently stages the shell as `/bin/init`. That may be the right choice, but the unused `init.c` misleads readers about the runtime path.

8. Add host-side tests for CPIO, ELF validation, PCI capability parsing, and page-table permission checks.

   This matters now because QEMU smokes are too slow and too coarse to carry the next phase alone.

9. Write down the SMP preconditions before scheduling on APs.

   This matters now because APs already boot. The next tempting step is to run work on them, but the lock, interrupt, console, and scheduler rules are not ready.

10. Update `doc/ARCHITECTURE.md` after the user-copy fix.

    This matters now because the docs currently imply a stronger user-pointer validation story than the code provides. After the fix, the architecture doc should state the exact user-copy contract.
