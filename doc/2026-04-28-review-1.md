## 2. Current State From Code

### 2.1 Boot path and platform assumptions

Two frontends. One shared core.

- **BIOS path:** [src/boot/bios/boot.asm](../src/boot/bios/boot.asm) → [src/boot/bios/kernel16.asm](../src/boot/bios/kernel16.asm) → [src/boot/bios/long64.asm](../src/boot/bios/long64.asm) → expand `kernel_bios.elf` at `0x100000` → build `BootInfo` at `0x500` → `kernel_main(BootInfo*, cpu*)`.
- **Limine/UEFI path:** OVMF → BOOTX64.EFI (Limine) → [src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp) walks Limine HHDM, translates Limine-virtual pointers (framebuffer, RSDP, SMBIOS, modules) to physical via page-table walk, loads `kernel_bios.elf` as a module, mirrors the BIOS handoff layout (`0x500` `BootInfo`, `0x6000` memory map, `0x7000` modules, `0x7200` strings), patches a low identity window, and jumps to the same `kernel_main`.
- **Shared contract:** [src/kernel/handoff/boot_info.hpp](../src/kernel/handoff/boot_info.hpp) is `#pragma pack(push, 1)` with offset `static_assert`s; the C++ struct and the NASM `.inc` mirror in [src/kernel/handoff/memory_layout.inc](../src/kernel/handoff/memory_layout.inc) stay aligned by hand.
- **Platform assumption:** the shared kernel is still **low identity-linked** at `0x00100000` ([src/kernel/linker/kernel_bios.ld](../src/kernel/linker/kernel_bios.ld)). The Limine shim only exists to bridge the higher-half ELF Limine wants with the low-linked kernel core.

The architecture cleanly separates "what differs between bootloaders" from "what the kernel actually consumes". The most visible long-term risk in this layer is naming: `kernel_bios.elf` is the shared kernel core, not BIOS-only logic. ARCHITECTURE explains this; the artifact name does not.

### 2.2 Memory management

- **Physical allocator:** [src/kernel/mm/page_frame.cpp](../src/kernel/mm/page_frame.cpp) is a bitmap allocator seeded from `std::span<const BootMemoryRegion>`. It marks all pages busy, frees only `BootMemoryType::Usable`, and reserves the bitmap, low-bootstrap, kernel image, modules, and framebuffer.
- **Page tables:** [src/kernel/mm/virtual_memory.cpp](../src/kernel/mm/virtual_memory.cpp) builds four-level page tables, supports `map_physical`, `allocate_and_map`, `protect`, `translate`, and explicit user-slot teardown (`destroy_user_slot`, see also [src/kernel/proc/user_program.cpp:173-184](../src/kernel/proc/user_program.cpp)). User mappings live under PML4 slot 1 (`kUserPml4Index = 1`); `kUserSpaceBase = 0x0000008000000000`, `kUserStackTop = 0x0000008040000000`.
- **Syscall copy boundary:** [src/kernel/mm/user_copy.cpp](../src/kernel/mm/user_copy.cpp) checks canonicality, range against `[kUserSpaceBase, kUserStackTop)`, overflow, then translates each page and rejects entries that don't have `User` (and `Write` for kernel-to-user). The 2026-04-24 critical bug (`copy_to_user` accepting kernel-only mappings) is genuinely fixed.

Carry-forward issues:
- **`mm` depends on `console`.** [src/kernel/mm/page_frame.hpp:11](../src/kernel/mm/page_frame.hpp) `#include "console/terminal.hpp"`. There is no use of `Terminal` in this header. This is a stale include from the pre-reorganization tree and inverts what should be a unidirectional dependency. Carried forward from 2026-04-27 — still present.
- **Effective page permissions are not validated.** `VirtualMemory::translate()` reports leaf flags only; the CPU's effective permission is the AND of every level. Today, user mappings propagate `User`/`Write` up the tree consistently, but no test or `effective_permissions()` helper enforces it.

### 2.3 Interrupts and exceptions

- IDT, exception stubs, IRQ stubs, and the `int 0x80` syscall gate live in [src/kernel/arch/x86_64/interrupt/interrupt.cpp](../src/kernel/arch/x86_64/interrupt/interrupt.cpp) plus the NASM half in [src/kernel/arch/x86_64/interrupt/inthandler.asm](../src/kernel/arch/x86_64/interrupt/inthandler.asm) and [irqhandler.asm](../src/kernel/arch/x86_64/interrupt/irqhandler.asm).
- [src/kernel/core/trap_dispatch.cpp](../src/kernel/core/trap_dispatch.cpp) classifies a `TrapFrame` into IRQ, syscall, or exception flow; [src/kernel/core/irq_dispatch.cpp](../src/kernel/core/irq_dispatch.cpp) acknowledges legacy IRQs (`lapic_eoi` + `outb 0x20, 0x20` and slave PIC EOI when `irq >= 8`), polls serial RX on every timer tick (`console_input_poll_serial`), bumps `g_timer_ticks`, wakes console readers, and reschedules on timer ticks; [src/kernel/core/fault.cpp](../src/kernel/core/fault.cpp) kills user faults and panics on kernel faults.
- The active periodic tick is the **PIT at 1000 Hz** ([src/kernel/core/kernel_main.cpp:275](../src/kernel/core/kernel_main.cpp) calls `set_timer(1000)`). The LAPIC timer is initialized but masked in [src/kernel/arch/x86_64/apic/lapic.cpp](../src/kernel/arch/x86_64/apic/lapic.cpp).

The hybrid (PIC + IOAPIC + LAPIC) interrupt model is consistent and works in QEMU, but it is not yet a clean substrate for MSI/MSI-X-driven storage or networking.

### 2.4 Scheduling and SMP state

- Scheduling: [src/kernel/sched/scheduler.cpp:8-23](../src/kernel/sched/scheduler.cpp) reaps dead threads, wakes child waiters, marks the current thread ready (when caller asked), then picks `next_runnable_thread` over the fixed `Thread[32]` table. Falls back to `kernel_idle_thread` ([src/kernel/sched/idle.cpp](../src/kernel/sched/idle.cpp)) which `sti; hlt`s.
- Tables: [src/kernel/proc/process.hpp:12](../src/kernel/proc/process.hpp) `kMaxProcesses = 32`; [src/kernel/proc/thread.hpp:18](../src/kernel/proc/thread.hpp) `kMaxThreads = 32`. Allocated from page frames during `init_tasks`.
- AP bring-up: [src/kernel/arch/x86_64/cpu/cpu.cpp:140-174](../src/kernel/arch/x86_64/cpu/cpu.cpp) installs an AP trampoline at `kApTrampolineAddress`, sets per-AP CR3, sends INIT/SIPI through `lapic_start_cpu`, and **busy-waits forever** on `c->booted == 0` with no timeout. APs themselves run `init()` ([cpu.cpp:125-138](../src/kernel/arch/x86_64/cpu/cpu.cpp)) which calls `cpu_init`, `ioapic_init`, `lapic_init`, sets `booted = 1`, then enters `cpu_idle_loop()` — `cli; hlt` forever. APs are off the IDT in any meaningful sense (the AP IDT slot at `kApStartupIdtAddress` is zeroed in `cpu_boot_others`).

This is intentional staging, but the implications are stricter than they look:
- **No AP can ever take an interrupt or IPI.** `cli; hlt` masks INTR, so any later "wake the AP" code must first reroute its idle loop.
- **Kernel-wide globals are single-CPU by assumption.** `page_frames`, `processTable`, `threadTable`, `g_platform`, `g_virtio_blk`, `g_timer_ticks`, `terminal[]`, console input, and driver state have no protection. Promoting AP scheduling will require either a global big lock or per-subsystem locking before any user thread runs on a non-BSP CPU.

### 2.5 User/kernel separation state

- ELF64 `ET_EXEC` only, `PT_LOAD` only, in [src/kernel/proc/user_program.cpp:61-170](../src/kernel/proc/user_program.cpp). Maps segment permissions from ELF flags (`W` → `PageFlags::Write`, no `X` → `PageFlags::NoExecute`), zero-fills `memsz - filesz`, allocates a 64 KiB user stack with a guard page below.
- Syscall ABI: [src/uapi/os1/syscall_numbers.h](../src/uapi/os1/syscall_numbers.h) defines 9 numbers (`write`, `exit`, `yield`, `getpid`, `read`, `observe`, `spawn`, `waitpid`, `exec`). Dispatch is one switch in [src/kernel/syscall/dispatch.cpp](../src/kernel/syscall/dispatch.cpp).
- Initrd-backed `exec`: [src/kernel/syscall/process.cpp](../src/kernel/syscall/process.cpp) loads a new image, switches the current thread's CR3 with `WriteCr3`, destroys the old address space, and the trap return takes the user back to a fresh entry point. The smoke tests reject `shell exec returned` and `echo: after-exec` to mechanically prove the old shell never resumes ([CMakeLists.txt:465-469](../CMakeLists.txt)).
- User programs in initrd: `/bin/init`, `/bin/sh`, `/bin/yield`, `/bin/fault`, `/bin/copycheck`, `/bin/ascii` — all six staged by [src/user/CMakeLists.txt:56-86](../src/user/CMakeLists.txt).

What's still missing on this surface:
- no argv/envp passing (the user stack is `iretq`-shaped only);
- no `fork`, no signals, no descriptor table beyond stdin/stdout/stderr-by-convention (`os1_sys_read` rejects any fd != 0; `os1_sys_write` accepts any fd and routes to the console);
- no current-directory or path namespace beyond the shell's literal `/bin/<name>` resolution in [src/user/programs/sh.cpp:524-553](../src/user/programs/sh.cpp).

Process and scheduling: useful on the BSP, early for SMP. SMP awareness is half-implemented: ACPI-derived topology, AP bring-up, per-CPU `struct cpu`, and TSS RSP0 plumbing are real, but APs are inert and the kernel has no synchronization vocabulary for promoting them.

Storage: the right shape (`BlockDevice` facade), wrong depth (single device, polling reads, no writes, no completion). Storage is the natural next major objective and the dependency root for filesystem-backed `exec`, multiuser config, and SSH host keys.

Accelerator awareness: PCI/PCIe discovery is in place; the prerequisites for accelerators are DMA-safe allocation, resource ownership, MSI-X, and pinning — none of which exist yet. This is fine; the prerequisites are also storage and networking prerequisites.

## 5. Overlooked Goals And Blind Spots

These are areas that are either not in [GOALS.md](../GOALS.md) explicitly or are weaker than their documented intent suggests. Adopt or rebut each.

### 5.1 Architecture and design

- **Pick a side on low-half vs higher-half.** The kernel is low identity-linked at `0x100000`. ARCHITECTURE notes this is a Limine-shim cost. As long as kernel-virtual = physical, every DMA-safe allocation, future user/kernel separation, and security-driven isolation choice is hard. The cost of moving rises with every additional driver.
- **Pick a side on `int 0x80` vs `SYSCALL`/`SYSRET`.** The current ABI is a user-callable interrupt gate ([interrupt.cpp](../src/kernel/arch/x86_64/interrupt/interrupt.cpp) registers vector 48 as DPL=3). `SYSCALL` is materially faster and is the modern convention. The ABI is small enough to migrate cheaply now; it will not stay small.
- **Decide what `kernel_namespaces.hpp` is for.** Either commit to migrating call sites into `os1::kernel::*`, or delete the header. Today it is read by one `#include` and consumed by zero translation units.

### 5.2 Observability and debuggability

- **No structured trace ring.** Observability is a snapshot UAPI, which is the right primitive but not enough on its own. A small kernel ring buffer for "what happened in the last 256 events" — IRQs, context switches, syscall numbers, faults — would be cheap and would make a class of "the smoke flaked once" investigations tractable.
- **No SMP observability.** When APs do anything, per-CPU counters need to exist. Plan the per-CPU counters now while there is one CPU to instrument; building them under contention later is harder.
- **No kernel panic dump.** [src/kernel/core/panic.cpp](../src/kernel/core/panic.cpp) and `halt_forever()` exist but a structured "what was the trap frame, what was the offending RIP, what was CR2/CR3" capture into serial would prevent ad-hoc `debug()` printk grafts.

### 5.3 Testability

- **Host-side unit tests for pure logic.** CPIO parsing, ELF validation, ACPI table walking (XSDT/RSDT length checks, MADT entry-type dispatch, MCFG segment enumeration), PCI capability walk termination, and `is_user_address_range` edges are pure functions of bytes. They belong in a `tests/` directory built without the cross compiler. Today they only get exercised by full QEMU smokes.
- **Property-style tests for `mm/user_copy.cpp`.** Canonicality, range, overflow, page-fault-on-read, write-permission-on-kernel-to-user. These are cheap to encode and they protect the most security-sensitive boundary in the kernel.
- **A smoke for the `idle thread online` path with no user programs.** The current smokes always require a shell. A "kernel boots, never starts /bin/init, lands in idle, accepts no input" smoke would isolate scheduler/idle correctness from userland behavior.

### 5.4 Security and trust boundaries

- **Document trust boundaries explicitly.** ARCHITECTURE describes ring 3 vs ring 0 and where the user-copy boundary is, but does not yet enumerate "what bootloader-supplied data the kernel still dereferences after `own_boot_info`" (answer: framebuffer, RSDP page, initrd modules) or "what fields the kernel should re-validate before trust" (answer: framebuffer pitch/bpp before drawing, RSDP signatures before walking ACPI, initrd entry sizes before allocating).
- **Plan the entropy story now.** It does not need to be implemented. It needs to be allocated to a subsystem so SSH planning is not blocked by *"where does randomness come from?"* later.

### 5.5 Portability boundaries

- **Audit `arch/x86_64` leakage.** A spot check of [src/kernel/core/](../src/kernel/core), [src/kernel/proc/](../src/kernel/proc), and [src/kernel/sched/](../src/kernel/sched) shows clean separation today: `arch/x86_64` is included only when CPU/IDT/segment specifics are needed. Keep this as a build-time invariant by making the non-`arch` directories not allow `arch/` includes by default and listing exceptions explicitly.
- **The `cpu*` argument in `kernel_main` is x86-specific.** The shared kernel entry signature `kernel_main(BootInfo*, cpu*)` already mixes arch-specific state into the portability seam. A future port (AArch64/RISC-V) will want a `BootInfo*` only and an `arch::current_cpu()` accessor.

### 5.6 Developer workflow

- **One-command "fast" path.** `cmake --build build --target smoke_observe` is the smallest meaningful smoke and runs in seconds. Make this the default `make smoke` analogue and document it.
- **`autoformat.sh` is a 4-line script.** [autoformat.sh](../autoformat.sh) is fine, but a `cmake --build build --target format` target would be discoverable.
- **No CONTRIBUTING.md.** Not a goal of a teaching OS, but if external readers are part of the audience (per GOALS), one paragraph on "how to add a syscall, a driver, a smoke" would be high-leverage.

## 6. Roadmap

Order is by dependency, not by attractiveness.

### 6.1 Near term (≤ 1 week of work each)

These are cheap, high-leverage, and unblock later items.

1. **Fix the BIOS image size assertion.** Add `cmake_print_variables`-style or `file(SIZE ...)` checks in [CMakeLists.txt](../CMakeLists.txt) that fail configure/build if `kernel_bios.elf` exceeds `OS1_KERNEL_IMAGE_SECTOR_COUNT × 512` or `initrd.cpio` exceeds `OS1_INITRD_IMAGE_SECTOR_COUNT × 512`. This converts the current silent truncation into a loud build error. Today the kernel ELF (153 728 B) is already past the slot (131 072 B); the trailing section table is being dropped.
2. **Add a build-time/smoke-time initrd manifest check.** Either generate a `manifest.txt` next to `initrd.cpio` and have the smoke assert it against expected paths, or embed an `os1_smoke_initrd_marker` smoke that runs `initrd` in the shell and asserts every staged path is present. This catches drift like `/bin/ascii` missing from observe markers.
3. **Update the "current review" pointer in `README.md`, `GOALS.md`, and `ARCHITECTURE.md`** to whatever is actually current. Better: define one `latest-review.md` symlink/redirect and stop maintaining named pointers.
4. **Update ARCHITECTURE's framebuffer terminal claims** to match shadow-buffer + derived geometry reality. Two edits, one minute.
5. **Add an AP boot timeout with a serial diagnostic.** Replace the unbounded `while (c->booted == 0)` in [cpu.cpp:169-171](../src/kernel/arch/x86_64/cpu/cpu.cpp) with a bounded loop that emits `cpu N failed to come up` on timeout and continues. A failed AP becoming an understood failure is worth more than a failed AP being a silent hang.
6. **Delete or activate `kernel_namespaces.hpp`.** Pick one. If keeping it, port one subsystem (e.g. `mm`) to consume `os1::kernel::mm::*` so the facade earns its existence.
7. **Remove `mm/page_frame.hpp`'s `#include "console/terminal.hpp"`.** It is unused and inverts the dependency direction.
8. **Add an `argv`/`envp` design note for `spawn`/`exec`.** Not the implementation — just the shape: SysV initial stack with `argc`, `argv[]`, `NULL`, `envp[]`, `NULL`, `auxv[]`, `AT_NULL`. Locking that down before the userland surface grows further is much cheaper than retrofitting it later.

### 6.2 Mid term (1–4 weeks of work each)

These build the substrate the next major milestones need.

1. **`BlockDevice` v2: real read path with completion + errors.** Define `BlockRequest`, sector counts, error codes, ownership of buffers. Polling is fine for now; the request lifetime model is what blocks a filesystem.
2. **Lift a shared virtio transport.** Move queue setup, feature negotiation, MMIO BAR mapping, and notification out of `virtio_blk.cpp` into [src/kernel/drivers/virtio/](../src/kernel/drivers/virtio). `virtio-net` will need this; doing it after writing the second driver is more painful.
3. **Move `virtio-blk` to MSI/MSI-X completion.** Polling reads keep the smoke green; they will not survive a filesystem under load. This is the right driver to learn MSI/MSI-X on because it already has clean BAR/queue structure.
4. **First read-only filesystem and filesystem-backed `exec`.** FAT32 (well-documented, easy reproducible image workflow) or a bespoke simple FS. Read-only first. Filesystem-backed loading is the bridge from "demo initrd" to "actual operating system".
5. **Minimal file-descriptor table in `Process`.** Console becomes fd 0/1/2 by default; filesystem opens become fd 3+. This is the inflection point that lets POSIX-shaped code start landing without rewrites.
6. **SMP synchronization contract.** Before any AP runs scheduler work, write down: (a) what locks protect `page_frames`, `processTable`, `threadTable`, `g_platform`, `terminal[]`, `console_input`, `g_virtio_blk`; (b) interrupt-masking rules; (c) lock ordering. A document, not yet code, is enough to discover where the model breaks.
7. **Host-side test harness.** A `tests/` subdirectory built with the host compiler, exercising CPIO parsing, ELF validation, ACPI table walking, PCI capability traversal, and `is_user_address_range` edges. CPIO and ELF code are already written in a way that is host-testable with minor extraction.

### 6.3 Later (significant scope each)

1. **APs in scheduling.** Treat as a concurrency milestone, not a scheduler patch. Run-queue-per-CPU, cross-CPU wakeup IPI, IRQ steering through IOAPIC, per-CPU current-thread accessor, audited globals.
2. **Networking in layers.** `virtio-net` → packet buffers → Ethernet → ARP → IPv4 → ICMP → UDP → TCP. Each is a small, tested layer.
3. **Pty + session model.** Required for SSH and for any "more than one shell". This is where the console-input layer's single-line-buffer assumption stops being enough.
4. **Multiuser foundation.** uid/gid in `Process`, file ownership, permission checks at the VFS, login policy.
5. **SSH.** Only after the substrate exists. SSH = TCP + crypto + entropy + credentials + ptys + sessions + filesystem-backed config + permission checks. Skipping any of those produces an SSH that does not deserve the name.
6. **Higher-half migration or direct-map decision.** Before multiuser/security/DMA isolation become expensive to retrofit.
7. **Framebuffer compositor.** Only after terminal + session maturity.
8. **Accelerator path.** PCI discovery exists; resource ownership, DMA pinning, command submission, completion, and access control come first.

### 6.4 Explicit dependency ordering

What must come **before** networking:
- shared virtio transport;
- DMA-safe physical allocation policy;
- IRQ allocation API and MSI/MSI-X support;
- driver observability (per-driver counters, surfaceable through `observe`);
- packet buffer ownership;
- the `BlockDevice` v2 lessons (because the same lifetime/error questions repeat for NICs).

What must come **before** SSH:
- networking;
- credentials + authentication storage;
- pty + sessions;
- filesystem-backed config (host keys, authorized_keys);
- entropy source;
- crypto policy decision (vendored vs in-tree).

What must come **before** GUI:
- input event abstraction (today input is a single byte stream);
- terminal + session model;
- display-buffer ownership across multiple clients;
- scheduler stability under interactive load;
- richer process control.

What must come **before** SMP expansion:
- locking vocabulary;
- preemption rules;
- per-CPU state ownership;
- scheduler IPIs;
- IRQ steering;
- audited concurrency rules for console/input/allocator/platform/driver state.

What must come **before** broad POSIX-shape userland growth:
- file descriptors in `Process`;
- filesystem namespace;
- argv/envp;
- consistent `errno`-style return convention;
- process-tree / wait semantics beyond the current single-child case.

What must come **before** accelerator work:
- PCI resource ownership (which devices own which BAR ranges);
- DMA-safe memory abstraction;
- MSI/MSI-X completion;
- memory pinning for user submissions;
- a deliberate user/kernel submission boundary.

## 7. Immediate Next Steps

These are the most valuable actions to take *next*. Each names why it matters now.

1. **Make BIOS image overflow a build error.** Today `kernel_bios.elf` (153 728 B) silently truncates to fit a 131 072 B slot in `os1.raw`. The boot still works because PT_LOAD content is in the early part of the ELF, but the safety margin is already gone. A `file(SIZE ...)` assertion in [CMakeLists.txt](../CMakeLists.txt) prevents the next 22 KiB of growth from causing a silent boot failure.

2. **Update README/ARCHITECTURE for `/bin/ascii` and the framebuffer geometry/shadow buffer reality.** Readers form their first model of the system from these documents; small false claims compound. This is a 10-minute fix that the 2026-04-27 review already identified and that has not been done.

3. **Repoint the "current review" link.** [README.md:3](../README.md), [README.md:229](../README.md), [GOALS.md:3](../GOALS.md), and [doc/ARCHITECTURE.md:5](ARCHITECTURE.md) all point at `Review 2026-04-23`. The link is at least three reviews stale and is the most prominent piece of doc surface area for new readers.

4. **Bound the AP boot wait.** The unbounded `while (c->booted == 0)` in [cpu.cpp:169-171](../src/kernel/arch/x86_64/cpu/cpu.cpp) turns AP startup failures into silent hangs. A timeout with a `cpu N failed` serial line is a few lines of code and the highest-leverage diagnostic improvement available.

5. **Decide the `BlockDevice` v2 contract before writing a filesystem.** Sector count, error model, request ownership, completion semantics. Picking these post-hoc forces filesystem code to either break the abstraction or revise it. This is the lever that unlocks Milestone D.

6. **Land host-side tests for `mm/user_copy`, CPIO, and ELF.** The user-copy boundary is the most security-sensitive surface in the kernel and currently has no fast feedback path. CPIO + ELF are simple parsers that can be tested in seconds on the host and are the foundation of `exec`.

7. **Draft the SMP synchronization contract.** Before any line of AP scheduler code lands, document: which locks protect which globals, what the lock order is, what runs with interrupts disabled. A document preempts a class of subtle bugs; the alternative is debugging contention through serial logs.

8. **Delete `kernel_namespaces.hpp` or commit to it.** It has zero call sites today. Keeping unused migration scaffolding adds noise and tells the wrong story about how kernel C++ should be organized.

9. **Plan `argv`/`envp` for `spawn`/`exec` now.** The user stack is `iretq`-shaped only ([user_program.cpp:166-168](../src/kernel/proc/user_program.cpp)). Locking a SysV initial-stack layout in before more programs ship is materially cheaper than a retrofit.

10. **Add an initrd manifest smoke or check.** The packaged initrd already drifts ahead of observe markers. Catching the drift mechanically scales as user programs grow.

---

End of review. The next major engineering objective should be **storage and synchronization substrate**, not another boot/shell refactor. The system is in a good shape to receive that work.
