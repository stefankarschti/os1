## 1. Executive Summary

### Current project state

`os1` is now a coherent x86_64 teaching OS with a real end-to-end path from firmware to protected user programs. The current implementation has:

- Two boot frontends: a legacy BIOS/raw-image path in [src/boot/bios](../src/boot/bios) and a Limine/UEFI path in [src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp).
- A shared low-half kernel entry in [src/kernel/core/kernel_main.cpp](../src/kernel/core/kernel_main.cpp).
- A normalized handoff ABI in [src/kernel/handoff/boot_info.hpp](../src/kernel/handoff/boot_info.hpp).
- ACPI, LAPIC, IOAPIC, PCI ECAM discovery, and legacy ISA IRQ routing under [src/kernel/arch/x86_64/platform](../src/kernel/arch/x86_64/platform).
- Physical and virtual memory management, user address-space creation, user-copy validation, and ELF loading under [src/kernel/mm](../src/kernel/mm), [src/kernel/process](../src/kernel/process), and [src/kernel/syscall](../src/kernel/syscall).
- Ring 3 user programs loaded from an initrd CPIO archive, including `/bin/init`, `/bin/sh`, `/bin/yield`, `/bin/fault`, `/bin/copycheck`, and `/bin/ascii`.
- A small syscall ABI in [src/uapi/os1/syscall_numbers.h](../src/uapi/os1/syscall_numbers.h), including read, write, spawn, waitpid, exec, and observe.
- A text terminal stack with VGA and framebuffer backends in [src/kernel/console](../src/kernel/console), [src/kernel/display](../src/kernel/display), and [src/kernel/drivers/fb](../src/kernel/drivers/fb).
- QEMU-oriented CMake targets and serial smoke tests in [src/kernel/CMakeLists.txt](../src/kernel/CMakeLists.txt) and [cmake/scripts/run_smoke.py](../cmake/scripts/run_smoke.py).

This is no longer just a boot experiment. It is a small kernel substrate with a working userland vertical slice.

### Strongest parts

The strongest architectural work is the normalized boot handoff. Both BIOS and Limine paths converge on the same `BootInfo` contract before entering the kernel, which keeps [src/kernel/core/kernel_main.cpp](../src/kernel/core/kernel_main.cpp) mostly firmware-agnostic. The current CMake boot contract check in [cmake/scripts/assert_kernel_boot_contract.py](../cmake/scripts/assert_kernel_boot_contract.py) also turns the low-memory image envelope into an enforced build property rather than a comment.

The second strong part is the protected-userland path. The project has a real ring transition, page-table isolation for user mappings, syscall dispatch through `int 0x80`, validated user copies, process/thread tables, ELF loading, exec, spawn, waitpid, and a shell. That is the correct base to build from before adding bigger subsystems.

The third strong part is debug ergonomics. Serial output, smoke markers, `observe` records, framebuffer/VGA terminal output, and QEMU run targets make the system inspectable enough to iterate on without a hardware lab.

### Biggest risks / gaps

The biggest risk is that the next major goals depend on infrastructure that is still mostly absent. Persistence, networking, SSH, GUI, POSIX growth, SMP expansion, and accelerator support all need lower-level contracts that do not exist yet.

The most important gaps are:

- Storage is still a smoke-tested device path, not a kernel storage subsystem. [src/kernel/drivers/virtio/virtio_blk.cpp](../src/kernel/drivers/virtio/virtio_blk.cpp) is polling-only and read-only, while [src/kernel/storage/block_device.hpp](../src/kernel/storage/block_device.hpp) is only a tiny facade.
- There is no filesystem or VFS. [src/kernel/vfs/README.md](../src/kernel/vfs/README.md) is a placeholder; execution still comes from the initrd CPIO path.
- SMP discovery and AP boot exist, but runtime scheduling is effectively BSP-only. APs are brought up and then parked in the idle loop in [src/kernel/arch/x86_64/cpu/cpu.cpp](../src/kernel/arch/x86_64/cpu/cpu.cpp).
- Networking and security are not started. [src/kernel/drivers/net/README.md](../src/kernel/drivers/net/README.md) and [src/kernel/security/README.md](../src/kernel/security/README.md) are placeholders.
- There are no host-side tests for parsers, allocators, page-table behavior, ELF loading, user-copy edge cases, ACPI/PCI parsing, or syscall state transitions.
- Several core kernel tables are fixed-size and global. They are acceptable for the current teaching slice but will become blockers for SMP, file descriptors, networking, and multiuser state.

## 2. Current State From Code

### Boot path and platform assumptions

The default path is UEFI/Limine. [limine.conf](../limine.conf) configures serial output, a framebuffer mode, the Limine kernel executable, and two modules: `kernel.elf` and `initrd.cpio`. The Limine shim in [src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp) is substantial: it reads Limine requests, translates Limine virtual addresses through the active page tables using HHDM, locates modules, loads low `PT_LOAD` segments from `kernel.elf`, builds a low `BootInfo`, ensures the low identity map, and calls the shared kernel entry.

The BIOS path remains a first-class compatibility path. [src/boot/bios/boot.asm](../src/boot/bios/boot.asm) loads the second-stage loader, [src/boot/bios/kernel16.asm](../src/boot/bios/kernel16.asm) handles A20, long-mode detection, E820, RSDP discovery, EDD disk reads, ELF expansion, and handoff construction, while [src/boot/bios/long64.asm](../src/boot/bios/long64.asm) builds temporary page tables and enters long mode.

The shared kernel starts in [src/kernel/core/kernel_main.cpp](../src/kernel/core/kernel_main.cpp). It copies the handoff data with `own_boot_info`, binds the initrd, initializes CPU state, initializes the physical allocator, reserves boot-critical regions, builds virtual memory mappings, activates paging, initializes platform state, initializes interrupt controllers, boots APs, selects display output, initializes terminal state, registers exception handlers, initializes input, creates the first kernel process/thread, loads `/bin/init`, arms the PIT, and starts the scheduler.

Platform assumptions are explicit and narrow:

- Target architecture is x86_64.
- QEMU/q35 is the real test platform.
- Both boot paths require ACPI tables for CPU/APIC topology and PCI discovery. [src/kernel/platform/init.cpp](../src/kernel/platform/init.cpp) now rejects missing ACPI data instead of falling back to legacy MP-table parsing.
- PCI discovery is through MCFG/ECAM in [src/kernel/arch/x86_64/platform/pci.cpp](../src/kernel/arch/x86_64/platform/pci.cpp).
- The kernel still relies on a low identity-linked physical layout defined by [src/kernel/handoff/memory_layout.h](../src/kernel/handoff/memory_layout.h) and [src/kernel/handoff/memory_layout.inc](../src/kernel/handoff/memory_layout.inc).

The boot ABI is compact and intentionally bounded. [src/kernel/handoff/boot_info.hpp](../src/kernel/handoff/boot_info.hpp) fixes the maximum memory map entries at 128 and modules at 16. That is fine for QEMU and the current project stage, but it is a real portability boundary for richer UEFI maps, more modules, or more device metadata.

The earlier BIOS image-size issue from prior reviews is no longer current. [src/kernel/CMakeLists.txt](../src/kernel/CMakeLists.txt) now derives the kernel image sector count from the reserved physical window and invokes [cmake/scripts/assert_kernel_boot_contract.py](../cmake/scripts/assert_kernel_boot_contract.py) before producing the raw image.

### Memory management

Physical memory management is bitmap-based. [src/kernel/mm/page_frame.cpp](../src/kernel/mm/page_frame.cpp) initializes a `PageFrameContainer`, frees usable E820/Limine memory ranges, and reserves the allocator bitmap, low boot regions, kernel reserved region, modules, and framebuffer. This is simple and appropriate for the current scale.

Virtual memory is handled in [src/kernel/mm/virtual_memory.cpp](../src/kernel/mm/virtual_memory.cpp). The kernel builds an identity map of physical memory, maps modules, maps the framebuffer, maps RSDP, and allocates per-process user mappings under the user PML4 slot defined in [src/kernel/handoff/memory_layout.h](../src/kernel/handoff/memory_layout.h). The address layout uses:

- user base: `0x0000008000000000`
- user image base: `0x0000008000400000`
- user stack top: `0x0000008040000000`
- user stack pages: 16
- kernel stack pages: 4

User memory access is mediated by [src/kernel/mm/user_copy.cpp](../src/kernel/mm/user_copy.cpp). It checks canonical addresses, detects range overflow, constrains accesses to the user region, walks page translations, and checks user/write permissions before copying.

The main memory-management risks are architectural rather than immediate:

- The kernel still assumes a low identity-linked world where physical addresses and useful kernel virtual addresses are closely coupled.
- `BootInfo`, modules, and framebuffer mappings are manually carried across the early boot boundary.
- `VirtualMemory::translate` reports leaf flags; it does not model effective permissions accumulated across intermediate page-table levels. The current mappings tend to propagate the expected user bits, but the invariant is not tested.
- There are no host-side tests for allocator edge cases, page-table cloning/destruction, guard pages, or user-copy boundary failures.

### Interrupts / exceptions

Interrupt and exception setup is real and cohesive. [src/kernel/arch/x86_64/interrupt/interrupt.cpp](../src/kernel/arch/x86_64/interrupt/interrupt.cpp) builds the IDT, remaps the PIC, registers IRQ hooks, and installs vector `0x80` as a DPL 3 syscall gate. Assembly stubs in [src/kernel/arch/x86_64/interrupt/irqhandler.asm](../src/kernel/arch/x86_64/interrupt/irqhandler.asm) and [src/kernel/arch/x86_64/interrupt/inthandler.asm](../src/kernel/arch/x86_64/interrupt/inthandler.asm) normalize trap entry and save state into either the current thread frame or the per-CPU interrupt frame.

[src/kernel/arch/x86_64/interrupt/trap.cpp](../src/kernel/arch/x86_64/interrupt/trap.cpp) dispatches IRQs, syscalls, and exceptions. [src/kernel/arch/x86_64/interrupt/fault.cpp](../src/kernel/arch/x86_64/interrupt/fault.cpp) makes an important distinction: user faults mark the current thread/process for termination and reschedule, while kernel faults dump state and halt.

The timer path in [src/kernel/arch/x86_64/interrupt/irq_dispatch.cpp](../src/kernel/arch/x86_64/interrupt/irq_dispatch.cpp) increments ticks, polls serial input, wakes console readers, and calls the scheduler. Keyboard IRQs feed console input. LAPIC/PIC EOIs are handled explicitly.

The design is good for a uniprocessor teaching kernel. It is not yet a safe SMP interrupt architecture because scheduling, blocking, and global state are not protected by a lock/preemption model.

### Scheduling / SMP state

Processes and threads are fixed-table objects under [src/kernel/process](../src/kernel/process). The code currently uses small bounded tables for processes and threads, creates a kernel idle thread, creates a first user process for `/bin/init`, and then runs a simple round-robin scheduler.

The scheduler is intentionally minimal. It is enough to demonstrate yield, blocking reads, process exit, waitpid, exec, and user-fault recovery. It is not yet a scalable process model.

SMP state is discovered and APs are booted, but APs do not participate in normal runtime work. [src/kernel/arch/x86_64/cpu/cpu.cpp](../src/kernel/arch/x86_64/cpu/cpu.cpp) initializes per-CPU structures and starts application processors; APs mark themselves booted and then enter a `cli; hlt` idle loop. The current `wait_for_ap_boot` is bounded, so the earlier "infinite AP wait" concern is resolved, but the fundamental SMP gap remains.

Before AP scheduling is expanded, the project needs:

- A lock vocabulary and lock-ordering rule.
- A preemption/interrupt-disable discipline.
- Per-CPU run queues or an explicit global scheduler lock.
- Audits of process, thread, console, allocator, terminal, PCI, block-device, and observe globals.
- Cross-CPU wakeup/IPI design.

### User/kernel separation state

User/kernel separation is one of the best-implemented vertical slices. [src/kernel/process/elf_loader.cpp](../src/kernel/process/elf_loader.cpp) loads static ELF64 executables from initrd, maps `PT_LOAD` segments with permissions derived from ELF flags, creates a user stack, and transfers to ring 3. [src/kernel/process/init.cpp](../src/kernel/process/init.cpp) starts `/bin/init`, and `/bin/init` then execs `/bin/sh`.

The syscall ABI is small and concrete. [src/uapi/os1/syscall_numbers.h](../src/uapi/os1/syscall_numbers.h) defines write, exit, yield, getpid, read, observe, spawn, waitpid, and exec. Kernel-side dispatch is under [src/kernel/syscall](../src/kernel/syscall); user-side wrappers live under [src/user/lib](../src/user/lib).

Implemented behavior includes:

- `write` to stdout/stderr.
- canonical line `read` from stdin.
- process spawn by initrd path.
- waitpid over child process state.
- in-place exec.
- observe records for system, process, CPU, PCI, and initrd information.
- user fault termination without killing the kernel.

Missing pieces are equally clear:

- No file descriptors beyond stdin/stdout/stderr conventions.
- No argv/envp initial stack.
- No fork.
- No signals.
- No process credentials.
- No namespaces.
- No filesystem-backed exec.
- No errno or POSIX-style error surface.
- `Thread::wait_address` and `Thread::wait_length` are generic fields overloaded for different wait reasons; `ChildExit` stores a pid/status payload in fields whose names describe memory waits. This is workable but brittle.

### Storage / filesystem state

Storage exists only as an early driver experiment. [src/kernel/drivers/virtio/virtio_blk.cpp](../src/kernel/drivers/virtio/virtio_blk.cpp) detects modern virtio-blk PCI devices, maps vendor-specific capabilities, negotiates `VERSION_1`, sets up one virtqueue, and performs polling sector reads. It validates known test-disk prefixes during smoke runs.

The block facade in [src/kernel/storage/block_device.hpp](../src/kernel/storage/block_device.hpp) exposes device name, sector size/count, driver state, and read/write callbacks. The write callback path currently returns failure for virtio-blk.

There is no filesystem implementation. [src/kernel/vfs/README.md](../src/kernel/vfs/README.md) is a placeholder. The only "filesystem-like" source of executable files is the initrd CPIO reader in [src/kernel/initrd](../src/kernel/initrd).

This makes persistence the next major dependency bottleneck. A filesystem should not be built directly on the current polling virtio-blk path without first defining request lifetime, error reporting, alignment, DMA ownership, completion, and eventually interrupt delivery.

### Drivers / device model

Device support is pragmatic and QEMU-focused:

- VGA text output in [src/kernel/display/vga.cpp](../src/kernel/display/vga.cpp).
- Linear framebuffer output in [src/kernel/drivers/fb](../src/kernel/drivers/fb).
- PS/2 keyboard input in [src/kernel/drivers/keyboard.cpp](../src/kernel/drivers/keyboard.cpp).
- Serial console input/output in [src/kernel/drivers/serial.cpp](../src/kernel/drivers/serial.cpp).
- PIT timer in [src/kernel/arch/x86_64/timer/pit.cpp](../src/kernel/arch/x86_64/timer/pit.cpp).
- LAPIC/IOAPIC support in [src/kernel/arch/x86_64/platform](../src/kernel/arch/x86_64/platform).
- PCI ECAM enumeration in [src/kernel/arch/x86_64/platform/pci.cpp](../src/kernel/arch/x86_64/platform/pci.cpp).
- virtio-blk in [src/kernel/drivers/virtio/virtio_blk.cpp](../src/kernel/drivers/virtio/virtio_blk.cpp).

There is not yet a real device model. There is no unified resource allocator for MMIO ranges, IRQ vectors, MSI/MSI-X tables, DMA-safe memory, driver binding, or hotplug. That is acceptable now, but it must be addressed before networking, accelerator support, or nontrivial storage.

### Networking state

Networking is not implemented. [src/kernel/drivers/net/README.md](../src/kernel/drivers/net/README.md) is only a placeholder. There is no NIC driver, Ethernet layer, ARP, IP, UDP, TCP, socket API, packet allocator, network observability, or userspace network process.

The correct prerequisite is not "start TCP." The prerequisite is a working interrupt-capable virtio transport, DMA-safe buffer ownership, driver stats, and a kernel event/wakeup model that can support asynchronous packet arrival.

### Security / multiuser state

Security is not implemented beyond ring separation and user-copy validation. [src/kernel/security/README.md](../src/kernel/security/README.md) is a placeholder.

Current code has no:

- user IDs or group IDs.
- credentials on processes.
- permission checks.
- path or filesystem access policy.
- login/session model.
- entropy source.
- cryptographic library.
- isolation beyond address-space separation.
- audit/logging model for security-relevant events.

That is not a criticism of the current milestone. It means SSH and multiuser goals are gated by several missing layers, not just by network code.

### GUI / terminal state

The implemented UI is a terminal-first stack. [src/kernel/console/terminal.cpp](../src/kernel/console/terminal.cpp) maintains multiple virtual terminals, supports switching, scrollback-like buffering behavior, and output rendering through either VGA or framebuffer backends. [src/kernel/drivers/fb/framebuffer.cpp](../src/kernel/drivers/fb/framebuffer.cpp) renders an 8x16 font into RGB/BGR framebuffers and uses dirty tracking.

Console input is shared between keyboard and serial input through [src/kernel/console/input.cpp](../src/kernel/console/input.cpp). Canonical line input wakes blocked readers through the scheduler.

There is no GUI, compositor, window manager, mouse, graphics API, or userspace terminal emulator. That is consistent with [GOALS.md](GOALS.md), which makes GUI optional and late.

### Build, tooling, and debug workflow

The build is CMake/Ninja-oriented and QEMU-first. [src/kernel/CMakeLists.txt](../src/kernel/CMakeLists.txt) builds BIOS and Limine artifacts, generates the initrd, creates a FAT EFI image, assembles a raw BIOS image, and defines run and smoke targets.

The debug workflow is stronger than many teaching kernels at this stage:

- serial logs are first-class.
- QEMU smoke tests use marker-based assertions.
- `observe` exposes structured kernel state to userland.
- the initrd includes small behavioral test programs.
- the BIOS boot contract now has an explicit assertion script.

The main gaps are:

- no host unit tests.
- no parser fuzzing or table-driven tests for CPIO, ELF, ACPI, PCI, or page tables.
- no CI description in the repo.
- no enforced formatting target, though [autoformat.sh](../autoformat.sh) exists.
- smoke coverage does not yet assert that every staged initrd binary is observable or executable; `/bin/ascii` is present but not covered by the current observe smoke markers.

## 4. Review of Progress Against Goals

This section uses [GOALS.md](GOALS.md) as the target-state reference.

### Goal A: Clean boot baseline

Completion level: high.

The project boots through BIOS and Limine/UEFI, reaches long mode, initializes the kernel, creates a terminal, starts userland, and runs smoke tests. The boot contract is now checked by CMake, and the shared `BootInfo` path is a strong foundation.

Remaining blockers:

- The low identity-linked layout is still a long-term constraint.
- The Limine shim is too large and owns too many responsibilities.
- `BootInfo` has fixed entry limits that may become restrictive on more realistic UEFI systems.

### Goal B: Modern platform handoff

Completion level: medium to high for QEMU/q35, low for broader hardware.

ACPI MADT/MCFG parsing, LAPIC, IOAPIC, interrupt overrides, PCI ECAM enumeration, and virtio-blk probing are implemented. That is enough for the stated QEMU-first platform.

Remaining blockers:

- No MSI/MSI-X support.
- No PCI resource ownership model.
- No generic IRQ allocation/binding model.
- No DMA-safe allocation and mapping story.
- No HPET/APIC timer migration.
- No USB/xHCI input path.

### Goal C: Protected userland and operator shell

Completion level: medium to high.

Ring 3 programs, syscall dispatch, user-copy validation, initrd ELF loading, shell, spawn/wait/exec, and observe records are implemented. This is the most complete goal after boot.

Remaining blockers:

- No argv/envp.
- No file-descriptor table.
- No filesystem-backed exec.
- No process credentials.
- No fork/signals.
- No typed wait state.
- No host tests for ELF, exec teardown, user-copy failure, or process lifecycle edge cases.

### Goal D: Persistence and local security

Completion level: low.

There is a polling read-only virtio-blk path and a placeholder VFS directory. There is no persistent filesystem, no block cache, no write path, no credentials, and no permissions.

Sequencing issue:

Persistence should not start with a full filesystem. It should start with a better block I/O contract, async or interrupt-capable completion, errors, stats, a minimal read-only filesystem target, and then writes.

### Goal E: Networking

Completion level: not started.

There is no network driver or stack. The net directory is a placeholder.

Sequencing issue:

Networking should wait until the virtio transport, interrupt model, DMA ownership, packet allocation, and wakeup/event path are cleaner. Otherwise the network stack will bake in polling assumptions from the current virtio-blk experiment.

### Goal F: SSH / remote operation

Completion level: not started.

SSH requires networking, entropy, crypto, credentials, sessions, pseudo-terminals, filesystem-backed configuration/keys, and robust process I/O. None of those layers exist yet.

Sequencing issue:

SSH should come after a local login/session model and after a minimal TCP stack is stable. It should not be the first consumer of security or terminal-session abstractions.

### Goal G: Optional desktop

Completion level: not started, correctly deferred.

Framebuffer text output is implemented, but there is no GUI. That matches the project goals: GUI should be late and optional.

Sequencing issue:

GUI should come after process I/O, input events, framebuffer ownership rules, and scheduler stability. A compositor before a file/session/process model would add surface area without improving the core OS.

### Cross-cutting goal: SMP shaping early architecture

Completion level: low to medium.

The code discovers CPUs and boots APs. That is useful. Runtime execution is still uniprocessor, and shared kernel state is not prepared for AP scheduling.

Sequencing issue:

Do not enable AP scheduling as a demonstration before defining synchronization and preemption rules. That would convert hidden uniprocessor assumptions into intermittent bugs.

### Cross-cutting goal: Accelerator awareness

Completion level: low.

PCI enumeration is the only real prerequisite currently present. There is no DMA mapping, IOMMU story, MSI-X, user/kernel submission model, memory pinning, or device isolation.

Sequencing issue:

Accelerator support should wait until the PCI resource model, DMA allocation, interrupt delivery, and user buffer ownership are explicitly designed.

## 5. Overlooked Goals / Blind Spots

### Architecture boundaries

The project needs a clearer boundary between platform discovery, resource ownership, and driver binding. Right now, platform code can enumerate PCI, and drivers can probe devices, but there is no central model for "this device owns this BAR, this IRQ, this DMA memory, and this lifecycle." That will matter for virtio-net, MSI-X, block queues, and accelerators.

The Limine shim should also be split. [src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp) currently combines pointer translation, ELF loading, module discovery, framebuffer handoff, page-table patching, and `BootInfo` construction. It works, but it is too dense for long-term maintenance.

### Observability

`observe` is a good start, but the kernel needs a small event ring or structured trace buffer before storage/network/SMP work. Useful event classes would include traps, process lifecycle, scheduler transitions, block requests, IRQ routing, PCI devices, and user-copy failures. Serial logs are not enough once asynchronous devices arrive.

### Testability

The repo leans heavily on QEMU smoke tests. That catches integration regressions, but it is slow and too coarse for parsers and boundary logic.

Host-side tests should cover:

- CPIO parsing.
- ELF loader validation.
- user-copy range and permission failures.
- page-table map/protect/translate/destroy behavior.
- ACPI MADT/MCFG parsing with recorded fixtures.
- PCI capability walking.
- process wait/exit/exec state transitions.
- block-device request contract once redesigned.

### Security principles

The project has ring separation but no security model. Before adding persistence or SSH, define explicit non-goals and first principles:

- Is this single-user forever, or will credentials exist?
- Are user processes mutually distrusting?
- Is filesystem permission checking in scope?
- Are kernel modules out of scope?
- What is the policy for user pointers, DMA buffers, and device access?

These choices should be made before SSH, not during SSH.

### Portability boundaries

QEMU-first is a good non-goal boundary, but the code should label assumptions as such. Examples: q35/MCFG expectations, fixed low memory windows, small `BootInfo` tables, PS/2 keyboard, PIT timer, and Limine framebuffer formats. Treat these as explicit portability limits, not accidental defaults.

### Developer workflow

The project would benefit from a small "verification ladder":

1. host unit tests for pure logic.
2. fast serial-only QEMU smoke.
3. full UEFI and BIOS smoke.
4. optional graphical framebuffer smoke.
5. manual debug run targets.

Right now, the smoke tests are useful, but too much correctness is only validated inside a complete boot.

### Long-term maintainability

Fixed global tables are fine at this stage, but the code needs comments or docs stating which fixed limits are intentional teaching simplifications and which are temporary. Process count, thread count, module count, memory-map count, terminal count, and queue sizes should eventually be collected into a limits document or observe record.

## 6. Roadmap

The roadmap below is ordered by dependency rather than by feature appeal.

### Near-term milestones

1. Repair documentation pointers and stale references.

   Remove `kernel_namespaces.hpp` references from [doc/ARCHITECTURE.md](ARCHITECTURE.md). This matters because the docs are now mostly accurate; the remaining broken pointers are avoidable confusion.

2. Add host-side tests for pure kernel logic.

   Start with CPIO, ELF validation, user-copy ranges, page-table permission behavior, and ACPI/PCI parser fixtures. This should happen before more device work, because new subsystems will otherwise be validated only through full QEMU boots.

3. Tighten initrd and smoke coverage.

   Generate or expose an initrd manifest, assert that staged binaries appear in `observe initrd`, and include `/bin/ascii` in smoke coverage. This prevents docs/build/userland drift.

4. Define a block-device v2 contract.

   Specify request objects, completion status, sector alignment, read/write semantics, ownership, timeout/error behavior, and statistics before building a filesystem.

5. Split common virtio transport from virtio-blk.

   Capability walking, feature negotiation, queue setup, notification, DMA allocation, and interrupt completion should not be duplicated in virtio-net later.

6. Define synchronization and preemption rules.

   Write down when interrupts may be disabled, what can sleep, what locks exist, and what code is BSP-only. This should precede AP scheduling and interrupt-driven storage/networking.

7. Improve process ABI basics.

   Add argv/envp stack setup and a small file-descriptor table before expanding shell behavior or POSIX-like APIs.

### Mid-term milestones

1. Implement MSI/MSI-X and IRQ resource allocation.

   This should come before networking and before high-throughput storage. Polling drivers should not set the model for the rest of the kernel.

2. Make virtio-blk interrupt-capable, then add writes.

   Move from polling smoke reads to real request completion. Add write support only after error handling and request ownership are explicit.

3. Add a minimal read-only filesystem.

   A simple FAT or custom read-only filesystem is enough to move exec and shell listing away from initrd-only semantics. Writes can come later.

4. Add file descriptors and filesystem-backed exec.

   This is the bridge from the current userland demo to a usable terminal OS.

5. Add a structured kernel event ring.

   Expose event records through `observe` or a dedicated syscall. Use it for traps, process lifecycle, IRQs, block I/O, and scheduler decisions.

6. Start AP scheduling only after lock rules exist.

   Bring up one additional runnable AP with a restricted workload, then expand. Do not flip all APs into the global scheduler at once.

### Later milestones

1. Networking.

   Implement virtio-net after common virtio transport, MSI/MSI-X, DMA buffer ownership, packet queues, and event observability are in place. Then add Ethernet, ARP, IPv4, UDP, and only then TCP.

2. SSH.

   Build this after TCP, entropy, crypto, credentials, sessions, pseudo-terminals, and persistent key/config storage exist.

3. GUI.

   Defer until process I/O, input events, framebuffer ownership, and scheduling are stable. The current framebuffer text path is enough for the project stage.

4. POSIX growth.

   Expand only after the basic object model exists: file descriptors, processes, paths, errors, argv/envp, and wait semantics. Avoid adding isolated POSIX calls without the underlying model.

5. Accelerator support.

   Defer until PCI ownership, DMA pinning, MSI-X, memory mapping policy, and user/kernel submission queues are designed. Accelerator support should not be a special-case driver bolted onto the current PCI scan.

### Explicit prerequisites for major goals

Before networking:

- shared virtio transport.
- MSI/MSI-X.
- DMA-safe memory ownership.
- IRQ allocation/binding.
- packet buffer lifecycle.
- driver statistics and event logging.

Before SSH:

- networking through TCP.
- entropy and crypto.
- credentials.
- pseudo-terminals and sessions.
- filesystem-backed configuration and keys.
- reliable process I/O.

Before GUI:

- input event abstraction.
- framebuffer ownership rules.
- process/session model.
- scheduler stability.
- memory ownership for graphical buffers.

Before SMP expansion:

- lock and preemption rules.
- audited globals.
- per-CPU scheduler state or a global scheduler lock.
- IPI/wakeup path.
- AP-safe console and allocator behavior.

Before POSIX growth:

- file-descriptor table.
- path/filesystem namespace.
- argv/envp.
- errno-style error contract.
- process lifecycle semantics.

Before accelerator support:

- PCI resource ownership.
- DMA allocation and pinning.
- MSI-X support.
- user/kernel buffer ownership rules.
- device fault and reset handling.

## 7. Immediate Next Steps

1. Fix the remaining documentation drift.

   Update [doc/latest-review.md](latest-review.md) and [doc/ARCHITECTURE.md](ARCHITECTURE.md). This matters now because most docs are close to accurate; leaving broken links and nonexistent file references makes future reviews harder and can send contributors toward stale code.

2. Add host tests for CPIO, ELF, user-copy, and page-table permission behavior.

   This matters now because the next subsystems will multiply state transitions. Full QEMU smoke tests are too coarse to catch boundary bugs cheaply.

3. Define the next block-device contract before adding a filesystem.

   This matters now because persistence is the next major goal, and the current virtio-blk path is a polling read-only experiment. A filesystem built directly on it would inherit the wrong assumptions.

4. Split common virtio mechanics out of virtio-blk.

   This matters now because virtio-net is a likely future NIC path. Capability parsing, queue setup, notify calculation, feature negotiation, and completion handling should be shared before a second virtio driver exists.

5. Add initrd manifest coverage and smoke `/bin/ascii`.

   This matters now because docs, build staging, and smoke assertions have already drifted once. A manifest-style check prevents silent userland omissions.

6. Write the SMP synchronization contract while APs are still parked.

   This matters now because the code can still be audited as a mostly uniprocessor kernel. Once APs run real scheduler work, missing lock rules will become harder to isolate.

7. Add argv/envp and a small file-descriptor table.

   This matters now because it is the next natural user/kernel ABI step after spawn, waitpid, and exec. It also prepares for filesystem-backed exec and better shell behavior.

8. Add an event ring or structured trace buffer.

   This matters now because storage, networking, and SMP all need post-boot observability. Serial text logs are not enough for asynchronous subsystem debugging.

9. Split [src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp) into smaller boot-shim units.

   This matters now because the shim is already doing HHDM translation, ELF loading, module discovery, framebuffer capture, page-table work, and handoff construction. Waiting will make the split riskier.

10. Replace generic wait payload fields with typed wait state.

    This matters now because process waiting is still small. Fixing the representation before more wait reasons arrive will keep scheduler and syscall code readable.
