## 1. Executive Summary

`os1` is a freestanding `x86_64` teaching/engineering OS. The project has visibly crossed the "coherent vertical slice" threshold:

- Dual-entry boot (legacy BIOS raw image + default Limine/UEFI ISO) converges on one kernel ABI (`BootInfo`) and one shared kernel core (`kernel_bios.elf`).
- The kernel boots on ACPI-derived topology (MADT/MCFG/XSDT), enumerates PCIe through ECAM, probes a modern `virtio-blk` PCI device, and validates raw-sector reads as part of every boot â€” on **both** boot paths.
- The system boots directly into a ring-3 shell staged as both `/bin/init` and `/bin/sh` from a `cpio newc` initrd. User programs are protected, memory-isolated, and fault-contained.
- A small but real syscall ABI (`int 0x80`, 9 calls) plus a versioned read-only observability UAPI (`os1/observe.h`, 5 record kinds) drive a scriptable shell with `sys`, `ps`, `cpu`, `pci`, `initrd`, `spawn`-backed command execution, and in-place `exec`.
- CI runs an eight-test CTest matrix (baseline/observe/spawn/exec Ã— UEFI/BIOS) on every push/PR and is mirrored for local `act`.

What's intentionally missing is now a narrow, well-framed list: block-layer abstraction above the virtio-blk smoke, filesystem-backed loading, networking (`virtio-net`), multiuser/security, MSI/MSI-X and fast-path syscalls, and SMP user-process scheduling. **Overall assessment:** healthy. The project has high signal-to-noise, strong internal consistency, and an explicit "clarity over cleverness" stance that holds up under scrutiny of the code.

## 3. What Exists Today (ground truth)

### 3.1 Boot paths

Two frontends, one kernel ABI:

- **Legacy BIOS raw image** â€” `boot.bin` (MBR) â†’ `kernel16.bin` (real-mode loader: A20, long-mode check, E820, EDD reads, RSDP scan, LME/NXE, long mode, ELF expand) â†’ `kernel_bios.elf` at `0x00100000`. Fixed LBA layout is generated at configure time via `cmake/templates/image_layout.inc.in`.
- **Default Limine/UEFI ISO** â€” OVMF â†’ Limine (`EFI/BOOT/BOOTX64.EFI`) â†’ `kernel_limine.elf` (higher-half shim). The shim [requests HHDM, memmap, framebuffer, bootloader info, cmdline, modules, RSDP, SMBIOS](../src/boot/limine/entry.cpp), parses the shared kernel module as ELF64, writes `PT_LOAD` segments through HHDM (not via identity), translates every Limine-virtual pointer to physical via `TranslateLimineVirtual()` / CR3 walk, installs a minimum low-identity window (PML4[0]â†’PML3[0]â†’PML2 huge pages), and finally jumps to `KernelMain(BootInfo*, cpu*)` on a low-half stack.

The two frontends converge on the same `BootInfo` struct at `0x0500` with mirrored arena layout (`0x6000`, `0x7000`, `0x7200`). This mirroring is intentional and documented in code â€” it lets the shared kernel stay low identity-linked without teaching it multiple boot ABIs.

### 3.2 Shared kernel contract

`BootInfo` ([src/kernel/bootinfo.h](../src/kernel/bootinfo.h)) is magic+versioned, `#pragma pack(1)`, and guarded by exhaustive `static_assert` on both size (128 bytes) and every field offset. On entry the kernel calls `OwnBootInfo()` to deep-copy the header, memory map, modules, and strings into kernel BSS â€” after that, bootloader staging memory is no longer referenced.

### 3.3 Kernel initialization sequence

`KernelMain` in [src/kernel/kernel.cpp:1645](../src/kernel/kernel.cpp#L1645) follows one deterministic sequence on both boot paths:

1. serial hello, own `BootInfo`
2. BSP `cpu` initialization (per-CPU page seeded by frontend)
3. bitmap page-frame allocator from `std::span<const BootMemoryRegion>`
4. reserve modules + framebuffer + kernel image ranges
5. build kernel-owned identity page tables for all usable RAM above `kKernelReservedPhysicalStart`
6. explicitly `MapIdentityRange()` boot-critical non-usable ranges: every module, the framebuffer, and the RSDP page
7. switch to kernel CR3
8. `platform_init()` â€” ACPI (RSDP â†’ XSDT/RSDT â†’ MADT + MCFG) â†’ PCIe ECAM enumeration â†’ `virtio-blk` probe â†’ sector-0/1 signature verification
9. PIC/IOAPIC/LAPIC init, AP bring-up
10. terminals, text-display backend selection
11. IDT + keyboard + console input + ISA IRQ enable
12. task tables, kernel idle thread
13. load `/bin/init` from initrd, start multitasking at 1000 Hz

### 3.4 Memory architecture

- **Physical:** `PageFrameContainer` is a bitmap allocator seeded from `BootMemoryRegion[]`. Low bootstrap, kernel image reservation, and arbitrary ranges (modules, framebuffer) are explicitly reserved. Bitmap is parked at `[0x20000, 0x60000)`.
- **Virtual:** the kernel remains low identity-linked (supervisor VA â‰¡ PA for the kernel's working set). User space lives in a dedicated PML4 slot (`kUserPml4Index = 1`, base `0x0000008000000000`), so user mappings cannot collide with the identity-mapped kernel range.
- **User layout:** image at `0x8000_0040_0000`, 64 KiB stack ending at `0x8000_4000_0000` with one guard page below. ELF `PT_LOAD` permissions are mapped to page flags (`W`, `NX`) from ELF p-flags.

### 3.5 CPU, GDT, TSS, traps

- Per-CPU `struct cpu` page carries GDT, TSS, kernel stack, current thread, and self-pointer.
- GDT has kernel code/data, user code/data (0x23/0x1B), and a 64-bit TSS. `TSS.RSP0` is updated on every context switch â€” hardware uses it, not hardware task switching.
- One unified `TrapFrame` drives exceptions, IRQs, syscalls, and the scheduler return path. `trap_dispatch()` in `kernel.cpp` branches only on vector range. User exceptions kill the current process; kernel exceptions halt.

### 3.6 Syscalls and observability

Nine syscalls on `int 0x80` ([src/kernel/syscall_abi.h](../src/kernel/syscall_abi.h)): `write`, `exit`, `yield`, `getpid`, `read`, `observe`, `spawn`, `waitpid`, `exec`. ABI uses the SysV AMD64 register layout.

[src/uapi/os1/observe.h](../src/uapi/os1/observe.h) defines a versioned fixed-record contract (`Os1ObserveHeader` + typed records). Five record kinds today:

| Kind | Record | Source |
| --- | --- | --- |
| `OS1_OBSERVE_SYSTEM` | `Os1ObserveSystemRecord` | boot source, console kind, ticks, pages, procs, cpus, pci count, virtio-blk presence + capacity |
| `OS1_OBSERVE_PROCESSES` | `Os1ObserveProcessRecord[]` | one row per live thread (pid/tid/cr3/states/name) |

All copying crosses the user/kernel boundary through `CopyToUser()`/`CopyFromUser()` + `CopyUserString()`, which walk the user page tables to translate pointers rather than relying on forced faults.

### 3.7 Process and scheduling model

`AddressSpace` / `Process` / `Thread` with fixed-capacity tables (`kMaxProcesses = 32`, `kMaxThreads = 32`) allocated out of page frames ([src/kernel/task.h](../src/kernel/task.h), [src/kernel/task.cpp](../src/kernel/task.cpp)). One thread per process in M5; scheduler is simple round-robin with a wraparound index.

Blocking is first-class: `ThreadWaitReason::{ConsoleRead, ChildExit}`. `TryCompleteConsoleRead` / `TryCompleteWaitPid` are split so the syscall entry can either fast-path or park the thread and hand it back to the scheduler. Deferred reaping (`reapDeadThreads()`) runs from another thread's stack so a dying thread never frees its own.

`exec` is implemented as image swap in place: load new image into a fresh PML4, swap the thread's `address_space_cr3`, `WriteCr3`, then tear down the old address space. No `fork`, no args/env, no fd table yet.

## 5. Strengths

1. **One kernel, two frontends.** The `BootInfo` + low-identity mirror strategy is the single biggest architectural win in this revision. Every other subsystem benefits because it only ever sees one boot model. The shim's virtâ†’phys translation step (`TranslateLimineVirtual`) is the detail that made this actually work.
2. **Observability UAPI is the right abstraction.** `observe(kind, buf, len)` returning a versioned header + typed records is strictly better than having the shell parse boot logs. It survives log-format churn, it's tested by the `smoke_observe*` pair, and it composes naturally with future record kinds (blocks, net, filesystems).
3. **CI matrix proves portability claims.** Having BIOS and UEFI both exercise `acpi`, `pci`, `virtio-blk smoke ok`, and `shell prompt ready` on every PR means "works on both boot paths" is a mechanical fact, not a hopeful assertion.
4. **Process teardown correctness.** The "deferred reap on another stack" discipline (`reapDeadThreads`) and the ring-3 trap policy (fault â†’ mark dying â†’ schedule away â†’ reaper later) avoid the class of bugs where a dying thread frees its own stack or address space. This is the kind of detail that usually requires painful debugging to get right.
5. **Small, asserted ABIs.** `BootInfo` offsets are all `static_assert`ed; `Os1ObserveHeader` carries an ABI version; ELF loader rejects memsz<filesz and file-offset overflow. The code is paranoid about the right things.
6. **Build is principled.** One place (CMakeLists.txt) owns OVMF discovery, fixed LBA layout, and firmware-absence signaling. CI cannot drift from the build's idea of "correct" firmware paths.

## 6. Weaknesses And Risks

1. **Low identity-linked kernel is a cap on future changes.** Keeping supervisor VA â‰¡ PA forever limits where physical memory can live (the PFA implicitly assumes `uint64_t` physical addresses are dereferenceable), forces `VirtualMemory::DestroyUserSlot` to accept `cr3` as a raw pointer, and excludes a clean higher-half transition unless we bite that bullet later. *Risk:* every new subsystem that dereferences physical pointers grows this technical debt.
2. **Fixed-capacity tables.** `kMaxProcesses = 32`, `kMaxThreads = 32`, `kPlatformMaxPciDevices`, `kPlatformMaxCpus` are all compile-time. Fine today, but the same "allocate from the PFA once" pattern should generalize â€” several failure paths right now are "table full â†’ abort". The PCIe cap is already one `qemu -device ...` away from being surprising.
3. **`virtio-blk` is a smoke probe, not a block layer.** `VirtioBlkReadSector` is a 1M-iteration busy-spin for queue progression, single-request at a time, polling-only, mapped to identity addresses. There is no block-device abstraction above it, so any filesystem work will have to bring its own driver contract.
4. **APs are dead-ended.** `cpu_bootothers` brings up APs but they land in `cpu_idle_loop` (`cli; hlt`). `firstRunnableUserThread` only runs on the BSP. Per-CPU data exists, but no per-CPU run queues, no cross-CPU IPI-driven wake. The scaffolding is right; the scheduler isn't there yet.
5. **Userland is one file with two names.** `src/user/programs/sh.cpp` serves as both init and shell. This is pragmatic and honest, but it means there is no real `init` (no service management, no argv/envp, no environment). Any "init owns the shell + spawns `getty`-equivalents" progression needs argv/env first.
6. **`src/user/programs/init.c` is orphaned.** It is not referenced by any CMake target. Either a leftover from pre-M5 or an intentional "keep around as a reference" â€” right now it's just dead code that a future reader will follow.
7. **Single syscall entry, no fast path.** `int 0x80` is cheap to reason about but ~5Ã— slower than `SYSCALL`/`SYSRET`. It'll matter once any workload is actually syscall-bound.
8. **No persistent storage path beyond initrd.** This is the direct next-milestone wall: without a block layer and a filesystem, `exec`/`spawn` can only ever run what's packed into the initrd.
9. **Polling-mode virtio and PIC-era IRQs.** MSI/MSI-X are not used; the IOAPIC is configured but `virtio-blk` ignores its IRQ line. Migrating to MSI at the same time we introduce a block layer would be cheap; doing it later will mean touching the new block layer again.
10. **No kernel unit tests.** The smoke matrix is excellent as an integration story, but there's no in-process test harness for pure kernel logic (e.g. `relinkRunnableThreads`, CPIO parsing, `ParseMcfg`). These are the parts most likely to break silently under refactoring.

## 8. Security And Safety

The project's threat model is "don't crash the kernel when userland is buggy"; it is not "defend against a hostile guest." Within that scope:

- User pointer translation is page-table walking (good), not trusted faults (also good).
- User ELF load validates `memsz >= filesz`, rejects segments outside the user PML4 slot, and clears the address space on any failure. Good.
- Shell built-in `exec` can launch any initrd path the caller spells â€” acceptable since there's no multi-user boundary today.
- `initrd_cpio` is trusted because the image is built by us; no checksum verification at runtime. Reasonable for now; revisit when persistent storage lands.

No changes urgent here. File this under "recheck at Milestone D."

## 9. Recommendations (prioritized)

### Near-term (next 1â€“2 milestones)

1. **Block-device abstraction above `virtio-blk`.** Introduce a narrow `BlockDevice` interface with `read_sectors(lba, count, buf)` and slot the current `VirtioBlkReadSector` behind it. This is the precondition for any filesystem work and is cheap right now â€” there's exactly one driver.
2. **Promote `virtio-blk` to MSI-X and to a real request-slab layout.** Do this before building a filesystem on top, not after. The queue-size and notify-offset logic is already in place.
3. **Extract `kernel/syscalls.cpp` from `kernel.cpp`.** Move `Sys*`, `SysObserve*`, `TryComplete*`, `Wake*`, `CopyToUser`/`CopyFromUser`. Leaves `kernel.cpp` as the init + dispatch + idle thread file.
4. **Argv/envp for user processes.** Small surface change (extend `spawn`/`exec` + push a SysV-shaped arg vector onto the initial user stack in `LoadUserElf`). Unlocks a real shell story (`echo` already wants it; every future user program will).
5. **Delete `src/user/programs/init.c`** or turn it into a second test binary. Dead code with a name that collides with a kernel-visible path is a trap for future readers.
6. **In-process kernel unit tests.** Start with things that don't need the scheduler: CPIO iteration, MADT/MCFG parsers, ELF header validation. The host can run them natively.

### Medium-term

7. **First filesystem.** FAT32 read-only is probably the cheapest first hop with the virtio-blk smoke disk as a convenient test medium. It also exercises the new `BlockDevice` interface.
8. **Per-CPU run queues + IPI-driven wake.** Promote APs from `cpu_idle_loop` to genuine schedulers. The cpu-page and TSS scaffolding is already there.
9. **Fast-path syscall entry (`SYSCALL`/`SYSRET`)** with `int 0x80` retained as a compatibility shim during the transition. This is a one-week change and pays off for every syscall-heavy workload afterward.
10. **`virtio-net` probe + a minimal loopback/echo.** Do it before the full TCP stack so the driver/IRQ model matures on a simpler target.

### Longer-term

11. **Higher-half kernel transition.** The cost keeps growing every time a new subsystem adds "just dereference this physical pointer" code. Paying it while there's still one driver and one filesystem is dramatically cheaper than paying it later.
12. **Multiuser/permissions.** Required for the SSH milestone and worth shaping before the TCP stack lands so identity can flow through session bring-up from day one.

## 10. Open Questions (for explicit decision)

- **Where does the kernel live virtually after Milestone D?** Low-identity-forever or higher-half-with-PMM-direct-map? This decision shapes every future driver.
- **What is the first persistent filesystem?** FAT32 for compatibility, or a bespoke simple FS (ext2-like) for minimality? Affects tool chain (mkfs) and testing story.
- **MSI/MSI-X now, or after networking?** Cost-of-delay vs. appetite.
- **One initrd staging vs two builds (`init` and `sh` diverge)?** Easy to keep "one binary, two names" forever, but if `init` grows service management it will want to be its own program.
- **Where do per-CPU run queues live â€” in `cpu.h` or a new `sched.h`?** This shapes how cleanly SMP scheduling drops in.
