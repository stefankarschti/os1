## 1. Executive Summary

`os1` is now past the “large kernel file with interesting pieces” phase. The current repository contains a coherent, source-split `x86_64` teaching OS with two boot frontends, one shared boot contract, a protected ring-3 userland, a shell, structured observability, ACPI/PCIe discovery, and a first modern block-device proof through `virtio-blk`.

The strongest parts are architectural clarity and end-to-end vertical slices:

- Boot is cleanly normalized through `BootInfo`: BIOS assembly in [../src/boot/bios/](../src/boot/bios), the Limine shim in [../src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp), and the shared kernel entry in [../src/kernel/core/kernel_main.cpp](../src/kernel/core/kernel_main.cpp) converge on the same handoff.
- The source-tree ownership split is real, not cosmetic. Kernel code now has meaningful homes under `core`, `handoff`, `mm`, `proc`, `sched`, `syscall`, `console`, `drivers`, `platform`, `storage`, `fs`, `vfs`, and `security`; this is reflected in [../src/kernel/CMakeLists.txt](../src/kernel/CMakeLists.txt).
- Ring-3 userland is no longer just a demo. `/bin/init` executes `/bin/sh`, the shell can run builtins and spawn initrd programs, user faults are contained, and `copycheck` tests negative syscall-copy cases.
- Observability is a real UAPI through [../src/uapi/os1/observe.h](../src/uapi/os1/observe.h), consumed by the shell instead of scraping boot logs.
- The build and smoke workflow is unusually good for this stage: CMake builds UEFI ISO and BIOS raw images, QEMU smokes cover baseline, observe, spawn, and exec paths on both boot frontends.

The biggest risks are now less about “can it boot?” and more about the next foundation layer:

- SMP is intentionally incomplete. APs boot, initialize local CPU state, and then park in an interrupt-disabled idle loop. The BSP owns real scheduling.
- Storage is still early. `virtio-blk` is now isolated under [../src/kernel/drivers/block/virtio_blk.cpp](../src/kernel/drivers/block/virtio_blk.cpp) and publishes a `BlockDevice`, but it is polling, read-only, single-device, and still used primarily as a boot smoke path.
- There is no filesystem namespace, descriptor model, multiuser security model, networking stack, pseudo-terminal layer, or GUI. The placeholders under [../src/kernel/vfs/README.md](../src/kernel/vfs/README.md), [../src/kernel/security/README.md](../src/kernel/security/README.md), and [../src/kernel/drivers/net/README.md](../src/kernel/drivers/net/README.md) are honest: these domains have not started.
- The docs are behind the code in several places. The most visible drift is `/bin/ascii` being present in the initrd build but absent from README/architecture artifact lists and smoke observe markers.

Overall: the project is healthy and more structured than the prior review assumed. The next work should avoid another broad refactor and instead harden the system contracts needed for persistence, device growth, and eventual SMP.

## 2. Current State From Code

### Boot And Platform

Current reality:

- The BIOS path lives under [../src/boot/bios/](../src/boot/bios). `boot.asm` loads the first sector of `kernel16.bin`, `kernel16.asm` finishes the 16-bit loader, gathers E820 memory, scans for RSDP, reads the fixed-layout kernel/initrd payloads, enters long mode through `long64.asm`, expands the low kernel ELF, builds `BootInfo`, and calls `kernel_main`.
- The UEFI path is a Limine frontend in [../src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp). It requests HHDM, memory map, framebuffer, bootloader info, command line, modules, RSDP, and SMBIOS; translates Limine virtual pointers to physical addresses; loads `kernel_bios.elf` as the shared low-half kernel; creates a low identity window; and jumps into `kernel_main`.
- [../src/kernel/handoff/boot_info.hpp](../src/kernel/handoff/boot_info.hpp) is the stable boot contract. The kernel immediately deep-copies it through `own_boot_info()`.
- [../src/kernel/core/kernel_main.cpp](../src/kernel/core/kernel_main.cpp) still owns the high-level bring-up sequence, but major responsibilities have moved out: trap dispatch, IRQ handling, fault policy, syscall dispatch, platform discovery, process lifecycle, scheduling, and drivers now live in separate modules.
- [../src/kernel/platform/init.cpp](../src/kernel/platform/init.cpp) is ACPI-first. Limine requires ACPI; BIOS can fall back to legacy MP tables through [../src/kernel/platform/legacy_mp.cpp](../src/kernel/platform/legacy_mp.cpp).

Assessment:

The boot architecture is one of the strongest pieces of the project. The one misleading artifact is naming: `kernel_bios.elf` is the shared low-half kernel, not BIOS-only code. [ARCHITECTURE](ARCHITECTURE.md) explains this correctly, but the name remains a future cleanup target.

### Memory Management

Current reality:

- [../src/kernel/mm/page_frame.cpp](../src/kernel/mm/page_frame.cpp) implements a bitmap physical page allocator seeded from the normalized boot memory map.
- [../src/kernel/mm/virtual_memory.cpp](../src/kernel/mm/virtual_memory.cpp) implements four-level page-table creation, identity mapping, user page allocation, protection, translation, kernel PML4 slot cloning, and user-slot teardown.
- [../src/kernel/mm/user_copy.cpp](../src/kernel/mm/user_copy.cpp) now performs the syscall copy boundary: canonical address checks, range checks against the user window, overflow checks, page translation, `User` permission checks, and `Write` checks for kernel-to-user copies.
- User mappings live in PML4 slot 1, with constants in [../src/kernel/handoff/memory_layout.h](../src/kernel/handoff/memory_layout.h). The kernel still relies on low supervisor identity mappings.

Assessment:

The critical user-copy permission bug from [Review 2026-04-24](2026-04-24-review.md) appears resolved in current source. It should no longer be carried forward as an active bug. What remains is a design risk: `VirtualMemory::translate()` reports leaf flags, while the effective CPU permission is the intersection of all page-table levels. Current user mappings are created through code that propagates the `User` bit up the tree, so this is an invariant rather than an observed exploit. It deserves a host test or an effective-permission helper before the VM layer gets more complex.

The physical allocator is adequate for the current system but still has low-level technical debt: [../src/kernel/mm/page_frame.hpp](../src/kernel/mm/page_frame.hpp) includes [../src/kernel/console/terminal.hpp](../src/kernel/console/terminal.hpp), which makes the lowest memory layer depend on console types for no visible allocator reason. That is an ownership smell, not an immediate runtime bug.

### Interrupts And Exceptions

Current reality:

- [../src/kernel/arch/x86_64/interrupt/interrupt.cpp](../src/kernel/arch/x86_64/interrupt/interrupt.cpp) builds the IDT, installs exception stubs, IRQ stubs, and the `int 0x80` syscall gate.
- [../src/kernel/core/trap_dispatch.cpp](../src/kernel/core/trap_dispatch.cpp) classifies traps into IRQ, syscall, or exception.
- [../src/kernel/core/irq_dispatch.cpp](../src/kernel/core/irq_dispatch.cpp) handles timer and keyboard IRQs, polls serial input on timer ticks, wakes console readers, acknowledges LAPIC/PIC paths, and schedules on timer ticks.
- [../src/kernel/core/fault.cpp](../src/kernel/core/fault.cpp) kills user-mode faulting threads and halts on kernel-mode exceptions.
- The active periodic tick comes from the PIT setup in [../src/kernel/drivers/timer/pit.cpp](../src/kernel/drivers/timer/pit.cpp) through `set_timer(1000)`. The LAPIC timer code exists, but the LAPIC timer is masked in [../src/kernel/arch/x86_64/apic/lapic.cpp](../src/kernel/arch/x86_64/apic/lapic.cpp).

Assessment:

The trap pipeline is now reasonably well separated and understandable. The remaining risk is transitional interrupt-controller ownership: PIC, IOAPIC, LAPIC, PIT, and legacy acknowledgements coexist. That is acceptable while QEMU smokes are the target, but it is not yet a clean interrupt substrate for MSI/MSI-X, interrupt-driven storage, networking, or SMP scheduling.

### Scheduling And SMP

Current reality:

- Processes and threads are fixed tables of 32 entries in [../src/kernel/proc/process.hpp](../src/kernel/proc/process.hpp) and [../src/kernel/proc/thread.hpp](../src/kernel/proc/thread.hpp).
- [../src/kernel/sched/thread_queue.cpp](../src/kernel/sched/thread_queue.cpp) performs simple runnable-thread selection over the fixed table.
- [../src/kernel/sched/scheduler.cpp](../src/kernel/sched/scheduler.cpp) reaps dead threads, wakes child waiters, and selects the next thread.
- [../src/kernel/sched/idle.cpp](../src/kernel/sched/idle.cpp) provides the BSP idle thread.
- APs are discovered from ACPI or legacy MP data, allocated CPU records in [../src/kernel/platform/topology.cpp](../src/kernel/platform/topology.cpp), started through [../src/kernel/arch/x86_64/cpu/cpu.cpp](../src/kernel/arch/x86_64/cpu/cpu.cpp), and then enter `cpu_idle_loop()` with `cli; hlt`.

Assessment:

SMP is topology plus AP bring-up, not SMP scheduling. This is a deliberate stage, but it is also the major concurrency cliff. Global structures such as `page_frames`, `processTable`, `threadTable`, terminal buffers, console input queues, platform state, and driver state are not protected by a lock/preemption model. That is fine only because APs do not run kernel work.

The AP startup wait loop in `cpu_boot_others()` still spins forever if an AP does not set `booted`. A timeout with a clear diagnostic would turn a hardware/platform failure from a silent boot hang into an understandable failure.

Limine/UEFI is the default, BIOS remains a compatibility target, ACPI MADT/MCFG are parsed, PCIe is enumerated, and `virtio-blk` is probed. This matches the platform goals well. Follow-on work should focus on MSI/MSI-X, HPET or better timer discovery, PCI resource ownership, and a driver registry.

### Protected Userland

Status: implemented as a vertical slice.

Ring-3 execution, ELF loading, syscall dispatch, user-copy validation, `spawn`, `waitpid`, `exec`, and fault containment are implemented. This satisfies Milestone C in [GOALS](../GOALS.md). ### Process And Scheduling Model

Status: useful on BSP, early for SMP.

The process/thread split and wait/reap behavior are real. SMP scheduling is not. The project should keep AP scheduling behind an explicit lock/preemption/run-queue design, not sneak it in as a small patch.

### Storage And Filesystems

Status: early.

The `BlockDevice` facade and `virtio-blk` driver are the right direction, but persistence has not started. The current initrd is still the only program namespace. Milestone D remains the natural next major objective.

### Networking And SSH

Status: not started.

This matches the intended sequence. Networking before storage/device/interrupt cleanup would create more surface area than the current kernel can comfortably support. SSH is later still, because it depends on networking, ptys, sessions, credentials, filesystem-backed configuration, and security-sensitive randomness.

### GUI And Desktop Direction

Status: framebuffer text only.

The framebuffer backend is a terminal presenter, not a GUI. This is correct for a terminal-first OS. GUI work should wait for input events, session/pty design, display buffer ownership, and more mature process control.

### Accelerator Awareness

Status: not started, but PCIe discovery is in place.

The goal of keeping future accelerator support possible is still mostly architectural. The real prerequisites are DMA-safe allocation, resource ownership, interrupt completion, memory pinning, and user/kernel submission boundaries.

## 5. Overlooked Goals And Blind Spots

- **Effective page permissions need tests.** `user_copy.cpp` checks leaf flags and user ranges. The VM layer should either expose effective permissions or be covered by tests proving intermediate page-table permission invariants are maintained.
- **The smoke suite can drift from the initrd.** `/bin/ascii` is now packaged but not checked in observe smokes. Add a build-time or smoke-time initrd manifest check before more programs are added.
- **BIOS image size limits are implicit.** The current initrd is well under 64 KiB, but the raw BIOS image layout has fixed sector counts and no obvious size assertion. The failure mode should be a build error, not a truncated payload.
- **SMP readiness is broader than scheduler code.** AP scheduling requires a kernel-wide lock/preemption story, console/input rules, allocator rules, platform-state rules, and driver rules.
- **Device model work should precede networking.** `virtio-net` will need shared virtio queue helpers, DMA ownership, IRQ completion, and observability. Starting with IP/TCP code would be backwards.
- **Terminal/session work is a prerequisite for SSH.** SSH is not just crypto plus TCP. It requires ptys, sessions, login policy, credentials, and filesystem-backed configuration.
- **Documentation needs a freshness policy.** Current docs are mostly good, but several “current review” links and artifact lists are stale within days. A review index or `latest-review` pointer would reduce link churn.
- **Host-side tests are still missing.** The project’s integration tests are strong, but pure parsing and permission logic still rely on whole-system QEMU tests or manual review.
- **Low identity mapping remains a strategic decision.** It keeps early development simple but will shape DMA, security, higher-half migration, and physical-memory access patterns.

## 6. Roadmap

### Near Term

1. **Update documentation drift around `/bin/ascii`, framebuffer terminal geometry, and current review links.** This matters because the docs are otherwise high quality; small stale claims now stand out and mislead readers.

2. **Add build-time size assertions for BIOS payload slots.** Check `kernel_bios.elf` and `initrd.cpio` against `OS1_KERNEL_IMAGE_SECTOR_COUNT` and `OS1_INITRD_IMAGE_SECTOR_COUNT` in [../CMakeLists.txt](../CMakeLists.txt). This prevents quiet raw-image truncation.

3. **Add an initrd manifest smoke marker or host check.** Make the packaged file list mechanically match expectations, including `/bin/init` and `/bin/ascii`.

4. **Add host tests for user-copy and VM permission behavior.** Prioritize canonical/range checks, write permission checks, read-only user pages, and effective permission propagation through intermediate page-table entries.

5. **Give AP startup a timeout and diagnostic.** `cpu_boot_others()` should not be able to hang forever without saying which AP failed.

6. **Start argv/envp design for `spawn` and `exec`.** This is a small but high-leverage userland ABI step before more programs or filesystem-backed loading.

7. **Reduce the remaining ownership leaks.** Remove the `mm` to `console` header dependency and shrink the transitional namespace facade over time.

### Mid Term

1. **Turn `BlockDevice` into a real read path.** Define error reporting, sector counts, request lifetime, and buffer ownership before adding a filesystem.

2. **Split shared virtio transport from `virtio-blk`.** Queue setup, feature negotiation, notification, and device status logic should be reusable by `virtio-net`.

3. **Implement MSI/MSI-X or a deliberate interrupt-completion path.** Polling block reads are fine for smoke; filesystems and networking need completions.

4. **Add a read-only filesystem and filesystem-backed `exec`.** Start with a reproducible image workflow and read-only semantics. Filesystem-backed loading should come before broad POSIX growth.

5. **Introduce a minimal file descriptor table.** Console and filesystem files should stop being special syscall cases before POSIX-like userland grows.

6. **Design kernel synchronization before AP scheduling.** Define spinlocks, interrupt masking rules, lock ordering, and run-queue ownership before APs run user work.

### Later

1. **Bring APs into scheduling.** Treat this as a concurrency milestone, not a scheduler-only patch.

2. **Implement networking in layers.** Start with `virtio-net`, packet buffers, Ethernet, ARP, IPv4, ICMP, UDP, then TCP.

3. **Build ptys, sessions, credentials, and login.** These are prerequisites for remote administration and SSH.

4. **Add SSH only after the substrate exists.** SSH needs TCP, crypto, entropy, credentials, ptys, sessions, filesystem-backed config, and permission checks.

5. **Revisit higher-half or direct-map layout.** Do this before multiuser security or serious DMA/device isolation becomes expensive to retrofit.

6. **Explore framebuffer compositor or GUI work only after terminal/session maturity.** A terminal compositor is plausible; a desktop should remain secondary.

7. **Approach accelerators through resource ownership first.** PCI discovery alone is not enough. DMA-safe allocation, memory pinning, command submission, completion, and access control come first.

### Dependency Ordering

- Before networking: shared virtio transport, interrupt completion, packet buffers, device observability, and DMA/resource ownership.
- Before SSH: networking, credentials, authentication storage, ptys, sessions, filesystem-backed config, entropy, and crypto policy.
- Before GUI: input events, terminal/session model, display-buffer ownership, scheduler stability, and process control.
- Before SMP expansion: locks, preemption rules, per-CPU state ownership, scheduler IPIs, interrupt routing, and console/input concurrency rules.
- Before POSIX growth: file descriptors, filesystem namespace, argv/envp, process lifecycle semantics, and error-code conventions.
- Before accelerator work: PCI resource ownership, DMA-safe memory, interrupt completion, memory pinning, and user/kernel submission boundaries.
