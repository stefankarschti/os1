## Current Architecture Assessment

The current stack is coherent for an educational kernel:

- boot chain: BIOS MBR -> stage-2 real-mode loader -> temporary long-mode transition -> simple ELF expansion -> `KernelMain`
- memory model: low bootstrap scratch + bitmap physical allocator + identity-mapped four-level paging
- CPU model: per-CPU page-sized `cpu` structure with GS-based access
- device model: VGA text mode, PS/2 keyboard, serial debug, PIC + IOAPIC + LAPIC
- scheduler model: PIT-driven round-robin kernel tasks with full register save/restore in assembly

The problem is not that the kernel has no architecture. It does. The problem is that the architecture is optimized for a 2008-2015 hobby-kernel path rather than a 2026 one:

- BIOS instead of UEFI or a modern handoff protocol
- MP tables instead of ACPI MADT
- fixed low-memory conventions instead of explicit ownership and allocation
- single-address-space kernel tasks instead of a clean process model
- older C++14 dialect instead of a safer freestanding C++20 style

That makes the project educationally valuable but strategically narrow.

## Roadmap Adapted For 2026

The roadmap below reflects the clarified project goals:

- modern boot path preferred over BIOS as the primary experience
- kernel kept bootloader-agnostic through a common handoff contract
- first “real OS” milestone is protected user-mode execution, not just wider boot coverage
- C++ remains the main implementation language, with assembly reserved for boot and narrow low-level boundaries

### Milestone 1: Boot Contract And Kernel Stabilization

Goal: make the current kernel easier to evolve by isolating boot assumptions and cleaning up fragile runtime dependencies.

Scope:

- introduce a bootloader-agnostic `BootInfo` or equivalent kernel handoff structure
- keep the current BIOS path working behind that contract
- add a headless QEMU smoke test and CI
- centralize all fixed memory-layout constants
- remove hard-coded task CR3 usage
- move task metadata off `0x408`
- adopt a freestanding C++20 subset

Why first:

- it is the lowest-risk path
- it improves both the future modern boot path and the future user-mode path
- it turns hidden bootstrap assumptions into explicit kernel interfaces

### Milestone 2: Process Model And Isolation

Goal: evolve from demo kernel tasks toward a first real protected userspace milestone.

Scope:

- TSS and kernel-entry stack handling
- page permissions suitable for user/kernel separation
- per-process address spaces
- syscall entry/exit path
- ELF user-program loading from an initrd or simple filesystem

Why second:

- this is the maintainer’s preferred “real OS” milestone
- the current QEMU/platform path is already sufficient to make progress here
- it creates a much more meaningful system boundary than broader early hardware support alone

### Milestone 3: Modern Default Boot Path

Goal: make a modern boot flow the primary way to start the kernel.

Scope:

- adopt Limine or a similar modern boot protocol
- support UEFI-first bring-up and framebuffer handoff
- map the boot protocol data into the common `BootInfo` contract
- keep BIOS as an optional secondary path if it remains low-maintenance

Why third:

- the kernel-handoff abstraction from Milestone 1 makes this much safer
- it advances modern platform support without forcing a bootloader-specific kernel design

### Milestone 4: Modern Platform Support

Goal: move from “boots via a modern loader” to “understands modern platform topology.”

Scope:

- add ACPI RSDP/XSDT/MADT parsing
- keep MP-table parsing only as fallback if BIOS support remains
- add PCI/PCIe enumeration
- prefer `virtio-blk` and `virtio-net` first, then NVMe, and defer USB/xHCI until later

Why fourth:

- it is still important, but it does not need to block the first isolated-user-program milestone
- it becomes easier once the kernel handoff and process foundations are less brittle

## C++20 Guidance For This Codebase

I would not jump from the current code straight into “full modern C++ kernel framework” mode. The safer path is a freestanding C++20 subset that improves invariants without forcing large library or ABI bets.

Recommended first moves:

- `constexpr` and `constinit` for descriptor constants, scancode tables, and memory-layout definitions
- `enum class` for IRQ vectors, exception numbers, and page flags
- `std::array` for fixed-size tables like GDT/IDT support arrays and scancode maps
- `std::span` for E820 memory blocks and other pointer-plus-count pairs
- `[[nodiscard]]` on allocator / VM / initialization functions that can fail
- `using enum` and `std::to_underlying` only where it makes bitfield code cleaner

Recommended constraints:

- keep exceptions, RTTI, and heap-heavy containers out of the kernel for now
- avoid introducing `std::string`, iostreams, or allocator-dependent abstractions before the runtime story is clearer
- prefer zero-cost language features and simple headers from the freestanding subset
- keep assembly for boot, interrupt entry/exit, syscall glue, and context switching where that remains the clearest and lowest-level expression of the machine contract

That path modernizes the code without fighting the nature of a small freestanding kernel.

## Suggested Hobby-OS Directions For 2026

If you want `os1` to become more than a historical bring-up kernel, these are the most compelling directions:

1. `Modern platform kernel`: switch to a UEFI-capable boot path, parse ACPI, enumerate PCIe, and use virtio/NVMe. This is the best route if the goal is “boot on modern machines and modern VMs.”
2. `Tiny Unix-like system`: keep text mode or framebuffer text, add ring-3 ELF loading, an initrd, a shell, and a few syscalls. This is the best route if the goal is “a complete small OS someone can actually use.”
3. `SMP observability lab`: make per-CPU scheduling, tracing, and interrupt routing first-class, with serial trace streaming and debug visualizations. This is a great route if the goal is learning and demos rather than broad feature coverage.
4. `Graphics-first hobby OS`: adopt a GOP or framebuffer console, composited terminals, bitmap fonts, and input devices. This is attractive if you want the project to feel more novel than a classic VGA text-mode kernel.

My recommendation is a blend of `1` and `2`: modern boot/platform bring-up plus a tiny user-mode environment.

## Confirmed Project Direction

The maintainer clarified the following goals after the initial review draft:

1. BIOS is not the preferred long-term primary path. A modern boot protocol such as Limine is acceptable, and likely desirable, if it gets the project to UEFI and framebuffer support faster.
2. BIOS does not need to be abandoned outright. The preferred architecture is a bootloader-agnostic kernel handoff so multiple boot paths can exist behind one common contract.
3. The next meaningful OS milestone is protected user-mode execution with ELF loading and syscall boundaries, not merely broader boot compatibility.
4. Boot and platform work should still happen where it unblocks that path, but it should not displace the user-mode milestone without a clear reason.
5. C++ should remain the primary language for the codebase. Assembly remains acceptable in narrow low-level paths such as boot, interrupt stubs, syscall transitions, and context switching.
6. Additional systems languages are not ruled out in principle, but they should only be introduced for concrete technical benefit and not by default.

## Remaining Open Decisions

The roadmap is much sharper now. The main decisions still worth making explicitly are:

1. Whether the first modern boot implementation should be Limine specifically or another comparable protocol
2. Whether the first user-program milestone should target a text-mode environment first, or jump directly to framebuffer-backed console work once a modern boot path lands
3. Whether BIOS support should be maintained continuously during the transition, or allowed to lag while the modern path becomes primary

## Recommended Next Step

Start with the immediate milestone above and treat it as a stabilization sprint, not a feature sprint. Once the kernel has CI, named memory-layout constants, dynamic task metadata, a real CR3 path, and a C++20 baseline, the ACPI and modern-boot work becomes much less risky.
