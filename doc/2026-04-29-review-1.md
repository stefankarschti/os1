## 1. Executive Summary

### Current project state

`os1` has matured from a low-linked teaching kernel into a higher-half kernel substrate with a complete protected-userland vertical slice. Concretely, the source tree now provides:

- A shared higher-half kernel linked at `0xFFFFFFFF80000000 + physical`, with separate `R E` / `R` / `RW` `PT_LOAD` segments and matching final-CR3 permissions ([src/kernel/linker/kernel_core.ld](../src/kernel/linker/kernel_core.ld), [src/kernel/core/kernel_main.cpp:69-80](../src/kernel/core/kernel_main.cpp#L69-L80)).
- A kernel-owned direct map at `0xFFFF800000000000`, replacing the broad final identity map of earlier revisions ([src/kernel/handoff/memory_layout.h:69-119](../src/kernel/handoff/memory_layout.h#L69-L119), [src/kernel/mm/boot_mapping.cpp](../src/kernel/mm/boot_mapping.cpp)).
- Two boot frontends that converge on the same higher-half entry through a shared `BootInfo` ABI: legacy BIOS in [src/boot/bios](../src/boot/bios) and a Limine/UEFI shim relocated to `0xFFFFFFFF90000000` in [src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp).
- Build-time contract assertions for both the shared kernel envelope ([cmake/scripts/assert_kernel_boot_contract.py](../cmake/scripts/assert_kernel_boot_contract.py)) and the Limine shim layout ([cmake/scripts/assert_limine_shim_contract.py](../cmake/scripts/assert_limine_shim_contract.py)).
- ACPI-only platform discovery (RSDP → XSDT → MADT/MCFG), LAPIC/IOAPIC/PIC bring-up, PCI ECAM enumeration, and AP startup with bounded wait under [src/kernel/platform](../src/kernel/platform) and [src/kernel/arch/x86_64](../src/kernel/arch/x86_64).
- Ring-3 user programs loaded from a `cpio newc` initrd: `/bin/init`, `/bin/sh`, `/bin/yield`, `/bin/fault`, `/bin/copycheck`, `/bin/ascii` ([src/user/programs](../src/user/programs)).
- A 9-call syscall ABI on `int 0x80`: `write`, `exit`, `yield`, `getpid`, `read`, `observe`, `spawn`, `waitpid`, `exec` ([src/uapi/os1/syscall_numbers.h](../src/uapi/os1/syscall_numbers.h), [src/kernel/syscall/dispatch.cpp](../src/kernel/syscall/dispatch.cpp)).
- A multi-terminal text stack with VGA and framebuffer backends ([src/kernel/console](../src/kernel/console), [src/kernel/drivers/display](../src/kernel/drivers/display)).
- A polling, read-only `virtio-blk` device path that publishes a minimal `BlockDevice` facade ([src/kernel/drivers/block/virtio_blk.cpp](../src/kernel/drivers/block/virtio_blk.cpp), [src/kernel/storage/block_device.hpp](../src/kernel/storage/block_device.hpp)).
- An eight-test ctest matrix that covers UEFI and BIOS shell, observe, spawn, and exec smokes ([CMakeLists.txt:608-880](../CMakeLists.txt#L608-L880)).
- Two new design drafts that begin to define the project's *native* (post-POSIX) interface: [doc/shell-language/elf_interface_spec.md](shell-language/elf_interface_spec.md) and [doc/shell-language/os1-shell-language-first-draft.md](shell-language/os1-shell-language-first-draft.md).

The BSP runs the entire system. APs are discovered, started, marked `booted`, and parked in `cpu_idle_loop()` ([src/kernel/arch/x86_64/cpu/cpu.cpp:15-25](../src/kernel/arch/x86_64/cpu/cpu.cpp#L15-L25)).

### Strongest parts

1. **Address-model maturity.** The higher-half migration removed the largest remaining architectural debt. Every new subsystem now starts from a clean kernel/user/direct-map split with W^X permissions and CMake-provided `--defsym` linker contracts. This was the best moment to do the migration — the cost would have grown with every additional driver or syscall that treated physical addresses as `T*`.

2. **Boot-contract discipline.** The kernel is essentially bootloader-agnostic after `own_boot_info()`. Both boot paths build the same `BootInfo` arena at `0x500..0x8000` and call the same higher-half `kernel_main(BootInfo*, cpu*)`. Build-time scripts enforce the kernel envelope, the Limine shim base, and the BIOS image-slot layout.

3. **Operator slice.** Spawn / waitpid / exec / observe form a complete-enough operator surface to drive the system from a serial prompt without parsing logs. The smoke matrix exercises both boot paths against this surface, so regressions in this layer are caught automatically.

4. **Source-tree readability.** The split between `arch/x86_64`, `core`, `handoff`, `mm`, `proc`, `sched`, `syscall`, `console`, `drivers`, `platform`, `storage`, `fs`, `vfs`, `security`, `debug`, and `util` makes ownership obvious. Empty directories with `README.md` placeholders ([src/kernel/vfs/README.md](../src/kernel/vfs/README.md), [src/kernel/security/README.md](../src/kernel/security/README.md), [src/kernel/drivers/net/README.md](../src/kernel/drivers/net/README.md)) act as growth seams that prevent future work from being homed in the wrong place.

### Biggest risks / gaps

1. **The block path is still smoke-grade.** [src/kernel/drivers/block/virtio_blk.cpp:234](../src/kernel/drivers/block/virtio_blk.cpp#L234) is a 1-million-iteration spin on `used->idx`, [src/kernel/drivers/block/virtio_blk.cpp:272-275](../src/kernel/drivers/block/virtio_blk.cpp#L272-L275) hard-returns failure for writes, and the `BlockDevice` facade in [src/kernel/storage/block_device.hpp](../src/kernel/storage/block_device.hpp) carries no request, completion, error, or async semantics. **No filesystem or VFS work should sit on top of this without a v2 contract.** This is the same finding as the prior review and has not been addressed.

2. **No host-side tests still exist.** The CI workflow in [.github/workflows/ci.yml:71-72](../.github/workflows/ci.yml#L71-L72) runs only ctest, which is a QEMU smoke loop. CPIO, ELF, user-copy, page-table walks, ACPI MADT/MCFG parsers, and PCI capability walks have no unit coverage. Every new subsystem will have to be validated through full boots until this is fixed. The repo has no `tests/` or `host_tests/` directory and no host toolchain branch in CMake.

3. **The Limine shim is now 1297 lines** ([src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp)). It owns request consumption, HHDM virtual translation, ELF physical loading, transition-mapping installation, framebuffer capture, module discovery, BootInfo construction, and the final `iretq`-style transfer. The prior review flagged it; the higher-half migration grew it further.

4. **Synchronization vocabulary does not exist.** `grep` finds exactly two atomic operations in the kernel — both `xchg(&cpu->booted, 1)` ([src/kernel/arch/x86_64/cpu/cpu.cpp:159, 171](../src/kernel/arch/x86_64/cpu/cpu.cpp#L159)). There is no spinlock, no IRQ-disable discipline, no per-CPU vs global state distinction, and no documented preemption model. As long as APs stay parked, this is fine. The moment any AP runs scheduler or driver code, the kernel will have undefined behavior.

5. **Wait state is still untyped.** [src/kernel/proc/thread.hpp:50-54](../src/kernel/proc/thread.hpp#L50-L54) overloads the same `wait_address`/`wait_length` pair for `ConsoleRead` (buffer pointer, length) and `ChildExit` (status pointer, pid). [src/kernel/syscall/wait.cpp:107](../src/kernel/syscall/wait.cpp#L107) reads them back in that overloaded order. The bug surface from this representation grows with every new wait reason.

6. **No process credentials, no FDs, no argv/envp.** [src/kernel/proc/process.hpp:32-40](../src/kernel/proc/process.hpp#L32-L40) carries `pid`, state, address space, exit status, parent, name. There is no uid/gid, no fd table, no argv/envp loader path. Every goal beyond Goal C in [GOALS.md](../GOALS.md) requires at least the FD table.

7. **Two large design drafts have no implementation strategy.** [doc/shell-language/elf_interface_spec.md](shell-language/elf_interface_spec.md) defines a `.os.interface` ELF section schema, and [doc/shell-language/os1-shell-language-first-draft.md](shell-language/os1-shell-language-first-draft.md) defines a typed shell language with hosted objects, properties, methods, and events. They directly target GOALS.md's "native system interface" objective. Nothing in the kernel or syscall ABI exposes any of those concepts yet, and there is no staged path connecting the drafts to the running system. Without one, these drafts will rot the same way an early-2026 GUI design would have.

## 2. Current State From Code

### Boot path and platform assumptions

Two boot frontends converge on the same higher-half kernel:

- **Limine/UEFI** ([src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp)). The shim consumes Limine HHDM, memmap, framebuffer, RSDP, SMBIOS, modules, command-line, and bootloader-info responses; translates Limine-virtual pointers through the active page tables; loads `kernel.elf` by `p_paddr`; validates `p_vaddr == p_paddr + kKernelVirtualOffset`; installs only the transition mappings the kernel needs; and jumps through `limine_enter_kernel`. The shim itself is linked at `0xFFFFFFFF90000000` so it does not collide with the shared kernel window.
- **BIOS** ([src/boot/bios](../src/boot/bios)). `boot.asm` chain-loads `kernel16.asm`, which handles A20, long-mode probing, EDD reads, E820 collection, RSDP scan, BIOS cursor capture; `long64.asm` builds temporary tables and aliases the first 2 MiB into both low identity and the higher-half kernel window before jumping high; the loader expands `kernel.elf` to its `p_paddr` low load.

Both paths arrive at `kernel_main(BootInfo*, cpu*)` in [src/kernel/core/kernel_main.cpp](../src/kernel/core/kernel_main.cpp). The bring-up sequence is deterministic:

1. `own_boot_info()` deep-copies header, memory map, modules, strings into kernel BSS ([src/kernel/handoff/boot_info.cpp](../src/kernel/handoff/boot_info.cpp)).
2. `cpu_initialize_record(g_cpu_boot)` and `cpu_init()` set up GDT/TSS/`gs` for the BSP.
3. `page_frames.initialize(memory_regions, ...)` builds a bitmap allocator from `BootMemoryRegion[]`.
4. `reserve_tracked_physical_range(...)` reserves modules and framebuffer.
5. `VirtualMemory kvm` builds the final kernel CR3: bootstrap-low identity for AP/handoff stack, kernel image by `R E`/`R`/`RW` sections, direct map for usable RAM, explicit modules / framebuffer / RSDP windows.
6. `kvm.activate()` installs the new CR3, `g_kernel_direct_map_ready = true`, and the boot CPU pointer is rebound through the direct map (with the `g_cpu_boot->self` rebind that the higher-half migration review correctly called out).
7. `platform_init()` (ACPI → topology → PCI ECAM → virtio-blk probe) in [src/kernel/platform/init.cpp](../src/kernel/platform/init.cpp).
8. `pic_init`, `ioapic_init`, `lapic_init`, `cpu_boot_others(g_kernel_root_cr3)`. APs land in `cpu_idle_loop()`.
9. Display selection, terminal allocation, IDT registration, keyboard init, console-input init, ISA IRQ routing.
10. Process / thread tables allocated (`kMaxProcesses = 32`, `kMaxThreads = 32`), kernel idle thread created.
11. `load_user_program(/bin/init)` → `enter_first_thread`.

Hard platform assumptions:

- x86_64 only.
- ACPI is required. [src/kernel/platform/init.cpp:32-37](../src/kernel/platform/init.cpp#L32-L37) returns failure if the RSDP or required tables are missing — there is no MP-table fallback and no Device Tree path.
- PCI is ECAM/MCFG only.
- `BootInfo` caps memory regions at 128 and modules at 16 ([src/kernel/handoff/boot_info.hpp:13-14](../src/kernel/handoff/boot_info.hpp#L13-L14)). Real UEFI memory maps are routinely larger than this.
- Low bootstrap identity exceptions remain for the AP trampoline mailbox at `0x20-0x40` and the live BSP handoff stack page until the BSP migrates onto a kernel-thread stack.

### Memory management

Physical: bitmap-based `PageFrameContainer` in [src/kernel/mm/page_frame.cpp](../src/kernel/mm/page_frame.cpp). Allocator bitmap is fixed at `kPageFrameBitmapBaseAddress = 0x20000` for `kPageFrameBitmapSizeBytes = 0x40000`, capping the allocator at 8 GiB of tracked pages. There is no buddy / slab / cache / freelist on top.

Virtual: `VirtualMemory` in [src/kernel/mm/virtual_memory.cpp](../src/kernel/mm/virtual_memory.cpp). Owns root allocation, four-level walks, NX-aware permission writes, user-slot teardown, and CR3 activation. User CR3s clone only `kKernelPml4Index` and `kDirectMapPml4Index` ([src/kernel/mm/virtual_memory.cpp:275-286](../src/kernel/mm/virtual_memory.cpp#L275-L286)) — slot 0 is no longer cloned into user space.

User layout (still 1 process slot, slot 1):

- `kUserSpaceBase  = 0x0000_0080_0000_0000`
- `kUserImageBase  = 0x0000_0080_0040_0000`
- `kUserStackTop   = 0x0000_0080_4000_0000`
- `kUserStackPages = 16` (64 KiB)
- `kKernelThreadStackPages = 4` (16 KiB)

User-copy validation in [src/kernel/mm/user_copy.cpp](../src/kernel/mm/user_copy.cpp) checks canonical addresses, range overflow, the user region constraint, and walks page entries to confirm User+Write permission before letting the kernel touch user memory. The leaf-flag accumulation issue from earlier reviews still stands: `VirtualMemory::translate` reports leaf flags only.

Real risks here are now architectural rather than urgent:

- The bitmap allocator wastes a fixed 256 KiB whether the host has 256 MiB or 32 GiB.
- There is no kernel `kmalloc`/slab. Every dynamic kernel allocation is a page granule.
- No guard pages on kernel thread stacks. A 16 KiB kernel stack overflow walks into whatever the page-frame allocator handed out next.

### Interrupts / exceptions

IDT, PIC remap, IOAPIC, LAPIC, timer (PIT), keyboard, and the `int 0x80` user-callable gate are all set up in [src/kernel/arch/x86_64/interrupt/interrupt.cpp](../src/kernel/arch/x86_64/interrupt/interrupt.cpp). The trap entry stubs in [src/kernel/arch/x86_64/interrupt/inthandler.asm](../src/kernel/arch/x86_64/interrupt/inthandler.asm) and [src/kernel/arch/x86_64/interrupt/irqhandler.asm](../src/kernel/arch/x86_64/interrupt/irqhandler.asm) build a unified `TrapFrame` for exceptions, IRQs, syscalls, and scheduler return.

Kernel faults dump and halt; user faults mark the thread/process dying and reschedule ([src/kernel/core/fault.cpp](../src/kernel/core/fault.cpp)). The timer hook polls serial RX, increments `g_timer_ticks`, and calls the scheduler ([src/kernel/core/irq_dispatch.cpp](../src/kernel/core/irq_dispatch.cpp)).

This is sound for a uniprocessor kernel but is **not** an SMP interrupt architecture: there is no per-CPU IRQ vector reservation, no MSI/MSI-X support, no IPI infrastructure, no edge-vs-level normalization beyond the ACPI-derived `InterruptOverride` set, no IRQ ownership/priority model, and no threaded IRQ mechanism. Devices that need fast or deferred completion (any future virtio-net or NVMe) cannot be wired up cleanly to this design as-is.

### Scheduling / SMP state

`Thread` and `Process` are fixed-size objects, both in 32-entry tables ([src/kernel/proc/thread.hpp:18](../src/kernel/proc/thread.hpp#L18), [src/kernel/proc/process.hpp:12](../src/kernel/proc/process.hpp#L12)). Scheduling is a simple round-robin `next_runnable_thread` over the thread table ([src/kernel/sched/scheduler.cpp](../src/kernel/sched/scheduler.cpp)). The BSP also has a kernel idle thread.

SMP discovery, AP startup, and per-CPU `cpu` page allocation all work. `wait_for_ap_boot` is bounded ([src/kernel/arch/x86_64/cpu/cpu.cpp:13](../src/kernel/arch/x86_64/cpu/cpu.cpp#L13)). After AP startup, `init()` runs `ioapic_init()` and `lapic_init()` per AP and then drops into `cpu_idle_loop()` (`cli; hlt` forever).

The synchronization gap is concrete and large:

- `processTable`, `threadTable`, `g_next_pid`, `g_timer_ticks`, `page_frames`, `terminal[]`, `active_terminal`, `console_input` state, the PCI device list, and the virtio queue state are all unsynchronized.
- `xchg` is the only atomic primitive used by kernel C++ code.
- There is no `KernelLock`, no `IrqGuard`, no `local_irq_disable` discipline, no per-CPU run queue.

This is acceptable today because only the BSP runs anything other than idle. The moment a second CPU runs scheduler, IRQ, or driver code, every one of those globals is a race.

### User/kernel separation state

This is the most complete vertical slice in the project. ELF64 `ET_EXEC` images are loaded from initrd by [src/kernel/proc/user_program.cpp](../src/kernel/proc/user_program.cpp), with `PT_LOAD` segments mapped under the user PML4 slot, ELF flag → `PageFlags` translation (NX when `!X`), bounds-checked against `kUserSpaceBase`/stack guard, and `memsz - filesz` zeroed. User threads are entered through `iretq` ([src/kernel/proc/thread.cpp](../src/kernel/proc/thread.cpp)).

Syscall ABI (vector `0x80`, DPL 3, [src/kernel/syscall/dispatch.cpp](../src/kernel/syscall/dispatch.cpp)):

| number | name      | implemented behavior |
| ------ | --------- | -------------------- |
| 1      | `write`   | fd ∈ {1, 2}; bytes copied to console |
| 2      | `exit`    | mark thread dying, schedule |
| 3      | `yield`   | reschedule keeping current ready |
| 4      | `getpid`  | returns process pid |
| 5      | `read`    | fd 0 only, line-buffered, blocks on `ConsoleRead` |
| 6      | `observe` | fixed-record snapshots: system, processes, cpus, pci, initrd |
| 7      | `spawn`   | initrd-path child launch |
| 8      | `waitpid` | blocks on `ChildExit` |
| 9      | `exec`    | initrd-path in-place image replacement |

Notable absences (each one is a prerequisite for at least one GOALS.md milestone):

- No file descriptors, no fd table.
- No `argv` / `envp` on initial stack. The shell cannot pass arguments.
- No `fork`. `spawn(path)` is the only child-creation primitive.
- No signals.
- No `getuid`/`setuid`/credentials surface.
- No `dup`/`pipe`/`open`/`close`.
- No filesystem-backed `exec` — the kernel only knows initrd paths.
- No errno discipline; failure is a uniform `-1`.
- No `mmap`/anonymous memory; user heap does not exist.
- Wait state is still encoded by overloading `wait_address`/`wait_length`.

### Storage / filesystem state

`virtio-blk` driver ([src/kernel/drivers/block/virtio_blk.cpp](../src/kernel/drivers/block/virtio_blk.cpp)):

- Modern PCI device only (`vendor 1AF4`, `device 1042`).
- One queue, target size 8.
- No MSI/MSI-X — `queue_msix_vector` is set to `kVirtioNoVector` ([src/kernel/drivers/block/virtio_blk.cpp:451](../src/kernel/drivers/block/virtio_blk.cpp#L451)).
- Reads spin on `used->idx` for up to one million iterations.
- Writes are stubbed: `write_block_device` returns `false` unconditionally ([src/kernel/drivers/block/virtio_blk.cpp:272-275](../src/kernel/drivers/block/virtio_blk.cpp#L272-L275)).
- The smoke contract is "sector 0 begins with `OS1 VIRTIO TEST DISK SECTOR 0 SIGNATURE`."

`BlockDevice` facade ([src/kernel/storage/block_device.hpp](../src/kernel/storage/block_device.hpp)) exposes `name`, `sector_count`, `sector_size`, two function pointers, and `driver_state*`. Notable: no request structure, no completion/error path, no async/queue semantics, no flush/barrier, no statistics, no DMA-buffer ownership rule.

There is no filesystem. [src/kernel/vfs/README.md](../src/kernel/vfs/README.md) is still a placeholder. Loading executable bytes from anywhere except initrd is impossible.

### Drivers / device model

Implemented:

- VGA text + framebuffer text ([src/kernel/drivers/display/text_display.cpp](../src/kernel/drivers/display/text_display.cpp)).
- PS/2 keyboard ([src/kernel/drivers/input/ps2_keyboard.cpp](../src/kernel/drivers/input/ps2_keyboard.cpp)).
- Serial console (in-line under `arch/x86_64/asm` and console paths).
- PIT timer ([src/kernel/drivers/timer/pit.cpp](../src/kernel/drivers/timer/pit.cpp)).
- LAPIC / IOAPIC / 8259 PIC.
- PCI ECAM enumeration with BAR sizing.
- Single virtio-blk PCI driver.

There is no device model, in any meaningful sense. There is no:

- Driver-binding table.
- IRQ vector allocator.
- MSI/MSI-X table allocator.
- BAR / MMIO region resource owner.
- DMA-safe buffer allocator with explicit physical/virtual lifecycle.
- Hot-add / hot-remove path.
- `bus/`-level abstractions ([src/kernel/drivers/bus/README.md](../src/kernel/drivers/bus/README.md) is empty).
- Shared virtio transport layer ([src/kernel/drivers/virtio/README.md](../src/kernel/drivers/virtio/README.md) is empty).

The same finding from earlier reviews applies: any second virtio driver (likely virtio-net) will currently force a near-duplicate of capability walking, queue setup, and notify-offset calculation already present in `virtio_blk.cpp`.

### Networking state

Not started. [src/kernel/drivers/net/README.md](../src/kernel/drivers/net/README.md) is the only artifact.

### Security / multiuser state

Not started past ring separation and user-pointer validation. [src/kernel/security/README.md](../src/kernel/security/README.md) is the only artifact. There is no uid/gid type, no credential field on `Process`, no permission check anywhere, no entropy source, no crypto.

### GUI / terminal state

The framebuffer text path renders 8x16 glyphs into 32-bpp RGB/BGR with shadow-buffer dirty tracking ([src/kernel/drivers/display/text_display.cpp](../src/kernel/drivers/display/text_display.cpp)). The terminal model owns 12 logical terminals and a fixed cell grid. Console input from PS/2 and serial RX share one canonical line buffer ([src/kernel/console/console_input.cpp](../src/kernel/console/console_input.cpp)).

There is no GUI, no compositor, no input event abstraction, no mouse, no graphics API. This matches GOALS.md, which scopes GUI as late and optional.

### Build, tooling, and debug workflow

CMake/Ninja, x86_64-elf cross toolchain, NASM, cpio, xorriso. The CI workflow ([.github/workflows/ci.yml](../.github/workflows/ci.yml)) runs `cmake --preset default -DOS1_REQUIRE_UEFI_SMOKE=ON`, `cmake --build --preset default`, `cmake --build build --target os1_bios_image`, then `ctest`. No host-test phase.

CTest registers eight QEMU smoke tests covering UEFI and BIOS for baseline shell, `observe`, `spawn`, and `exec` behavior. The smoke runner ([cmake/scripts/run_smoke.py](../cmake/scripts/run_smoke.py)) drives serial input/output through QEMU.

Notably absent:

- Any host-compiled tests for parsers (`CPIO`, `ELF`, `MADT`, `MCFG`, PCI capabilities).
- Any unit tests for `VirtualMemory`, `PageFrameContainer`, `user_copy`, ELF loader, exec teardown, process lifecycle.
- Any structured kernel event ring / trace buffer beyond serial `debug()` lines and `observe()` snapshots.
- A formatting-enforcement target ([autoformat.sh](../autoformat.sh) exists; nothing fails CI when it is not run).
- Smoke coverage for `/bin/ascii`, which is staged in initrd but not asserted by any test.

## 4. Review of Progress Against Goals

[GOALS.md](../GOALS.md) is the target-state reference. Current completion levels per its A–G map and cross-cutting goals:

### Goal A — Clean boot and kernel baseline (M1)

**Completion: very high.** Both boot paths reach long mode, copy `BootInfo`, build kernel-owned page tables, switch CR3, and reach userland. Build-time contracts assert image envelope, kernel layout, and shim base. Higher-half migration further cemented this work.

Remaining: BSP still runs on a low handoff stack page until it migrates onto a kernel-thread stack. AP trampoline mailbox is still a low identity exception. Both are intentional bring-up bridges, but they should be tracked as known-debt items rather than quietly retained.

### Goal B — Modern platform handoff (M3 + M4)

**Completion: medium-to-high for QEMU/q35; low for broader hardware.**

Implemented:
- ACPI RSDP → XSDT/RSDT → MADT/MCFG.
- LAPIC/IOAPIC bring-up, PIC remap.
- PCI ECAM enumeration, BAR sizing.
- virtio-blk smoke device.
- AP startup with bounded wait.

Not implemented (each one is a real prerequisite for richer device work):
- **MSI/MSI-X.** No vector allocator, no MSI capability handler, no IOAPIC-to-MSI migration.
- **HPET / APIC timer.** Still PIT-only at 1000 Hz.
- **AML / ACPI device + power management.** No interpreter, no `_PRT`, no S-state handling.
- **PCI resource owner.** No structured BAR/IRQ ownership; drivers reach into the platform device list.
- **DMA-safe allocator.** No documented contract for "this physical range is queue/descriptor memory and is not movable."
- **USB/xHCI** and **HID keyboard/mouse**.
- **Generic IRQ allocation/binding** beyond `platform_enable_isa_irq`.

### Goal C — Protected userland and operator shell (M2 + M5)

**Completion: medium-to-high.** Ring-3 execution, `int 0x80` ABI, `spawn`/`waitpid`/`exec`, observe records, deferred reaping after user fault, dual VGA/framebuffer terminals.

Remaining (each one is small individually but blocks Goal D and beyond):
- argv/envp on the initial user stack.
- A per-process file-descriptor table (even just `{stdin, stdout, stderr}` typed slots).
- Filesystem-backed `exec` once a VFS lands.
- `fork`-or-equivalent (the shell language draft suggests a structured spawn rather than fork; this needs a decision).
- Signals or an event-delivery primitive (needed for the events facet of the native interface draft).
- Process credentials.
- Typed wait state.
- Errno discipline beyond `-1`.

### Goal D — Persistence and local security (next major milestone)

**Completion: low.**

The block path is smoke-grade. There is no filesystem. There are no credentials or permission checks. This is the largest dependency cluster in the codebase right now, and it has not moved since the prior review. It is also where the GOALS.md sequencing notes are most explicit: the milestone should split into a block-layer half and a multiuser-foundation half, and not be conflated.

### Goal E — Networking foundation

**Completion: not started.** Correctly blocked behind Goal D. Adding a NIC before the storage path has interrupt-driven completion, DMA buffer ownership, and event observability would bake polling assumptions into the first packet path.

### Goal F — SSH-capable remote administration

**Completion: not started.** Correctly blocked behind Goal D + E. Also blocked on entropy, crypto, PTYs, sessions, persistent key/config storage. SSH is at least 4–6 milestones away.

### Goal G — Optional desktop layer

**Completion: not started, correctly deferred.** The framebuffer text path is the right amount of GUI for the project stage.

### Cross-cutting — SMP shaping early architecture

**Completion: low.** AP discovery and bring-up exist. Runtime is BSP-only. The lock vocabulary is empty. The cost of fixing this *grows* with every additional kernel global.

### Cross-cutting — Accelerator awareness

**Completion: low.** PCI enumeration is the only present prerequisite. There is no DMA story, no IOMMU, no MSI-X, no user/kernel submission contract. Correctly deferred but should not be allowed to silently shape the system into a CPU-only design — see §5.

### Cross-cutting — Native (post-POSIX) system interface

**Completion: zero implementation, two new design drafts.** The drafts under [doc/shell-language/](shell-language/) clearly intend to serve GOALS.md's "Native system interface" objective ("kernel-managed objects, rights-bearing handles, synchronous and asynchronous calls, properties and introspection, event subscription and delivery"). The kernel currently exposes none of those primitives. This is a brand-new dependency cluster that the rest of the project does not yet acknowledge. See §5 and §6.

## 5. Overlooked Goals / Blind Spots

### The native object model has no skeleton in the kernel yet

[doc/shell-language/elf_interface_spec.md](shell-language/elf_interface_spec.md) describes a `.os.interface` ELF section carrying a typed schema of properties, methods, and events. [doc/shell-language/os1-shell-language-first-draft.md](shell-language/os1-shell-language-first-draft.md) describes a typed shell language that loads hosted objects, calls methods synchronously or asynchronously, subscribes to events, and treats this object surface as the system's primary interface "beyond POSIX." Both drafts assume a kernel that can:

- mint object handles with rights bits,
- route property reads/writes to handlers,
- dispatch method calls (synchronous and asynchronous),
- deliver events to subscribers,
- enforce per-handle permission checks.

The kernel today exposes nine `int 0x80` syscalls with five `observe` record types. There is no handle table, no rights bit, no event/subscription primitive, no async dispatch. Worse, the next obvious near-term work — file descriptors and waitable objects — could be done in two ways:

1. As POSIX-style integer FDs with `read`/`write`/`close`, and bolt on a parallel object/handle ABI later.
2. As typed kernel handles from day one, with `read`/`write` etc. expressed as method calls on those handles, and a POSIX shim layered on top later.

GOALS.md and the new drafts both prefer (2). If the project adopts (1) by accident — by adding file descriptors first and worrying about handles later — the FD table and the handle table will fight for the same role, and one of them will be ripped out.

**This is the single biggest architectural decision currently sitting unmade.** It needs an explicit milestone-level design before Goal D's userspace half lands.

### The repository has no unit-test culture

The CI runs only QEMU smokes ([.github/workflows/ci.yml:71-72](../.github/workflows/ci.yml#L71-L72)). There is no `host_tests/`, no GoogleTest/Catch2 dependency, no CMake target that compiles parser code with a host compiler. Every parser change has to round-trip through a full boot to be validated.

This is cheap to fix and unlocks the next four milestones:

- CPIO parser fixtures.
- ELF64 header / program-header validation.
- `MADT`, `MCFG`, ACPI checksum, `RSDP` / `RSDT` / `XSDT` header parsing with recorded byte fixtures.
- PCI capability walking with synthetic config-space fixtures.
- `VirtualMemory` map/protect/translate against a fake `PageFrameContainer`.
- `user_copy` boundary and permission failures.

It also unblocks the FD/handle work in §5.1: the right way to validate "calling `close()` on a stale handle returns the right error" is *not* to boot QEMU.

### No structured kernel event log

`debug()` (serial-only `printf`) is the kernel's only post-boot trace channel. `observe()` exposes static snapshots, not events. As soon as anything asynchronous lands — interrupt-driven storage, virtio-net, SMP scheduling — debugging becomes "stare at serial logs and hope the timing works out."

A small fixed-capacity event ring with typed records (trap, fault, scheduler transition, IRQ, block-completion, PCI-bind, user-copy-failure) exposed through a new `observe` kind is one of the cheapest debugging investments the project can make. It is also a hard prerequisite for SMP debugging.

### Synchronization vocabulary needs to exist *before* APs run

A spinlock, a `KIrqGuard` (cli/sti pair), a per-CPU `cpu->current_cpu_id` field used for assertions, and a written-down "this code runs only on the BSP" rule cost less than a day. Adding them after AP scheduling lands costs weeks of intermittent bugs. The right time is now.

### Boot-stack and AP-trampoline graduation

Two intentional low identity windows still exist ([src/kernel/core/kernel_main.cpp:138-160](../src/kernel/core/kernel_main.cpp#L138-L160) and the AP trampoline at `kApTrampolineAddress`). They should be tracked in a single "graduation list" so they do not silently expand. Milestone B explicitly mentions this.

### Resource ownership model is missing

PCI device records live in a global array; drivers grab them by index. There is no notion of "device X owns BAR Y, IRQ Z, DMA range W, and lifecycle V." Without it, virtio-net will copy virtio-blk quirks, and a future xHCI driver will copy both. This becomes painful very quickly.

### Portability boundaries are not labelled

GOALS.md scopes AArch64 and RISC-V as future ports. The current code makes that aspirational. Concrete portability assumptions to label explicitly in `arch/x86_64`-adjacent code:

- PML4-based 4-level paging.
- ACPI as the *only* discovery source.
- PCI-as-bus universally.
- `int 0x80` as the syscall mechanism.
- PIT/PIC/IOAPIC/LAPIC as the timer/IRQ stack.
- 4 KiB pages baseline.

None of these need to be portable today. They need to be *labelled* as x86-isms, so that the line between generic kernel and arch port is easier to draw later.

### Developer workflow lacks a verification ladder

The project would benefit from a documented test ladder: (1) host parser/logic tests, (2) a fast serial-only boot smoke, (3) full UEFI + BIOS smokes, (4) optional graphical smoke, (5) manual debug runs. Right now, only stages (3) and (5) exist.

## 6. Roadmap

Ordered by dependency, not by attractiveness. Items keep most of the prior review's sequencing where it has not moved, and add the native-interface decision and the higher-half-migration follow-ups.

### Near-term (next 1–3 milestones)

1. **Decide the native-interface stance before Goal D's userspace half.** Choose between (a) POSIX-FDs-first then handles-later, (b) handles-first then POSIX-shim-later, or (c) both ABIs side-by-side. Capture the decision as a milestone design doc that names which kernel structures get added (`Handle` vs `FileDescriptor`, where rights bits live, what `read`/`write` look like across both, how `observe` evolves, how the shell language draft consumes it). This decision blocks every following item.

2. **Establish a host-test harness.** Add a CMake `host_tests/` target with one of GoogleTest or doctest. Land the first three test groups together: CPIO parser, ELF64 header validation, ACPI MADT parsing with recorded fixtures. Wire `host_tests` into CI before the smoke step. *Why now:* every following item adds parser/state-machine code that will rot without unit coverage.

3. **Define the SMP synchronization contract while APs are still parked.** Land a kernel header that declares: `Spinlock`, `IrqGuard`, the "BSP-only" annotation, a `KASSERT_ON_BSP` macro, and a one-page note on locking order. Audit the current globals (`processTable`, `threadTable`, `g_next_pid`, `terminal[]`, `console_input`, the PCI list, page frames) and tag them with "BSP-only for now." *Why now:* the cost of adding these primitives is low while only the BSP runs anything.

4. **Land argv/envp on the initial user stack.** Tiny user/kernel ABI step. Lets `/bin/sh` actually invoke commands with arguments and unblocks meaningful filesystem-backed exec later. *Why now:* it is small, decoupled from §6.1, and unblocks every piece of shell work.

5. **Add a structured kernel event ring.** New `observe` kind, fixed-capacity ring, typed events for traps, scheduler transitions, IRQs, block I/O, PCI bind, user-copy failure. Implementation-side it is one file. *Why now:* this is the single best debugging investment for everything async that follows.

6. **Define the v2 BlockDevice contract on paper, then in code.** Request struct, completion callback, error codes, alignment rules, write semantics, ownership of buffers, statistics, queue depth, in-flight bookkeeping. Write the contract as a doc first, then refit `virtio_blk` to it. *Why now:* every filesystem and the future virtio-net cannot share completion infrastructure until this exists.

7. **Split a common virtio transport out of `virtio_blk`.** Cap walking, feature negotiation, queue setup, notify offset, used-index polling, eventually MSI/MSI-X completion. *Why now:* virtio-net is the next likely device. The duplicate cost compounds.

8. **Tighten initrd / smoke coverage and graduate `/bin/ascii`.** Either include `/bin/ascii` in a smoke or document the deliberate omission. Generate or expose an initrd manifest and assert staged binaries appear in `observe initrd`. *Why now:* drift is cheap to prevent and expensive to discover.

9. **Split [src/boot/limine/entry.cpp](../src/boot/limine/entry.cpp) into pluggable units.** HHDM translator, request consumer, ELF loader, BootInfo builder, transition-mapping installer. *Why now:* the file is now ~1300 lines and any change risks regressing one of the boot paths.

10. **Replace overloaded wait fields with a typed wait state union.** [src/kernel/proc/thread.hpp:50-54](../src/kernel/proc/thread.hpp#L50-L54). Easy refactor with the host tests in place.

### Mid-term (after the near-term cluster lands)

1. **Implement MSI/MSI-X and IRQ resource allocation.** Vector allocator, per-device MSI-X table walker, integration with the IOAPIC fallback. Should land before any second virtio device.

2. **Make `virtio-blk` interrupt-driven and add writes.** Move from the spin loop to MSI completion. Add the write path under the v2 contract.

3. **Add a minimal read-only filesystem.** FAT32 read-only or a custom-tuned simple FS. Mount it through a first-cut VFS, add `open`/`read`/`close` (or the handle-equivalent decided in §6.1), keep initrd as a fallback during transition.

4. **Add file descriptors / kernel handles per the §6.1 decision.** Tied to filesystem-backed `exec`. The shell should be able to `exec /usr/bin/echo` from a real filesystem.

5. **Begin the multiuser foundation.** uid/gid types, credential field on `Process`, file ownership fields in the new FS, passing-all-checks default permission policy. Aim for "single user, but the right shape." This is the half of GOALS.md Milestone D that is *not* the block layer.

6. **Bring up one AP for restricted work.** With the synchronization contract from §6.3, run timer accounting or virtio completion on a single AP. Do not flip all APs into the global scheduler at once.

### Later

1. **Networking.** virtio-net on the new transport, packet allocator, ARP, IPv4, UDP, TCP. Only after MSI-X, the v2 block contract, and the event ring exist.

2. **Native interface kernel surface.** Once §6.1's decision is implemented and FDs/handles exist, expose properties, methods, and events at a kernel level. Build the user-side runtime that loads `.os.interface` sections and turns them into typed objects. This is also when the shell language draft can become a real interpreter.

3. **SSH.** After networking, persistent storage, credentials, entropy, crypto, PTYs, and sessions exist.

4. **GUI.** After everything above. The framebuffer text path is plenty for a long time.

5. **Accelerator support.** After PCI resource ownership, DMA pinning, MSI-X, and a user/kernel submission ABI exist.

### Explicit prerequisites for major goals (carry-forward, lightly updated)

- **Before networking:** common virtio transport, MSI/MSI-X, DMA-safe ownership, IRQ allocator, packet buffer lifecycle, event ring.
- **Before SSH:** networking through TCP, entropy, crypto, credentials, PTYs/sessions, persistent storage, reliable process I/O.
- **Before GUI:** input event abstraction, framebuffer ownership rules, process/session model, scheduler stability, graphical buffer ownership.
- **Before SMP scheduling:** lock + preemption rules, audited globals, per-CPU runqueue (or one global lock), IPI/wakeup path, AP-safe console + allocator + page tables.
- **Before POSIX growth:** file-descriptor or handle table, path namespace, argv/envp, errno or equivalent error contract, process lifecycle semantics.
- **Before accelerator support:** PCI resource ownership, DMA allocation/pinning, MSI-X, user/kernel submission queues, fault/reset path.
- **Before native object interface:** §6.1 decision, handle/rights primitive in kernel, event/subscription syscalls, ELF `.os.interface` loader in user runtime.

## 7. Immediate Next Steps

In priority order. Each item is justified by what it unblocks now.

1. **Make the native-interface stance decision** (§6.1).
   *Why now:* it is the only architectural choice in the project that is both pending and simultaneously gates the next two milestones. Adding FDs first without the decision will lock in a POSIX-shaped abstraction that the GOALS.md "object-oriented system model" goal will eventually fight. The reverse mistake is symmetric. A two-page design doc that names the chosen direction is the most valuable thing the project can produce this month.

2. **Stand up a host test harness with three first parsers** (§6.2).
   *Why now:* every following step adds new parsing/state-machine code. The marginal cost of adding host tests later is much higher than now, and CI is currently QEMU-only.

3. **Write the SMP synchronization contract while APs are still parked** (§6.3).
   *Why now:* the kernel can still be audited as uniprocessor. Once APs run scheduler or driver code, every unprotected global becomes an intermittent bug.

4. **Add argv/envp** (§6.4).
   *Why now:* it is small, fully decoupled from §6.1, and unlocks meaningful shell tests with arguments.

5. **Add the structured kernel event ring** (§6.5).
   *Why now:* every async subsystem in the roadmap (interrupt-driven block, networking, SMP, accelerator) becomes much easier to debug with this in place.

6. **Define the BlockDevice v2 contract on paper before any new storage code** (§6.6).
   *Why now:* the current `BlockDevice` will not survive contact with a filesystem, and virtio-net will inherit the same polling shape if it lands first.

7. **Split common virtio transport out of `virtio_blk`** (§6.7).
   *Why now:* the second virtio driver is on the medium-term roadmap. Duplicating capability walks and queue setup is avoidable.

8. **Tighten initrd / smoke coverage including `/bin/ascii`** (§6.8).
   *Why now:* the same finding has appeared in two consecutive reviews. Drift is cheap to prevent now.

9. **Split the Limine shim** (§6.9).
   *Why now:* the file is at ~1300 lines and continues to grow. Boot-shim regressions are hard to debug.

10. **Replace overloaded wait fields with a typed wait state** (§6.10).
    *Why now:* the wait reasons will multiply (`BlockIo`, `IpcRecv`, `Timer`, `Event`). Fixing the representation while there are two reasons is trivial; later it will not be.
