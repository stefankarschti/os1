## 1. Executive Summary

### Current project state

`os1` is now a higher-half C++20 kernel with two boot frontends, an ACPI-only platform layer, a small protected-userland slice driven by the x86_64 `SYSCALL`/`SYSRET` fast path, and a host-side unit-test harness wired into CI. The tree is approximately 5,100 lines of kernel C++ across 18 named subsystems (`grep`-counted from [src/kernel/](../src/kernel/)), plus the BIOS and Limine boot frontends and a small `cpio newc` userland under [src/user/](../src/user/).

Concrete substrate:

- Shared kernel linked at `kKernelVirtualOffset + physical` with `R E` / `R` / `RW` `PT_LOAD` segments and matching final-CR3 permissions in [src/kernel/core/kernel_main.cpp:51-81](../src/kernel/core/kernel_main.cpp#L51).
- Kernel-owned direct map at `kDirectMapBase`, accessed through `kernel_physical_pointer<T>` in [src/kernel/handoff/memory_layout.h:111-129](../src/kernel/handoff/memory_layout.h#L111).
- Two boot paths converging on `kernel_main(BootInfo*, cpu*)`: BIOS in [src/boot/bios](../src/boot/bios) and Limine/UEFI in [src/boot/limine](../src/boot/limine), now decomposed into `entry.cpp` (278 lines), `serial.cpp`, `paging.cpp` (324 lines), `elf_loader.cpp`, and `handoff_builder.cpp`.
- ACPI RSDP → XSDT/RSDT → MADT/MCFG; IOAPIC/LAPIC/PIC bring-up; PCI ECAM enumeration with BAR sizing; AP startup with bounded wait. APs park in `cpu_idle_loop()` ([src/kernel/arch/x86_64/cpu/cpu.cpp:16-26](../src/kernel/arch/x86_64/cpu/cpu.cpp#L16)).
- One device driver: modern `virtio-blk` with three-page split queue, `VERSION_1`-only feature set, polling completion, read-only.
- Ring-3 user procs loaded from initrd `cpio newc`. Six programs: `/bin/init`, `/bin/sh`, `/bin/yield`, `/bin/fault`, `/bin/copycheck`, `/bin/ascii`.
- 9-call syscall ABI delivered through x86_64 `SYSCALL`/`SYSRET`, set up in [src/kernel/arch/x86_64/cpu/syscall.cpp](../src/kernel/arch/x86_64/cpu/syscall.cpp). The `int 0x80` IDT gate has been removed; only `IA32_LSTAR`/`IA32_FMASK`/`IA32_STAR`/`EFER.SCE` are programmed and the `syscall_entry` symbol in [src/kernel/arch/x86_64/interrupt/irqhandler.asm:126](../src/kernel/arch/x86_64/interrupt/irqhandler.asm#L126) is the only entry. The lingering `int 0x80` reference in [src/uapi/os1/syscall_numbers.h:6](../src/uapi/os1/syscall_numbers.h#L6) is now a stale comment.
- Typed `ThreadWaitState` union in [src/kernel/proc/thread.hpp:30-58](../src/kernel/proc/thread.hpp#L30), replacing the previous overloaded `wait_address`/`wait_length` representation.
- A real synchronization vocabulary: `Spinlock`, `IrqGuard`, `OS1_BSP_ONLY`, `KASSERT_ON_BSP()` in [src/kernel/sync/smp.hpp](../src/kernel/sync/smp.hpp), and a documented locking order in [doc/2026-04-29-smp-synchronization-contract.md](2026-04-29-smp-synchronization-contract.md).
- A 256-record kernel event ring with seven event types (trap, sched-transition, irq, block-io, pci-bind, user-copy-failure, smoke-marker) in [src/kernel/debug/event_ring.cpp](../src/kernel/debug/event_ring.cpp), now used by the user-fault path, scheduler, virtio-blk, user-copy boundary, and PCI bind.
- Host unit tests under [tests/host/](../tests/host) using the vendored GoogleTest submodule. 38 cases registered today, covering `elf64`, `freestanding/string`, `boot_info`, `cpio_newc`, `page_frame`, `user_address`, `virtual_memory`, `user_elf`, observe ABI, kernel event ring, and `align`/`fixed_string`. CI runs the host suite *before* the cross build in [.github/workflows/ci.yml:43-50](../.github/workflows/ci.yml#L43).

### Strongest parts

1. **Velocity on roadmap items.** Five of the prior review's ten near-term items closed in one push: host tests, the SMP contract, the kernel event ring, `SYSCALL`/`SYSRET` migration, typed wait state, and the Limine shim split. None are half-implemented; each lands with code, tests where applicable, and matching documentation. This is the difference between a teaching-OS that drifts and one that is converging.
2. **The synchronization story is now ready for AP work.** [src/kernel/sync/smp.hpp](../src/kernel/sync/smp.hpp) gives the kernel a real `Spinlock`, an `IrqGuard` whose constructor reads `RFLAGS` once and only re-enables interrupts that were enabled, and a `KASSERT_ON_BSP()` macro that the BSP-only mutators in `process.cpp`, `thread.cpp`, and `kernel_state.cpp` already enforce. The `OS1_BSP_ONLY` annotation labels every global that is presently bootstrap-CPU-owned: `processTable`, `threadTable`, `g_next_pid`, `g_next_tid`, `g_idle_thread`, `terminal[]`, `active_terminal`, the console-input queue, `page_frames`, `g_kernel_root_cr3`, and the platform/PCI lists. The locking order in the contract doc names ten levels and the rules around interrupt-vs-data locks. This is a kernel that can let an AP run real work without descending into unbounded race-condition debugging — provided the migration path in §54-63 of the contract is followed.
3. **Observability ladder.** Three layers are now usable: `debug()` lines on serial for human triage; `observe()` snapshots for static state (`sys`, `ps`, `cpu`, `pci`, `initrd`, `events`); and the kernel event ring for ordered, typed events with sequence numbers. The shell built-in `events` displays 256 records with `pid`/`tid`/`cpu`/`flags`/four args. The user-copy boundary already records `OS1_KERNEL_EVENT_USER_COPY_FAILURE` with reason codes. The block path already records `BLOCK_IO` begin/success/failure with sector + length. This puts the project ahead of where every previous review predicted it would be at this stage.
4. **Boot-contract maturity.** Both paths build the same `BootInfo` arena at `0x500..0x8000`, jump through `kernel_main(BootInfo*, cpu*)`, and `own_boot_info()` deep-copies into kernel BSS so the kernel core is bootloader-agnostic from line two onward. Build-time scripts assert the kernel envelope and the Limine shim base. The Limine frontend now has clean ownership split: `entry.cpp` orchestrates, `paging.cpp` translates Limine-virtual pointers, `elf_loader.cpp` inspects/loads the kernel image by `p_paddr`, `handoff_builder.cpp` mirrors the BIOS arena layout. The shim is now ~280 lines of orchestration plus ~720 lines of named helpers; it was a single ~1300-line file as recently as the prior review.
5. **Host-test foundation.** Three of the project's most error-prone parsers — `cpio_newc`, ELF64 header validation, `BootInfo` ownership — have direct host coverage. So does the new event ring, page-frame allocation, the virtual-memory wrapper, and user-address validation. Every later subsystem (filesystem, network, MSI vector allocation, ACPI AML if it ever lands) can copy this seam. The tests live under [tests/host/](../tests/host) as a separate CMake project so the freestanding cross build is preserved.

### Biggest risks / gaps

1. **`virtio-blk` is still smoke-grade.** [src/kernel/drivers/block/virtio_blk.cpp:261](../src/kernel/drivers/block/virtio_blk.cpp#L261) spins one million times on `used->idx`; [src/kernel/drivers/block/virtio_blk.cpp:317-320](../src/kernel/drivers/block/virtio_blk.cpp#L317) hard-returns `false` for writes; the device is bound through `queue_msix_vector = kVirtioNoVector`. The `BlockDevice` facade in [src/kernel/storage/block_device.hpp](../src/kernel/storage/block_device.hpp) is a name + sector geometry + two synchronous function pointers, with no request/completion/error/queue concept. Any filesystem on top of this contract will have to be ripped out when the contract changes. **Same finding as the prior review; the new infrastructure (event ring, IRQ refactor headroom from `IrqGuard`) makes this easier to address now, not harder.**
2. **The native object/handle decision is still unmade.** The two design drafts under [doc/shell-language/](shell-language/) (ELF interface metadata + typed shell language) describe a kernel that mints object handles with rights bits and dispatches properties/methods/events. The kernel today exposes nine syscalls and zero handle primitives. The next big userspace step — file descriptors and waitable objects — can be done as POSIX FDs first or as kernel handles first. GOALS.md and the drafts both prefer the latter; doing the former by accident will lock in a POSIX-shaped abstraction that the explicit "object-oriented system model" goal will eventually fight. This was the prior review's number-one item; it has not moved.
3. **No process credentials, no FD/handle table, no argv/envp.** [src/kernel/proc/process.hpp:32-42](../src/kernel/proc/process.hpp#L32) carries `pid`, `state`, `address_space`, `exit_status`, `parent`, `name` — and nothing else. Goal D (multiuser foundation) and Goal F (SSH) are blocked behind every one of these. The `/bin/sh` shell cannot invoke commands with arguments today; `os1::user::spawn(path)` and `os1::user::exec(path)` accept only a path, no `argv`/`envp`.
4. **No filesystem, no VFS.** [src/kernel/vfs/](../src/kernel/vfs/) is still a placeholder folder. `exec` and `spawn` resolve through [src/kernel/fs/initrd.cpp](../src/kernel/fs/initrd.cpp). Storage growth and persistence growth share a single bottleneck: the missing block-device v2 contract, then a first read-only filesystem on top of it.
5. **Driver model is not generalized.** The `bus/` and `drivers/virtio/` directories are still placeholders. A future `virtio-net` will currently force a near-duplicate of the cap-walk + queue setup + notify-offset logic in `virtio_blk.cpp`. There is no MSI/MSI-X allocator, no IRQ vector allocator, no DMA-buffer ownership rule, no driver-binding table, no resource owner.
6. **APs are still parked.** [src/kernel/arch/x86_64/cpu/cpu.cpp:16](../src/kernel/arch/x86_64/cpu/cpu.cpp#L16) is `cli; hlt` forever. The synchronization vocabulary now exists, but no kernel code outside the event ring acquires a `Spinlock` yet. The first AP that runs anything past `cpu_idle_loop()` is a separate engineering project, not a free byproduct of the new contract.
7. **Documentation drift around the syscall ABI.** [doc/ARCHITECTURE.md:736](ARCHITECTURE.md#L736) correctly says `SYSCALL`/`SYSRET`. The README is correct. But [src/uapi/os1/syscall_numbers.h:6](../src/uapi/os1/syscall_numbers.h#L6) still labels the ABI "the int 0x80 ABI" in its only top-of-file doc comment, and the prior review at [doc/2026-04-29-review.md:19](2026-04-29-review.md#L19) still lists "9-call syscall ABI on `int 0x80`." Both pre-date today's work. They need a one-line cleanup.

## 2. Current State From Code

### Boot path and platform assumptions

Both frontends end at `kernel_main(BootInfo*, cpu*)` in [src/kernel/core/kernel_main.cpp:84](../src/kernel/core/kernel_main.cpp#L84). The bring-up sequence is deterministic and unchanged from prior reviews in shape, but several pieces have grown:

1. `own_boot_info()` deep-copies header, memory map, modules, strings into kernel BSS.
2. `cpu_initialize_record(g_cpu_boot)` then `cpu_init()` set up GDT/TSS/`gs` for the BSP.
3. `page_frames.initialize(memory_regions, ...)` builds the bitmap allocator.
4. `reserve_tracked_physical_range` reserves modules and framebuffer.
5. `VirtualMemory kvm` builds the final kernel CR3: bootstrap-low identity for AP/handoff stack, kernel image by `R E`/`R`/`RW` sections, direct map, explicit modules/framebuffer/RSDP windows.
6. `kvm.activate()` switches CR3, `g_kernel_direct_map_ready = true`, BSP CPU pointer rebound through the direct map.
7. `platform_init()` runs ACPI → topology → PCI ECAM → virtio-blk probe.
8. `pic_init` / `ioapic_init` / `lapic_init` / `cpu_boot_others(g_kernel_root_cr3)`. APs park.
9. Display selection, terminal allocation (12 × `kPageSize`-rounded buffers), IDT registration, keyboard, console-input.
10. Fixed `Process[32]` and `Thread[32]` tables allocated; kernel idle thread created.
11. `load_user_program("/bin/init")` → `enter_first_thread`. `set_timer(1000)` lights the PIT.

Hard platform assumptions:

- x86_64 only.
- ACPI is required; legacy MP fallback was removed in commit f0b4e07 ([src/kernel/platform/init.cpp:32-39](../src/kernel/platform/init.cpp#L32)).
- PCI is ECAM/MCFG only.
- `BootInfo` caps: 128 memory regions, 16 modules ([src/kernel/handoff/boot_info.hpp:13-14](../src/kernel/handoff/boot_info.hpp#L13)). Real UEFI memory maps are routinely larger.
- Two intentional low identity exceptions remain after CR3 switch: the AP trampoline mailbox at `0x20-0x40` and the live BSP handoff stack page (mapped explicitly at [src/kernel/core/kernel_main.cpp:152-161](../src/kernel/core/kernel_main.cpp#L152)).

### Memory management

Physical: `PageFrameContainer` in [src/kernel/mm/page_frame.cpp](../src/kernel/mm/page_frame.cpp) — fixed bitmap at `kPageFrameBitmapBaseAddress = 0x20000`, sized `0x40000` bytes, capping the allocator at 8 GiB of tracked pages. No buddy/slab/freelist. Every kernel allocation is page-granular.

Virtual: `VirtualMemory` in [src/kernel/mm/virtual_memory.cpp](../src/kernel/mm/virtual_memory.cpp). Owns root allocation, four-level walks, NX-aware permission writes, user-slot teardown, CR3 activation. User CR3s clone only `kKernelPml4Index` and `kDirectMapPml4Index`; PML4 slot 0 is no longer cloned into user space.

User layout (still 1 process slot, slot 1):

- `kUserSpaceBase  = 0x0000_0080_0000_0000`
- `kUserImageBase  = 0x0000_0080_0040_0000`
- `kUserStackTop   = 0x0000_0080_4000_0000`
- `kUserStackPages = 16` (64 KiB)
- `kKernelThreadStackPages = 4` (16 KiB)

`mm/user_copy.cpp` validates canonical addresses, range overflow, the user region, and walks page entries to confirm `User+Write` permission before copies. Failures emit `OS1_KERNEL_EVENT_USER_COPY_FAILURE` with reason codes (`NULL_CONTEXT`, `BAD_RANGE`, `TRANSLATE`, `PERMISSION`, `STRING_TOO_LONG`) — that observability is new since the prior review and is genuinely useful for debugging spawned-program copy bugs.

The host suite covers `PageFrameContainer`, `VirtualMemory` map/protect/translate, and `user_address` boundary checks ([tests/host/mm/](../tests/host/mm/)). What it does not yet cover: kernel-stack guard pages (none exist), `kmalloc`/slab (none exists), and the NX/W^X invariant after the final kernel CR3 (testable but not asserted).

### Interrupts / exceptions

IDT, PIC remap, IOAPIC, LAPIC, PIT (1000 Hz), PS/2 keyboard. Notably absent: any `int 0x80` IDT gate. The IDT is set up entirely in [src/kernel/arch/x86_64/interrupt/interrupt.cpp:105-175](../src/kernel/arch/x86_64/interrupt/interrupt.cpp#L105), and the only registered vectors are `0x00..0x14`, `0x1D`, `0x1E`, plus `irq0..irq15`. Syscalls go through `IA32_LSTAR` only.

The trap entry path in [src/kernel/arch/x86_64/interrupt/irqhandler.asm](../src/kernel/arch/x86_64/interrupt/irqhandler.asm) builds a unified `TrapFrame` for exceptions, IRQs, syscalls, and scheduler returns. The C dispatcher in [src/kernel/core/trap_dispatch.cpp](../src/kernel/core/trap_dispatch.cpp) is a 23-line classifier: IRQ → `handle_irq`, syscall → `handle_syscall`, otherwise `handle_exception`.

Kernel faults dump and halt; user faults emit a `TRAP` event, mark thread/process dying, and reschedule ([src/kernel/core/fault.cpp](../src/kernel/core/fault.cpp)). The timer hook polls serial RX, increments `g_timer_ticks`, and calls `schedule_next(true)`.

The architecture is sound for a uniprocessor kernel. It is **not** an SMP interrupt architecture: no per-CPU vector reservation, no MSI/MSI-X, no IPI infrastructure, no edge-vs-level normalization beyond the ACPI-derived `InterruptOverride` set, no IRQ ownership/priority model, no threaded-IRQ mechanism. Any fast-completion device (virtio-net, NVMe) cannot be wired up cleanly here.

### Scheduling / SMP state

`Thread`/`Process` are fixed-size in 32-entry tables. Round-robin `next_runnable_thread` over the table ([src/kernel/sched/scheduler.cpp:22-47](../src/kernel/sched/scheduler.cpp#L22)). Idle thread on the BSP. `wait_for_ap_boot` is bounded ([src/kernel/arch/x86_64/cpu/cpu.cpp:14](../src/kernel/arch/x86_64/cpu/cpu.cpp#L14)). After AP startup, `init()` runs `ioapic_init()` and `lapic_init()` per AP and drops into `cpu_idle_loop()`.

The synchronization vocabulary now exists, but the in-tree usage is still narrow: only [src/kernel/debug/event_ring.cpp:17](../src/kernel/debug/event_ring.cpp#L17) holds a real `Spinlock`, and only the event-ring entry/snapshot paths take it. Every other listed BSP-only global is still single-writer-by-construction. The `KASSERT_ON_BSP()` calls in `proc/process.cpp`, `proc/thread.cpp`, and platform discovery enforce that contract at runtime in debug builds. This is the right shape for the current parked-AP world; it is also a clear migration plan for the moment any AP starts running real code.

### User/kernel separation state

This remains the most complete vertical slice. ELF64 `ET_EXEC` images load from initrd by [src/kernel/proc/user_program.cpp](../src/kernel/proc/user_program.cpp), with `PT_LOAD` segments mapped in user PML4 slot 1, ELF flag → `PageFlags` translation (NX when `!X`), bounds-checked against `kUserSpaceBase`/stack guard, and `memsz - filesz` zeroed. User threads enter via `iretq` ([src/kernel/proc/thread.cpp:191-199](../src/kernel/proc/thread.cpp#L191)).

Syscall ABI ([src/uapi/os1/syscall_numbers.h](../src/uapi/os1/syscall_numbers.h), [src/kernel/syscall/dispatch.cpp](../src/kernel/syscall/dispatch.cpp)):

| number | name      | implemented behavior                                  |
| ------ | --------- | ----------------------------------------------------- |
| 1      | `write`   | fd ∈ {1, 2}; bytes copied to console                  |
| 2      | `exit`    | mark thread dying, schedule                           |
| 3      | `yield`   | reschedule keeping current ready                      |
| 4      | `getpid`  | returns process pid                                   |
| 5      | `read`    | fd 0 only, line-buffered, blocks on `ConsoleRead`     |
| 6      | `observe` | versioned fixed-record snapshots: system / processes / cpus / pci / initrd / events |
| 7      | `spawn`   | initrd-path child launch                              |
| 8      | `waitpid` | blocks on `ChildExit`                                 |
| 9      | `exec`    | initrd-path in-place image replacement                |

Wait state is now typed:

```cpp
struct ThreadWaitState {
    ThreadWaitReason reason;
    union { ConsoleReadWaitState console_read; ChildExitWaitState child_exit; };
};
```

Each reason has its own struct. The wait callers in `console_read.cpp` and `wait.cpp` access by name, not by reusing two opaque integer fields. This kills the previously-flagged class of bugs where a future `ThreadWaitReason` would silently overload the same two slots.

Notable absences (each one blocks at least one GOALS.md milestone):

- No file descriptors / handle table.
- No `argv` / `envp` on the initial user stack.
- No `fork`. `spawn(path)` is the only child primitive.
- No signals or event-delivery primitive.
- No credentials surface (`uid`/`gid`/`creds`).
- No `dup`/`pipe`/`open`/`close`.
- No filesystem-backed `exec`.
- No errno; failure is uniform `-1`.
- No `mmap` / anonymous memory; no user heap.

### Storage / filesystem state

`virtio-blk` ([src/kernel/drivers/block/virtio_blk.cpp](../src/kernel/drivers/block/virtio_blk.cpp)):

- Modern PCI device only (`vendor 1AF4`, `device 1042`).
- One queue, target size 8.
- No MSI/MSI-X — `queue_msix_vector = kVirtioNoVector`.
- Reads spin on `used->idx` for up to a million iterations.
- Writes return `false`.
- Smoke contract: sector 0 begins with `OS1 VIRTIO TEST DISK SECTOR 0 SIGNATURE`.
- New since prior review: emits `OS1_KERNEL_EVENT_BLOCK_IO` events with `BEGIN`/`SUCCESS`/`FAILURE` flags.

`BlockDevice` facade ([src/kernel/storage/block_device.hpp](../src/kernel/storage/block_device.hpp)) exposes `name`, `sector_count`, `sector_size`, two function pointers, `driver_state*`. No request struct, no completion/error path, no async/queue semantics, no flush/barrier, no statistics, no DMA-buffer ownership rule.

There is no filesystem. [src/kernel/vfs/](../src/kernel/vfs/) holds only a README. Loading executable bytes from anywhere except initrd is impossible.

### Drivers / device model

Implemented:

- VGA text + framebuffer text ([src/kernel/drivers/display/text_display.cpp](../src/kernel/drivers/display/text_display.cpp)).
- PS/2 keyboard.
- Serial console (in-line under `arch/x86_64`, `console/`).
- PIT timer.
- LAPIC / IOAPIC / 8259 PIC.
- PCI ECAM enumeration with BAR sizing.
- Single virtio-blk PCI driver.

Missing (each blocks at least one GOALS.md driver-priority item):

- Driver-binding table.
- IRQ vector allocator.
- MSI/MSI-X table allocator.
- BAR / MMIO region resource owner.
- DMA-safe buffer allocator with explicit physical/virtual lifecycle.
- Hot-add / hot-remove path.
- `drivers/bus/` infrastructure (placeholder).
- `drivers/virtio/` shared transport (placeholder).

### Networking state

Not started. [src/kernel/drivers/net/README.md](../src/kernel/drivers/net/README.md) is the only artifact.

### Security / multiuser state

Not started past ring separation and user-pointer validation. [src/kernel/security/](../src/kernel/security/) is a placeholder. There is no `uid_t`, no credentials field on `Process`, no permission check anywhere, no entropy source, no crypto.

### GUI / terminal state

The framebuffer text path renders 8x16 glyphs into 32-bpp RGB/BGR with shadow-buffer dirty tracking. The terminal model owns 12 logical terminals and a fixed cell grid sized from the framebuffer. PS/2 + serial RX feed one canonical line buffer ([src/kernel/console/console_input.cpp](../src/kernel/console/console_input.cpp)).

There is no GUI, no compositor, no input event abstraction, no mouse, no graphics API. This matches GOALS.md, which scopes GUI as late and optional.

### Build, tooling, and debug workflow

CMake/Ninja, x86_64-elf cross toolchain, NASM, cpio, xorriso. CI ([.github/workflows/ci.yml](../.github/workflows/ci.yml)) now runs:

1. Configure host tests.
2. Build host tests.
3. Run host tests through ctest.
4. Install cross toolchain.
5. Configure freestanding root project with `OS1_REQUIRE_UEFI_SMOKE=ON`.
6. Build default UEFI artifact.
7. Build BIOS artifact explicitly.
8. Run the eight QEMU smoke matrix.

That ladder is the right shape: parser/policy bugs are caught before a single QEMU run starts. The host suite registers 38 test cases today across `common/`, `handoff/`, `mm/`, `proc/`, `fs/`, `kernel/event_ring`, `kernel/utility`, `syscall/observe`.

QEMU smoke matrix (eight tests): UEFI and BIOS variants of `os1_smoke`, `os1_smoke_observe`, `os1_smoke_spawn`, `os1_smoke_exec`. Driven by [cmake/scripts/run_smoke.py](../cmake/scripts/run_smoke.py).

## 4. Review of Progress Against Goals

[GOALS.md](../GOALS.md) lists Milestones A–G plus several cross-cutting goals. Current completion levels:

### Goal A — Clean boot and kernel baseline (M1)

**Completion: very high.** Both boot paths reach long mode, copy `BootInfo`, build kernel-owned page tables, switch CR3, and reach userland. Build-time scripts assert image envelope, kernel layout, and shim base. The higher-half migration cemented the address-model shape. Outstanding low-bootstrap exceptions (AP trampoline mailbox, BSP handoff stack page) are intentional bring-up bridges, but they should be tracked as known-debt items rather than quietly retained.

### Goal B — Modern platform handoff (M3 + M4)

**Completion: medium-to-high for QEMU/q35; low for broader hardware.**

Implemented: ACPI RSDP → XSDT/RSDT → MADT/MCFG; LAPIC/IOAPIC bring-up; PIC remap; PCI ECAM enumeration; BAR sizing; `virtio-blk` smoke device; AP startup with bounded wait.

Not implemented: MSI/MSI-X (no vector allocator, no MSI capability handler, no IOAPIC-to-MSI migration); HPET / APIC timer (still PIT-only at 1 kHz); AML/ACPI device + power management; PCI resource owner; DMA-safe allocator; USB/xHCI; HID keyboard/mouse; generic IRQ allocation/binding beyond `platform_enable_isa_irq`.

### Goal C — Protected userland and operator shell (M2 + M5)

**Completion: high for the slice it covers, low for what comes next.**

Done: ring-3 execution, `SYSCALL`/`SYSRET` ABI, `spawn`/`waitpid`/`exec`, observe records (six kinds), deferred reaping after user fault, dual VGA/framebuffer terminals, typed wait state, kernel event ring.

Remaining (each is small individually but blocks Goal D and beyond):

- argv/envp on the initial user stack.
- A per-process file-descriptor or handle table — see §5 for the architectural decision behind which.
- Filesystem-backed `exec` once a VFS lands.
- `fork` or a structured spawn (the shell-language draft suggests structured spawn; this needs an explicit decision).
- Signals / event-delivery primitive (also needed for the events facet of the native interface draft).
- Process credentials.
- Errno discipline beyond `-1`.

### Goal D — Persistence and local security (next major milestone)

**Completion: low. Largest dependency cluster in the codebase.**

The block path is smoke-grade. There is no filesystem. There are no credentials or permission checks. GOALS.md correctly splits this milestone into a block-layer half and a multiuser-foundation half; both halves have not started in code. Same finding as the prior two reviews.

### Goal E — Networking foundation

**Completion: not started.** Correctly blocked behind Goal D's block-layer half. Adding a NIC before the storage path has interrupt-driven completion, DMA buffer ownership, and event observability would bake polling assumptions into the first packet path.

### Goal F — SSH-capable remote administration

**Completion: not started.** Correctly blocked behind Goal D + E. Also blocked on entropy, crypto, PTYs, sessions, persistent key/config storage. SSH is at least 4–6 milestones away.

### Goal G — Optional desktop layer

**Completion: not started, correctly deferred.** The framebuffer text path is the right amount of GUI for the project stage.

### Cross-cutting — SMP shaping early architecture

**Completion: medium and rising.** Discovery, AP startup, `OS1_BSP_ONLY` annotations, `Spinlock`/`IrqGuard`, locking-order doc, `KASSERT_ON_BSP()` runtime checks. Runtime is still BSP-only, but the kernel is no longer one architecture-decision away from intermittent bugs the moment an AP runs anything. The remaining work is concrete migrations of currently-`OS1_BSP_ONLY` state, in the order documented in the contract.

### Cross-cutting — Accelerator awareness

**Completion: low.** PCI enumeration is the only present prerequisite. There is no DMA story, no IOMMU, no MSI-X, no user/kernel submission contract. Correctly deferred but should not be allowed to silently shape the system into a CPU-only design.

### Cross-cutting — Native (post-POSIX) system interface

**Completion: zero implementation, two design drafts ([doc/shell-language/](shell-language/)).** The drafts target GOALS.md's "Native system interface" objective. The kernel exposes none of the required primitives: no handle table, no rights bits, no event/subscription syscalls, no async dispatch. The decision around POSIX-FDs-first vs handles-first is still the single largest unmade architectural choice in the project.

## 5. Overlooked Goals / Blind Spots

### The native object model still has no kernel skeleton

[doc/shell-language/elf_interface_spec.md](shell-language/elf_interface_spec.md) defines a `.os.interface` ELF section carrying typed properties, methods, and events. [doc/shell-language/os1-shell-language-first-draft.md](shell-language/os1-shell-language-first-draft.md) defines a typed shell language that loads hosted objects, calls methods synchronously or asynchronously, subscribes to events. Both assume kernel primitives that do not exist:

- mint object handles with rights bits,
- route property reads/writes to handlers,
- dispatch method calls (synchronous and asynchronous),
- deliver events to subscribers,
- enforce per-handle permission checks.

The kernel exposes nine syscalls and six observe-record kinds. There is no handle table, no rights bit, no event/subscription primitive, no async dispatch. The next obvious near-term work — file descriptors and waitable objects — could be done in two ways:

1. POSIX-style integer FDs with `read`/`write`/`close`, and bolt on a parallel object/handle ABI later.
2. Typed kernel handles from day one, with `read`/`write` etc. expressed as method calls on those handles, and a POSIX shim layered on top later.

GOALS.md and the new drafts both prefer (2). If the project adopts (1) by accident — by adding file descriptors first and worrying about handles later — the FD table and the handle table will fight for the same role, and one of them will be ripped out. **This is the single biggest architectural decision currently sitting unmade.**

### A v2 BlockDevice contract is now blocking three milestones

The current `BlockDevice` is read-only-by-stub, polling-only, and has no completion model. Three milestones need a richer contract:

- **Goal D's block-layer half** cannot have a filesystem on top of two synchronous function pointers without re-doing the contract work.
- **Goal E's first NIC** will inherit the same shape if `virtio-net` lands before a v2 block contract — packet RX/TX wants requests, completions, and DMA ownership rules, the same as block.
- **Cross-cutting accelerator awareness** also wants a documented DMA-safe allocator. That is the same allocator. Designing it once for storage is much cheaper than designing it twice.

A two-page design doc that names: `BlockRequest`, completion callback, error codes, alignment rules, write semantics, buffer ownership, statistics, queue depth, and in-flight bookkeeping — is a few hours of work and unblocks the next two quarters of roadmap.

### Common virtio transport split is overdue

The next likely device is `virtio-net`. Without a shared transport in `drivers/virtio/`, a second driver will duplicate cap walking, feature negotiation, queue setup, and notify-offset calculation. The duplicate cost is exactly the effort to extract the shared layer; doing it inside `virtio_blk.cpp` will be more painful than starting from scratch on a third driver.

### IRQ vector and MSI/MSI-X allocator does not exist

Today every driver routes through `platform_enable_isa_irq`. Even moving `virtio-blk` from polling to interrupt-driven needs a vector allocator first. This is the right time to do it because there is exactly one driver to migrate and exactly zero callers to break.

### Boot-stack and AP-trampoline graduation list is still implicit

Two intentional low identity windows remain ([src/kernel/core/kernel_main.cpp:138-160](../src/kernel/core/kernel_main.cpp#L138)). They should be tracked in a single "graduation list" in `ARCHITECTURE.md` so they do not silently expand. Milestone B in GOALS.md explicitly mentions this.

### Resource ownership model is still missing

PCI device records live in a global array; drivers grab them by index. There is no notion of "device X owns BAR Y, IRQ Z, DMA range W, and lifecycle V." Without it, virtio-net will copy virtio-blk quirks, and a future xHCI driver will copy both. Same finding as the prior two reviews; still unaddressed.

### Portability boundaries are still unlabelled

GOALS.md scopes AArch64 and RISC-V as future ports. The current code makes that aspirational. Concrete portability assumptions to label explicitly in `arch/x86_64`-adjacent code:

- PML4-based 4-level paging.
- ACPI as the only discovery source.
- PCI-as-bus universally.
- `SYSCALL`/`SYSRET` as the syscall mechanism.
- PIT/PIC/IOAPIC/LAPIC as the timer/IRQ stack.
- 4 KiB pages baseline.

None of these need to be portable today. They need to be *labelled* as x86-isms, so that the line between generic kernel and arch port is easier to draw later.

### Initrd manifest assertion would prevent silent drift

`/bin/ascii` is staged but not asserted by any registered smoke. Every prior review has flagged this. Five minutes of `expect`-style assertion in the observe smoke would close the loop.

### Kernel `kmalloc`/slab is not yet needed but will be soon

Every kernel allocation today is a page. The first filesystem will want a small-object allocator for inode caches, dentry caches, etc. The first network stack will want one for skbuffs/mbufs. Naming the constraint now (so no one accidentally designs a filesystem expecting a working `kmalloc`) is cheaper than discovering it later.

## 6. Roadmap

Ordered by dependency, not attractiveness. Items keep most of the prior review's ordering where it has not moved, drop the items that PRs #15-#19 closed, and re-time everything else.

### Near-term (next 1–3 milestones)

1. **Decide the native-interface stance before Goal D's userspace half** ([§5](#5-overlooked-goals--blind-spots)). Choose between (a) POSIX-FDs-first then handles-later, (b) handles-first then POSIX-shim-later, or (c) both side-by-side. Capture the decision as a milestone design doc that names which kernel structures get added (`Handle` vs `FileDescriptor`), where rights bits live, what `read`/`write` look like across both, how `observe` evolves, how the shell-language draft consumes it. *Why now:* this decision blocks every following userspace item and was the prior review's number-one finding.
2. **Define the `BlockDevice` v2 contract on paper, then in code.** Request struct, completion callback, error codes, alignment rules, write semantics, buffer ownership, statistics, queue depth, in-flight bookkeeping. Write the contract as a doc first, then refit `virtio_blk` to it. *Why now:* every filesystem and the future virtio-net cannot share completion infrastructure until this exists; the kernel event ring can already represent the events it produces.
3. **Land argv/envp on the initial user stack.** Tiny ABI step. Lets `/bin/sh` invoke commands with arguments and unblocks meaningful filesystem-backed exec later. Decoupled from the §6.1 decision. *Why now:* small, isolated, unblocks every shell improvement.
4. **Split common virtio transport out of `virtio_blk`.** Cap walking, feature negotiation, queue setup, notify-offset, used-index polling — pulled into `drivers/virtio/`. Refit virtio-blk on top. *Why now:* virtio-net is the next likely device and the duplicate cost compounds.
5. **Add an MSI/MSI-X capability walker and IRQ vector allocator.** Even the existing virtio-blk wants this to leave its million-iteration spin behind. *Why now:* introducing it with one driver is much easier than retrofitting two.
6. **Migrate `virtio-blk` from spin polling to interrupt-driven completion, and add the write path under the v2 contract.** *Why now:* uses the new vector allocator, fills out the v2 contract, and produces a first device whose completion semantics resemble what virtio-net and NVMe will need.
7. **Add a minimal read-only filesystem.** FAT32 read-only or a simple bespoke FS, mounted through a first-cut VFS. Add `open`/`read`/`close` (or the handle-equivalent decided in §6.1). Keep initrd as a fallback during transition. *Why now:* unblocks filesystem-backed `exec`, multiuser identity files, and persistent shell config.
8. **Add file descriptors / kernel handles per the §6.1 decision.** Tied to filesystem-backed `exec`. The shell should be able to `exec /usr/bin/echo` from a real filesystem. *Why now:* this is the actual content of Goal D's userland half.
9. **Introduce a graduation list for low-identity windows and BSP-only globals in `ARCHITECTURE.md`.** Explicit table: AP trampoline mailbox, BSP handoff stack page, every `OS1_BSP_ONLY` declaration. *Why now:* prevents silent expansion and gives later AP work a checklist.
10. **Tighten initrd / smoke coverage including `/bin/ascii`.** Either include it in a smoke or document the deliberate omission. Generate or expose an initrd manifest and assert that staged binaries appear in `observe initrd`. *Why now:* this finding has now appeared in three consecutive reviews.

### Mid-term (after the near-term cluster lands)

1. **Begin the multiuser foundation.** uid/gid types, credential field on `Process`, file ownership in the new FS, default permission policy. Aim for "single user, but the right shape." This is the half of GOALS.md Milestone D that is *not* the block layer.
2. **Bring up one AP for restricted work.** With the synchronization contract and the §6.1-decided handle/FD model, run timer accounting or virtio completion on a single AP. Do not flip all APs into the global scheduler at once. The migration order in [doc/2026-04-29-smp-synchronization-contract.md](2026-04-29-smp-synchronization-contract.md) is the right map.
3. **Introduce a kernel `kmalloc`/slab.** Once the filesystem and any network code arrive, page-granule allocations alone will not be enough.
4. **Begin networking on the new transport.** virtio-net, packet allocator, ARP, IPv4, UDP, TCP. Only after MSI-X, the v2 block contract, the event ring (already done), and the interrupt-driven block path exist.
5. **Native interface kernel surface.** Once §6.1's decision is implemented and FDs/handles exist, expose properties, methods, and events at a kernel level. Build the user-side runtime that loads `.os.interface` sections and turns them into typed objects. This is also when the shell-language draft can become a real interpreter.

### Later

1. **SSH.** After networking, persistent storage, credentials, entropy, crypto, PTYs, and sessions exist.
2. **GUI.** After everything above. The framebuffer text path is plenty for a long time.
3. **Accelerator support.** After PCI resource ownership, DMA pinning, MSI-X, and a user/kernel submission ABI exist. The kernel event ring is a good substrate for completion observability when this lands.

### Explicit prerequisites for major goals (carry-forward, lightly updated)

- **Before networking:** common virtio transport, MSI/MSI-X, DMA-safe ownership, IRQ allocator, packet buffer lifecycle.
- **Before SSH:** networking through TCP, entropy, crypto, credentials, PTYs/sessions, persistent storage, reliable process I/O.
- **Before GUI:** input event abstraction, framebuffer ownership rules, process/session model, scheduler stability, graphical buffer ownership.
- **Before SMP scheduling:** lock + preemption rules (now present), audited globals (now annotated), per-CPU runqueue (or one global lock), IPI/wakeup path, AP-safe console + allocator + page tables.
- **Before POSIX or handle growth:** §6.1 decision, file-descriptor or handle table, path namespace, argv/envp, errno or equivalent error contract, process lifecycle semantics.
- **Before accelerator support:** PCI resource ownership, DMA allocation/pinning, MSI-X, user/kernel submission queues, fault/reset path.
- **Before native object interface:** §6.1 decision, handle/rights primitive in kernel, event/subscription syscalls, ELF `.os.interface` loader in user runtime.

## 7. Immediate Next Steps

In priority order. Each item is justified by what it unblocks now.

1. **Make the native-interface stance decision** (§6.1).
   *Why now:* it is the only architectural choice in the project that is both pending and simultaneously gates the next two milestones. Adding FDs first without the decision will lock in a POSIX-shaped abstraction that the GOALS.md "object-oriented system model" goal will eventually fight. The reverse mistake is symmetric. A two-page design doc that names the chosen direction is the most valuable thing the project can produce this week.
2. **Define the `BlockDevice` v2 contract on paper before any new storage code** (§6.2).
   *Why now:* the current `BlockDevice` will not survive contact with a filesystem, and virtio-net will inherit the same polling shape if it lands first. The kernel event ring already gives the contract a place to publish completions.
3. **Add argv/envp** (§6.3).
   *Why now:* small, fully decoupled from §6.1, and unlocks meaningful shell tests with arguments. This was on the prior review's list and remains the cheapest ABI step.
4. **Split common virtio transport out of `virtio_blk`** (§6.4).
   *Why now:* virtio-net is the second virtio device on the medium-term roadmap. Duplicating capability walks and queue setup is avoidable and gets harder once two drivers exist.
5. **Add an MSI/MSI-X capability walker and IRQ vector allocator** (§6.5).
   *Why now:* required by both interrupt-driven block and any future NIC. Easier with one driver to migrate.
6. **Migrate `virtio-blk` to interrupt-driven completion and add writes** (§6.6).
   *Why now:* fills out the v2 contract with a real driver, retires the million-iteration spin, and lets the next filesystem step reuse a real completion model.
7. **Land a one-page graduation list for low-identity windows and BSP-only globals** (§6.9).
   *Why now:* the synchronization contract is in place; the ARCHITECTURE.md surface should track the open positions explicitly so AP work has a checklist.
8. **Tighten initrd / smoke coverage including `/bin/ascii`** (§6.10).
   *Why now:* third consecutive review has carried this forward. Drift is cheap to prevent now.
9. **Documentation cleanup pass.** Drop the "int 0x80 ABI" wording in [src/uapi/os1/syscall_numbers.h:6](../src/uapi/os1/syscall_numbers.h#L6); mark `SYSCALL`/`SYSRET` as resolved in [GOALS.md:519](../GOALS.md#L519); add a `kernel_event` arrow in the ARCHITECTURE.md diagram.

    *Why now:* the architectural prerequisites will then exist. The `bus/`, `vfs/`, and `security/` directories already have placeholder ownership.
